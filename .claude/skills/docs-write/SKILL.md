# docs/ 執筆ルール

`docs/` は「コードを読んでも得られない**なぜ**（背景・意思決定・設計判断）」を残す永続ドキュメント。執筆・更新時は以下を守る。

## 基本原則

1. **対象は `docs/` 配下のみ**。コード実装・修正は行わない。
2. **最小限のルール**：必須事項のみ記述し、**コードを読めば分かることは書かない**（API エンドポイント・コンポーネント分割・型定義の詳細など）。
3. **単体完結**：ドキュメント単体で理解できる内容にする。
4. **コード参照を避ける**：ファイルパス・関数名・実装断片への依存的な説明をしない。
5. **4スペースインデント**：Markdown のネストは常にスペース 4 つ（タブ・2スペースは使わない）。
6. 出力は日本語。

## 手順

1. **目的の明確化**：何を・なぜ書く／直すかを先に定める。
2. **整合性の確認**：関連ドキュメントの見出し構造・粒度・用語を把握し、同カテゴリで一貫させる。既存ドキュメント間に矛盾があれば、**編集前に報告し合意を得てから**進める。
3. **執筆**：目的・前提・判断基準・制約を中心に書く。コードを見れば分かる具体説明は削除する。
4. **導線の同期**：目次・README・一覧があれば参照リンクを最小限で更新する。

## tasks/ → docs/ 移行（開発完了後）

`tasks/{feature-name}/design.md` の「docs/ への移行メモ」に従って永続化する。

- `requirements.md` → `docs/services/{service}/requirements.md` に統合
- `external-design.md` → `docs/services/{service}/external-design.md` に統合（UI 変更があった場合）
- `design.md` の重要な設計決定 → `docs/services/{service}/architecture.md` に **ADR** として追記
- 反映が完了したら `tasks/{feature-name}/` を削除する

ADR は「背景・問題 / 決定 / 根拠・トレードオフ」の形式で書く（実装詳細は書かない）。

## 品質チェック

- [ ] `docs/` 配下のみを変更している
- [ ] 最小限のルールに従っている（コードを読めば分かることを書いていない）
- [ ] コード参照に依存していない
- [ ] Markdown インデントが 4 スペースで統一されている
- [ ] 関連ドキュメントと用語・方針が整合している
