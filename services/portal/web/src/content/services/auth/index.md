## nagiyu Auth とは

nagiyu Auth は、nagiyu プラットフォームを構成する各サービスに対して、統一されたシングルサインオン（SSO）を提供する認証サービスです。Google OAuth 2.0 を基盤としており、Google アカウントさえあれば各サービスへのログインが可能です。一度ログインすると、すべての nagiyu サービスをシームレスに利用できます。

## 主な機能

### Google OAuth 2.0 によるログイン

Google アカウントを使ってシンプルかつセキュアにログインできます。パスワードの管理が不要で、Google の高度なセキュリティ（二段階認証など）をそのまま活用できます。

### シングルサインオン（SSO）

一度 nagiyu Auth でログインすると、Quick Clip・Codec Converter・Stock Tracker など、nagiyu プラットフォームのすべてのサービスで再ログインなしに利用できます。

### セッション管理

ログイン状態はセキュアな HTTP-only Cookie によって管理されます。セッションの有効期限や複数デバイスでのログイン状況を管理画面から確認・制御できます。

### アカウント情報の管理

ダッシュボードからプロフィール情報の確認・更新や、連携しているサービスの確認ができます。

## セキュリティ

### OAuth 2.0 の採用

nagiyu Auth は Google OAuth 2.0 の Authorization Code Flow を採用しています。パスワードはnagiyu のサーバーに保存されず、認証情報は Google が管理します。

### HTTPS 通信

すべての通信は HTTPS（TLS）で暗号化されています。認証トークンや Cookie はセキュアフラグが設定されており、HTTPS 接続でのみ送信されます。

### CSRF 保護

クロスサイトリクエストフォージェリ（CSRF）攻撃を防ぐための対策が実装されています。

## 対象ユーザー

nagiyu Auth は、nagiyu プラットフォームの各サービスを利用するすべてのユーザーが対象です。エンドユーザーとして利用するだけでなく、開発者として nagiyu の API を利用する場合にも認証が必要になります。

## 利用の開始

[https://auth.nagiyu.com](https://auth.nagiyu.com) にアクセスし、「Google でログイン」ボタンをクリックするだけで登録・ログインが完了します。初回ログイン時に自動的にアカウントが作成されます。

## 背景・開発動機

nagiyu プラットフォームは複数のサービスで構成されており、最初は各サービスで認証を個別に実装することも考えました。しかし、サービスが増えるにつれて「ログインのたびに別々の認証画面が出る」体験の悪さが見えてきました。私が最も気になったのは、同じユーザーが同じブラウザで複数のサービスを行き来するのに、何度もログインを求められる煩わしさです。それを解消するために、認証を一か所に集約し、一度のログインでプラットフォーム全体を使える仕組みとして nagiyu Auth を作りました。パスワードをこちらのサーバーに保存しない設計を選んだのも、ユーザーの認証情報を預かる責任を最小限にしたいという個人的な判断です。Google OAuth を採用することで、Google 側の高度なセキュリティ（二段階認証や不審なログインの検知）をそのまま活用できる点も大きな理由です。

## 想定ユーザー

私が主に想定しているのは、nagiyu プラットフォームの各サービス（Quick Clip・Codec Converter・Stock Tracker など）を利用するユーザーです。「認証の設定で詰まりたくない」という人に向けて、Google アカウントを持っていればすぐ使えるよう設計しています。また、nagiyu の API を組み合わせてアプリケーションを作りたい開発者や、プラットフォームを管理するシステム管理者も想定しています。なお、Google アカウントを持っていないユーザーは現時点では利用できません。複数の OAuth プロバイダー対応は Phase 2 以降で検討しています。

## 他ツールとの違い

nagiyu Auth の最も特徴的な点は、`.nagiyu.com` ドメイン全体で JWT クッキーを共有するという設計です。これにより、Auth でログインしたセッションがそのまま admin・tools など他のサービスに自動的に引き継がれます。各サービスは Auth に問い合わせることなく独自に JWT を検証できるため、Auth が一時的に停止していても既存のセッションは有効なままです。また、ロールベースアクセス制御（RBAC）の定義をデータベースではなくコードで管理しているため、権限の変更がコードレビューを経てデプロイされます。ただし、この設計はあくまで nagiyu プラットフォームという閉じたエコシステム向けの実装であり、外部サービスとのフェデレーションや大規模な組織向け認証基盤としての利用は現時点のスコープ外です。
