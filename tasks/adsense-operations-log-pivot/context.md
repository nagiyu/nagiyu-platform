## 2. 2026-07-14 調査の結論（今回の却下原因）

**技術要件・体裁は問題ではない**（実地監査で確認済み: ads.txt / robots / sitemap / 必須ページ / タグページ・/services の 301 撤去 / 全サブドメイン noindex、すべて本番反映済み）。原因は記事コンテンツそのもの。可能性の高い順に:

1. **テンプレートの金太郎飴構造（最有力）**: 全 28 記事中 24 本が「はじめに → 本文 → 実装ノート → ハマったポイント →（現在の運用）→ まとめ」の同一 H2 シーケンス。「個人開発で運用している」が 22 記事、「本記事では」が 22 記事に反復。2026-06-22 に 25 本を一括リフレーミングした結果、updatedAt も同日に揃い、均質性がむしろ悪化。`nextjs16-metadata-api` には一人称セクションが後付けであることを自白する記述まである。
2. **本文の独自性不足**: 記事の 6〜7 割は本文主要部が公式ドキュメント・定番チュートリアルの再構成で、独自価値が末尾セクションに局所化。
3. **量産の痕跡**: 16 本が 2026-05-02 の 1 コミットで一括追加され publishedAt を 3/15〜4/30 に backdate。さらに `weekly-blog-article.yml`（2026-05-19〜）が毎週定型仕様（7〜9KB・はじめに→まとめ）で AI に記事を書かせる Issue を自動起票しており、均質記事の供給源になっている。
4. 補助要因: カテゴリ非リンク・単一一覧のみの浅いサイト構造、著者実在性シグナルの弱さ。

過去 5 回の対策はすべて「構造・体裁・量」の外形改善で、コンテンツの製造方法（テンプレ量産）には一度も手が入っていなかった。これが「対策しても通らない」の構造的敗因。

### 記事の独自性評価（改編時の資産選別に使う）

- **強い（一次情報・移設候補）**: `dynamodb-gsi-single-update-limit` / `drawio-setid-prototype-collision` / `auth-signout-reverse-proxy-redirect` / `jest-coverage-threshold` / `ecs-fargate-vs-lambda`（+ `aws-batch-architecture` の実装ノート部・`cdk-iam-least-privilege`）
- **弱い（統合・削除候補）**: `nextjs-ssg-markdown`（実装ノートで本文サンプルを自己否定）/ `eventbridge-rule-scheduling`（本文セクション実質 2 つ）/ `lambda-cold-start`（結論が「どちらも使っていない」）/ `dynamodb-single-table` / `s3-presigned-url` / `nextjs-generate-static-params` 等の定番解説群

## 3. 決定した方向性（運営者と合意済み・2026-07-14）

**案 1「個人開発の運用ログ」メディアへの改編を採用**。Portal（技術解説メディア）で審査に挑み続けるのをやめ、nagiyu.com を「AWS 上で複数サービスを運用する個人開発の公開運用記録」に再定義する。

### コンテンツの柱

- **月次 AWS コストレポート**: 実際の請求データ・サービス別内訳・先月比・削減施策の結果
- **障害・ヒヤリハット記録**: `dynamodb-gsi` / `drawio-setid` 型の実障害起点記事（監査で独自性トップだった型）
- **設計判断の公開**: リポジトリ既存の ADR の一般向け翻訳
- リリースノート解説・四半期ふりかえり

### この方向が強い理由

- 全記事が自分にしか書けない実データを含み、**Information Gain がジャンルレベルで構造的に保証される**（テンプレ構成でも中の数値が毎回違う実測値なので均質判定されにくい）
- 実は #3262 の調査コメント（2026-05-26）で立てた「玄関＋制作日誌・運用記録のハイブリッド」仮説と同じ方向（当時は採用されず「記事を足す」路線が継続された）
- Portal の Next.js SSG 基盤・必須ページ・構造化データ・E-E-A-T 外形をそのまま流用できる
- **資産の廃棄はほぼ発生しない**: 「Portal を捨てる」のではなく看板を「技術メディア」→「運用記録」に掛け替え、強い記事（上記リスト）は移設・統合する

### 運用上の必須事項

- **`weekly-blog-article.yml` の現行運用（定型 AI 記事の週次量産）は停止する**。ただし廃止ではなく「Cost Explorer から実データを取得して月次コストレポートの下書きを生成する」用途への転用を検討（自動化を弱点から資産に変える）
- 公開記事は「実データ＋人の判断コメント」構成とし、テンプレ H2 シーケンスの一律適用をやめる
- ドメインは **nagiyu.com のまま**（AdSense 審査はルートドメイン単位で申請時点の現状を見る。却下履歴によるドメインの恒久的不利は公式仕様として存在しない）

### 却下した代替案（再提案しないこと）

- **案 2: データ/リファレンスサイト（料金早見等）**: ツール型・公式情報再掲型は AdSense と相性が悪く、Tools で既に 2 敗した路線の変形のため不採用
- **案 3: 生活・趣味系の別ジャンルブログ**: AI に書かせれば同じ壁、手書きなら技術資産が活きないため不採用
- **新ドメインでの再挑戦**: AdSense はサイト（ルートドメイン）ごとに追加審査のため、新ドメインで合格しても nagiyu.com には効かない。不採用
