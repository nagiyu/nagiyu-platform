# 週次ドキュメントレビュー 第13週（2026-03-30） - 要件定義

---

## 1. ビジネス要件

### 1.1 背景・目的

[Issue #2474](https://github.com/nagiyu/nagiyu-platform/issues/2474) は 2026年第13週（2026-03-30）の週次ドキュメントレビューである。
実装時のPRでは気づきにくいドキュメント間の**構造的な問題**と**二重管理の不整合**を発見・修正する。

### 1.2 対象ユーザー

- **プライマリー**: プラットフォーム開発者（コーディング規約・CI/CDルールの整合性を保つ）
- **セカンダリー**: Copilot エージェント（正確なルールを参照してコードを生成する）

### 1.3 ビジネスゴール

- ドキュメント間の不整合を解消し、ルールの信頼性を維持する
- 開発者が正しいルールを参照できる状態にする

## 2. 機能要件

### 2.1 ユースケース

#### UC-001: Priority 1 - 二重管理の整合性チェック（必須）

- **概要**: 二重管理されているドキュメント間の整合性を確認・修正する
- **アクター**: プラットフォーム開発者
- **前提条件**: 対象ドキュメントが存在する
- **正常フロー**:
    1. 各ドキュメントペアを比較する
    2. 不整合箇所を特定する
    3. Single Source of Truth（SSoT）を確認し、従属側ドキュメントを修正する
- **代替フロー**: 不整合がない場合はチェック完了としてマークする
- **例外フロー**: SSoT が不明確な場合は議論して決定する

#### UC-002: Priority 2 - 構造的整合性チェック（推奨）

- **概要**: モノレポ構成・ブランチ戦略・テストデバイス設定の整合性を確認・修正する
- **アクター**: プラットフォーム開発者
- **前提条件**: 対象ドキュメントが存在する
- **正常フロー**:
    1. 関連ドキュメント間の記述を比較する
    2. 実際のファイル構成・CI設定と照合する
    3. 不整合を修正する

#### UC-003: Priority 3 - ドキュメント間整合性チェック（推奨）

- **概要**: 広範なドキュメント間のルール記載・リンク・重複の整合性を確認する
- **アクター**: プラットフォーム開発者
- **前提条件**: 対象ドキュメント群が存在する
- **正常フロー**:
    1. 対象ファイル群でルール記述を横断確認する
    2. 矛盾・リンク切れ・意図しない重複を特定する
    3. 必要に応じて修正または参照リンク化する

#### UC-004: Priority 4 - 実装との乖離チェック（任意）

- **概要**: ドキュメントの記述が実際の実装・インフラ構成と乖離していないか確認する
- **アクター**: プラットフォーム開発者
- **前提条件**: 対象ドキュメントと実装が存在する
- **正常フロー**:
    1. ドキュメントが記述する構成と実際のコードを照合する
    2. 乖離があれば修正する

## 3. 非機能要件

### 3.1 保守性・拡張性要件

- 修正後のドキュメントは、次回の週次レビューで再発しない品質を保つ
- SSoT（Single Source of Truth）を明確にし、更新コストを下げる

---

## 4. ドメインオブジェクト

| エンティティ | 説明 |
| ----------- | ---- |
| SSoT ドキュメント | ルールの正式な定義元（例: `docs/development/rules.md`） |
| 従属ドキュメント | SSoT を参照・転記しているドキュメント（例: `copilot-instructions.md`） |
| 不整合 | 同一ルールに対して複数ドキュメント間で内容が異なる状態 |

---

## 5. スコープ外

- ❌ 実装コードの修正（ドキュメントのみ対象）
- ❌ 新規ルールの策定（既存ルールの整合性のみ確認）
- ❌ テストカバレッジの実際の向上（設定・ドキュメントの整合のみ確認）
