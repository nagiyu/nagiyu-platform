# Stock Tracker アラート改善

## 概要

Stock Tracker のアラート編集機能に複数のバグが存在する。
アラートのデフォルト値・フォールバック回りの設計を見直し、以下の問題を解消する。

## 関連情報

- Issue: #2216
- タスクタイプ: サービスタスク（stock-tracker）
- 対象: `services/stock-tracker/`

## 現状の問題

### 問題1: 本文編集中のフリーズ

アラートのモーダルで通知本文を編集しているとフリーズすることがある。
フリーズ後は保存ボタンや他のセレクトボックスが機能しなくなる。

**原因の仮説:**

- `AlertSettingsModal.tsx` では 21 個の `useState` を個別に管理しており、
  `applyFormData()` が各 setter を順次実行するため再レンダリングが連鎖している。
- 通知本文の入力時は `setNotificationBody` と `clearFieldError` の 2 回 setState が呼ばれ、
  コンポーネント全体が再レンダリングされる。

### 問題2: 条件変更時にデフォルト値が更新されない

条件（operator）や目標価格を変更しても、アラート本文のデフォルト値が変わらない。

**原因:**

- `buildFormData()` は `useEffect` 内でモーダル開閉時のみ実行される。
- 編集モードでは `editTarget.notificationBody || defaultNotificationText.body` の形でデフォルト値を設定するが、
  既存値がある場合は OR の右辺（デフォルト値）が評価されないため、
  条件変更後もデフォルト値が再計算・反映されない。

### 問題3: アラートを編集して保存しても元に戻る

既存アラートのタイトル・本文を編集して保存しても、再度開くと元に戻っている。

**原因:**

- PUT API の更新ロジックで `notificationTitle.trim() || undefined` として
  空文字列を `undefined` に変換している。
- `validateAlert()` ではタイトル・本文のバリデーションが実装されていないため、
  空でも保存がそのまま通ってしまう。
- フロントエンド側でユーザー入力が空になった場合に undefined を送信し、
  サーバー側がフォールバックを行う前提となっているが、実際にはその処理がない。

## 要件

### 機能要件

- FR1: サーバー側でフォールバックを行わない。タイトルまたは本文が空の場合はエラーを返す。
- FR2: クライアント側でもタイトル・本文が空の場合はバリデーションエラーを表示する。
- FR3: 条件（operator, targetPrice など）を変更した場合、
      アラート本文の状態に関わらずデフォルト値で上書きする。
    - ただし状態遷移が複雑になる場合は「デフォルトの本文を設定するボタン」で代替する。
- FR4: アラートの編集・保存後に再度開いた時、保存した値が正しく反映されていること。

### 非機能要件

- NFR1: テストカバレッジ 80%以上を維持すること（`core/src/` を重点的にカバー）。
- NFR2: TypeScript strict mode 準拠。
- NFR3: エラーメッセージは日本語で定数化すること（`ERROR_MESSAGES` オブジェクト）。

## 実装のヒント

### FR1・FR2: タイトル・本文の必須バリデーション追加

- `core/src/validation/index.ts` の `validateAlert()` にタイトル・本文の必須チェックを追加する。
    - `notificationTitle` が未定義または空文字列の場合はエラーを返す。
    - `notificationBody` が未定義または空文字列の場合はエラーを返す。
- `web/app/api/alerts/[id]/route.ts` の PUT ハンドラーにおいて、
  空文字列を `undefined` に変換する処理（`.trim() || undefined`）を見直す。
    - 空文字列はそのままバリデーションに渡し、バリデーション側でエラーにする。
- `AlertSettingsModal.tsx` のクライアント側バリデーションも同様に、
  タイトル・本文が空の場合はフィールドエラーを設定してフォーム送信を阻止する。

### FR3: 条件変更時のデフォルト値自動更新

Issue 提案の方針: 条件変更時に常にデフォルト値で上書きする。

- 条件フィールド（conditionMode, operator, targetPrice, rangeType, minPrice, maxPrice 等）の
  変更を検知し、`getDefaultNotificationText()` の結果を `notificationTitle`・`notificationBody` に反映する。
- 代替案として、フォームに「デフォルト値を設定」ボタンを追加し、
  クリック時にデフォルト値を再計算・設定する方式も許容する。
  自動上書き方式では「ユーザーが手動で本文を編集した直後に条件変更した場合も上書きされる」
  という副作用が発生し、入力が多くのイベントハンドラーと絡み合う場合は
  「どのイベント後に自動更新するか」の判定が複雑になる。そのような場合はボタン方式で回避する。

### 問題1のフリーズ解消（任意）

- フリーズの再現条件を特定した上で対応する。
- 状態管理のリファクタリングが必要な場合は `useReducer` への移行を検討するが、
  スコープが大きくなるため、別 Issue として切り出すことも可。

### FR4: 保存後の値の整合性

- FR1・FR2 の対応により空文字列のまま保存されなくなることで、
  再度開いた時に元に戻る問題は解消される見込み。
- PUT API が正常に更新を反映できていることをユニットテストで確認する。
