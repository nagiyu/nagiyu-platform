# Issue #2342 週次ドキュメントレビュー対応 - 要件定義

---

## 1. ビジネス要件

### 1.1 背景・目的

週次ドキュメントレビュー（Issue #2342）の調査により、以下の 2 点が発見された。

1. **リンク切れチェックスクリプトのバグ**（最重要）
   `.github/workflows/templates/weekly-review-body.md` に掲載されている「ドキュメント間リンク切れ確認コマンド」が、相対パスの解決ロジックに誤りを含んでいる。
   結果として週次レビューのたびに 70 件以上の false positive（実際にはリンク切れではない検出）が発生し、レビュアーが不必要な調査コストを払い続けている。

2. **バッチパッケージのカバレッジ例外が testing.md に未記載**
   `services/niconico-mylist-assistant/batch/jest.config.ts` の `coverageThreshold` が意図的に未設定であるにもかかわらず、その設計判断が `docs/development/testing.md` に明記されていない。
   週次レビューの「Jest Coverage ⇄ testing.md 整合性」チェックで毎回疑義が生じるリスクがある。

本作業はこれらの問題を修正・明文化し、週次ドキュメントレビューの品質と効率を継続的に向上させることを目的とする。

### 1.2 対象ユーザー

- **プライマリー**: 週次ドキュメントレビューを担当する開発者・レビュアー
- **セカンダリー**: プラットフォーム全体のドキュメントを参照する開発者

### 1.3 ビジネスゴール

- 週次レビュー Issue に含まれるリンク切れ false positive をゼロにする
- テスト戦略の例外設計をドキュメントに明記し、将来の誤解・問い合わせコストを削減する

## 2. 機能要件

### 2.1 ユースケース

#### UC-001: 週次レビューでリンク切れチェックを実行する

- **概要**: レビュアーが `weekly-review-body.md` に掲載された確認コマンドを手元または CI で実行し、真のリンク切れのみを検出する
- **アクター**: 週次ドキュメントレビュー担当者
- **前提条件**: `docs/` 配下に相対パスでリンクが記述されたドキュメントが存在する
- **正常フロー**:
    1. 担当者が確認コマンドを実行する
    2. スクリプトが各 Markdown ファイルの位置を基準として相対パスを解決する
    3. 実在しないリンク先のみ「リンク切れ」として出力される
    4. 担当者が出力された真のリンク切れを調査・修正する
- **代替フロー**: リンク切れが 0 件の場合、何も出力されず正常終了する
- **例外フロー**: スクリプト実行エラー時はエラーメッセージを出力して異常終了する

#### UC-002: バッチパッケージのカバレッジ例外を確認する

- **概要**: 開発者が `docs/development/testing.md` を参照してバッチパッケージのカバレッジ設定の正当性を確認できる
- **アクター**: 開発者・週次ドキュメントレビュー担当者
- **前提条件**: `testing.md` にバッチパッケージの扱いに関するセクションが存在する
- **正常フロー**:
    1. 開発者が `testing.md` のカバレッジ方針セクションを参照する
    2. バッチパッケージは `coverageThreshold` を設定しない設計判断とその理由が記載されている
    3. 開発者が意図を理解した上でバッチパッケージのテスト戦略を判断できる
- **例外フロー**: なし

### 2.2 機能一覧

| 機能ID | 機能名 | 説明 | 優先度 |
| ------ | ------ | ---- | ------ |
| F-001 | リンク切れチェックスクリプト修正 | `weekly-review-body.md` の確認コマンドを相対パスを正しく解決するロジックに差し替える | 高 |
| F-002 | testing.md カバレッジ例外の明記 | バッチパッケージが `coverageThreshold` を設定しない設計判断とその理由を `testing.md` に追記する | 中 |

### 2.3 想定画面の概要

本タスクは UI を持たない。変更対象はすべて Markdown ドキュメントおよびスクリプトファイルである。

## 3. 非機能要件

### 3.1 パフォーマンス要件

| 項目 | 要件 |
| ---- | ---- |
| 実行時間 | `docs/` 全体のリンクチェックを 30 秒以内に完了する |
| 対象ファイル数 | 現状の `docs/` ファイル数（数十〜数百ファイル）を問題なく処理できる |

### 3.2 セキュリティ要件

- スクリプトはファイルシステムの読み取り専用操作のみ実施する
- 外部ネットワーク通信は行わない
- シェルインジェクションに対して安全な実装パターンを選択する

### 3.3 可用性要件

| 項目 | 要件 |
| ---- | ---- |
| スクリプト実行環境 | GitHub Actions の標準ランナー（ubuntu-latest）および開発者のローカル環境で動作すること |
| 依存ランタイム | Python 3 または Node.js（いずれも ubuntu-latest に標準搭載）に限定する |

### 3.4 保守性・拡張性要件

- スクリプトを `.github/workflows/scripts/` に分離する場合、スタンドアロンで実行可能な形式とする
- `weekly-review-body.md` への掲載コマンドは簡潔に保ち、複雑なロジックは外部スクリプトに委譲する
- テストカバレッジ例外の記述は `testing.md` の既存構造に沿って追記し、可読性を損なわないようにする

## 4. ドメインオブジェクト

| エンティティ | 説明 |
| ----------- | ---- |
| Markdown リンク | `[テキスト](パス.md)` 形式で記述された docs/ 内の相対パスリンク |
| リンク起点ファイル | リンクを記述している Markdown ファイル。相対パス解決の基準となる |
| リンク先ファイル | リンクが指す Markdown ファイル。起点ファイルの位置からの相対パスで解決される |
| カバレッジ例外 | 設計上の理由により `coverageThreshold` を設定しないパッケージの扱いに関するルール |

## 5. スコープ外

- ❌ 外部サイト（https://〜）へのリンク切れチェック
- ❌ アンカーリンク（`#section-name`）の有効性チェック
- ❌ `docs/` 以外のディレクトリ（`tasks/`, `services/` 等）のリンクチェック
- ❌ niconico-mylist-assistant/batch 以外のバッチパッケージのカバレッジ設定変更
- ❌ 週次レビュー workflow の自動化・CI 統合（スクリプト修正のみ対応）

## 6. 用語集

| 用語 | 定義 |
| ---- | ---- |
| false positive | 実際にはリンク切れでないにもかかわらず、スクリプトが「リンク切れ」と誤検出すること |
| 相対パス解決 | ファイルの格納場所を基準として相対パス（`../../README.md` 等）を絶対パスに変換する処理 |
| Fast CI | 短時間で完了する CI ジョブ群。バッチパッケージの統合テストは対象外とする設計 |
| weekly-review-body.md | 週次ドキュメントレビュー Issue の本文テンプレート。確認コマンドやチェックリストを含む |
