# 定期 Issue の改善 - 要件定義

## 1. ビジネス要件

### 1.1 背景・目的

現在、以下の定期実行ワークフローが Issue を自動作成している:

- `daily-refactoring-check.yml` — 日次コード共通化調査
- `weekly-npm-check.yml` — 週次 npm パッケージ管理
- `weekly-docs-review.yml` — 週次ドキュメントレビュー

これらのワークフローが作成する Issue の本文には、エージェントへの作業指示が直接埋め込まれている。
しかし、Issue はあくまで「事実の記録」にとどめ、エージェントへの指示は別の仕組みで分離することが望ましい。

また、`create-task-issues.yml` は手動実行のみで、定期 Issue からは呼び出されていない。
定期 Issue についても同ワークフローを通じてサブ Issue（タスクドキュメント作成・実装・クリーンアップ）を自動生成し、各エージェントに割り当てることで、開発フロー全体の一貫性を確保する。

### 1.2 対象ユーザー

- 定期 Issue を起点に作業を進めるエージェント（task.proposal, task.implement 等）
- ワークフローを管理・運用するリポジトリオーナー

### 1.3 ビジネスゴール

- Issue 本文を「事実のみ」に統一し、エージェント指示と分離する
- 定期 Issue 作成後に自動でサブ Issue を作成し、各エージェントを割り当てる
- 手動で `create-task-issues.yml` を呼び出す手順を不要にする

## 2. 機能要件

### 2.1 ユースケース

#### UC-001: 定期 Issue の事実のみ記述

- **概要**: 定期ワークフローが作成する Issue 本文に、エージェント指示を含めない
- **アクター**: GitHub Actions（定期ワークフロー）
- **正常フロー**:
    1. 定期ワークフローが実行される
    2. 調査・収集した事実情報（診断結果、メトリクス等）のみで Issue 本文を生成する
    3. Issue を作成する

#### UC-002: 定期 Issue 作成後のサブ Issue 自動生成

- **概要**: 定期 Issue 作成後、`create-task-issues.yml` を呼び出してサブ Issue を自動生成する
- **アクター**: GitHub Actions（定期ワークフロー → create-task-issues.yml）
- **正常フロー**:
    1. 定期ワークフローが親 Issue を作成する
    2. 作成した Issue 番号を後続ジョブに渡す
    3. `create-task-issues.yml`（再利用可能ワークフロー）を呼び出す
    4. 3 つのサブ Issue（タスクドキュメント作成・実装・クリーンアップ）が親 Issue のサブ Issue として作成される
    5. 各サブ Issue が適切なエージェントにアサインされる

#### UC-003: サブ Issue から親 Issue を参照

- **概要**: サブ Issue の本文に親 Issue への参照リンクが含まれる
- **アクター**: GitHub Actions（create-task-issues.yml）
- **正常フロー**:
    1. サブ Issue 作成時に親 Issue の番号を本文の「参照:」欄に設定する
    2. エージェントがサブ Issue を参照した際に、親 Issue の事実情報を辿れる

### 2.2 機能一覧

| 機能ID | 機能名 | 説明 | 優先度 |
| ------ | ------ | ---- | ------ |
| F-001 | Issue 本文の事実化 | 定期ワークフローの Issue 本文からエージェント指示を除去し、事実情報のみにする | 高 |
| F-002 | `create-task-issues.yml` の再利用化 | `workflow_call` トリガーを追加し、他ワークフローから呼び出せるようにする | 高 |
| F-003 | 定期ワークフローからのサブ Issue 自動生成 | 定期ワークフローが Issue 作成後に `create-task-issues.yml` を呼び出す | 高 |
| F-004 | サブ Issue への親 Issue 参照追加 | サブ Issue 本文の「参照:」欄に親 Issue 番号を自動設定する | 中 |

## 3. 非機能要件

### 3.1 信頼性・エラーハンドリング

- `create-task-issues.yml` の呼び出し失敗は `set -euo pipefail` で検知し、ステップを失敗扱いにする
- 既存のエラーハンドリング方針（各 Issue 作成ステップで `::error::` 出力）を維持する

### 3.2 保守性

- 定期ワークフロー間で重複するサブ Issue 作成ロジックを `create-task-issues.yml` に集約する
- Issue 本文テンプレートはファイル分離を維持する

### 3.3 後方互換性

- `create-task-issues.yml` の `workflow_dispatch` による手動実行を引き続き利用できること
- 既存の `mode` 選択肢（`task-document-completion`）が変更なく動作すること

---

## 4. ドメインオブジェクト

| エンティティ | 説明 |
| ------------ | ---- |
| 親 Issue | 定期ワークフローが作成する事実記録用 Issue |
| サブ Issue | 親 Issue に紐付く作業用 Issue（タスクドキュメント作成・実装・クリーンアップ） |
| 再利用可能ワークフロー | `workflow_call` で呼び出せる `create-task-issues.yml` |

---

## 5. スコープ外

- ❌ 各サブ Issue の具体的な本文内容（指示文）の変更（既存の文言を流用）
- ❌ 定期ワークフローのスケジュール・重複チェックロジックの変更
- ❌ サブ Issue のラベル設定
- ❌ 新しい定期ワークフローの追加
