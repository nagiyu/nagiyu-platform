# さくっとクリップ - Batch ジョブサイジング修正 要件定義

---

## 1. ビジネス要件

### 1.1 背景・目的

7.76 GB の動画をアップロードして処理を依頼すると、AWS Batch ジョブが 3 時間でタイムアウトして失敗する問題が発生している。

原因は以下の 2 点:

1. **ティア境界値の不整合**: `selectJobDefinition` は `< 8 GiB` を `large` (2 vCPU, 3 時間タイムアウト) に分類する。7.76 GB ≈ 7.23 GiB は `large` に分類されるが、2 vCPU では 3 時間以内に処理が完了しない。

2. **FFmpeg 並列プロセス数が vCPU 数を超過**: 処理パイプラインは常時 4 つの FFmpeg プロセスを並列実行する。`large` (2 vCPU) では各プロセスの実効 CPU が 0.5 vCPU 相当に落ち、処理時間が大幅に延びる。

本修正により、ユーザーがアップロードした動画ファイルが適切なリソースで処理され、タイムアウトによる失敗を解消する。

### 1.2 対象ユーザー

さくっとクリップを利用する一般ユーザー（大容量動画をアップロードするケース）。

### 1.3 ビジネスゴール

- 7.76 GB 程度の動画ファイルがタイムアウトせず正常に処理されること
- ジョブ定義のティア構成が FFmpeg の並列実行数と整合していること

## 2. 機能要件

### 2.1 ユースケース

#### UC-001: 大容量動画の見どころ抽出

- **概要**: 1 GiB 以上の動画ファイルをアップロードし、見どころ抽出バッチが正常完了すること
- **アクター**: ユーザー
- **前提条件**: 動画ファイルが S3 にアップロード済み
- **正常フロー**:
    1. ファイルサイズに応じたジョブ定義が選択される
    2. AWS Batch Fargate ジョブが起動する
    3. 動画ファイルをダウンロードし FFmpeg で解析する
    4. 見どころが DynamoDB に保存される
    5. ジョブが COMPLETED ステータスになる
- **例外フロー**: ファイルサイズが境界値を超える場合、より上位のティアが選択されること

### 2.2 機能一覧

| 機能ID | 機能名 | 説明 | 優先度 |
| ------ | ------ | ---- | ------ |
| F-001 | ティア境界値の修正 | `large` の上限を 8 GiB → 4 GiB に変更する | 高 |
| F-002 | `large` ジョブの vCPU 引き上げ | `large` ジョブ定義の vCPU を 2 → 4 に変更する | 高 |

---

## 3. 非機能要件

### 3.1 パフォーマンス要件

| 項目 | 要件 |
| ---- | ---- |
| large ジョブ (1〜4 GiB) | 3 時間タイムアウト以内に処理完了 |
| xlarge ジョブ (4 GiB 以上) | 8 時間タイムアウト以内に処理完了 |

### 3.2 セキュリティ要件

変更なし。

### 3.3 可用性要件

変更なし。

### 3.4 保守性・拡張性要件

- ティア境界値はコード上の定数として明確に定義し、変更理由をコメントで補足すること
- ADR として設計決定を記録すること

---

## 4. ドメインオブジェクト

| エンティティ | 説明 |
| ------------ | ---- |
| JobDefinitionSize | バッチジョブのサイズティア (`small` / `large` / `xlarge`) |

---

## 5. スコープ外

- ❌ FFmpeg 並列実行構造のリファクタリング（処理ロジック変更は対象外）
- ❌ `small` ジョブ定義の変更（1 GiB 未満は現状の 1 vCPU で十分）
- ❌ タイムアウト値の変更
- ❌ メモリ割り当ての変更
