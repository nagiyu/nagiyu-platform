# stock-tracker 一時アラート失効バッチ バグ修正 要件定義書

---

## 1. ビジネス要件

### 1.1 背景・目的

StockTracker の「一時アラート失効バッチ」（`temporary-alert-expiry`）が本番環境で失敗している。
バッチは取引終了後に期限切れとなった一時通知アラートを自動無効化する役割を担っているが、
DynamoDB に格納された一部アラートのサブスクリプションデータが無効なため、バッチ全体が停止してしまっている。

バッチが停止することで、以下の問題が発生する。

- 期限切れの一時アラートが無効化されず、意図しない通知が送信され続ける可能性がある
- 手動対応が必要となり、運用コストが増大する

本タスクでは根本原因を特定し、バッチが個別の不正データに影響されることなく安定稼働できるよう修正する。

### 1.2 対象ユーザー

- **プライマリー**: StockTracker を利用する一般ユーザー（一時アラートを設定しているユーザー）
- **セカンダリー**: 運用・保守担当者

### 1.3 ビジネスゴール

- 一時アラート失効バッチが毎時安定して実行されること
- 不正データが含まれていても、有効なアラートの処理が継続されること

## 2. 機能要件

### 2.1 ユースケース

#### UC-001: 一時アラートの失効処理（正常系）

- **概要**: 取引終了後、期限切れの一時アラートを自動無効化する
- **アクター**: EventBridge Scheduler（毎時起動）
- **前提条件**: DynamoDB に有効な一時アラートが存在する
- **正常フロー**:
    1. `MINUTE_LEVEL` および `HOURLY_LEVEL` の全アラートを取得する
    2. 各アラートを評価し、有効・期限切れ・取引時間外を確認する
    3. 条件を満たすアラートの `Enabled` を `false` に更新する
    4. 処理統計をログ出力して正常終了する
- **代替フロー**: 対象アラートがなければスキップして正常終了する
- **例外フロー**: 個別アラートの処理に失敗した場合はログ出力してスキップし、後続アラートの処理を継続する

#### UC-002: 不正データのスキップ（バグ修正対象）

- **概要**: サブスクリプションデータが欠損・不正なアラートが存在しても、バッチ全体を停止させない
- **アクター**: EventBridge Scheduler（毎時起動）
- **前提条件**: DynamoDB に不正なサブスクリプションデータを持つアラートが存在する
- **正常フロー**:
    1. バッチがアラートを取得・マッピングする際に不正データを検出する
    2. 不正なアイテムの情報（`AlertID`、`UserID`）をワーニングとしてログ出力する
    3. 当該アイテムをスキップし、後続アイテムの処理を継続する
    4. 処理統計に `skippedInvalidData` としてカウントアップする
- **代替フロー**: なし
- **例外フロー**: なし

### 2.2 機能一覧

| 機能ID | 機能名 | 説明 | 優先度 |
| ------ | ------ | ---- | ------ |
| F-001 | 不正データ耐性 | マッピング失敗アイテムをスキップし処理継続する | 高 |
| F-002 | ページネーション対応 | `getByFrequency` の結果が 50 件を超える場合も全件取得する | 中 |

### 2.3 想定画面の概要

- 画面変更なし（バッチ処理のみ）

## 3. 非機能要件

### 3.1 パフォーマンス要件

| 項目 | 要件 |
| ---- | ---- |
| 応答時間 | Lambda タイムアウト（現行設定値）以内に完了すること |
| スループット | 全アラートを 1 実行で処理できること（ページネーション対応） |

### 3.2 セキュリティ要件

- 不正データのログ出力において、個人情報（subscription の endpoint 等）を含めない
- 既存のアクセス制御・IAM 権限設定を変更しない

### 3.3 可用性要件

| 項目 | 要件 |
| ---- | ---- |
| 稼働率 | バッチが毎時正常に実行されること |
| RTO | 不正データが発生しても次の実行サイクルまでに自動回復すること |
| RPO | 不正データアイテムを除く全有効アラートが処理されること |

### 3.4 保守性・拡張性要件

- 修正はバッチ処理と repository 層に限定し、最小変更とする
- 不正データのログにより、後続の DynamoDB データ修正作業をサポートできること

## 4. ドメインオブジェクト

| エンティティ | 説明 |
| ----------- | ---- |
| Alert | 株価アラートの設定情報。通知条件・頻度・一時フラグを含む |
| AlertEntity | DynamoDB のキーを持たないビジネスオブジェクト |
| PushSubscription | Web Push 通知用のサブスクリプション情報（endpoint / keys） |

---

## 5. スコープ外

- ❌ 不正なサブスクリプションデータの自動修復（バックフィル）
- ❌ subscription 形式の移行（旧形式フォールバックの削除）※ 別 Issue として追跡すること
- ❌ UI・API エンドポイントの変更
- ❌ 日次バッチ（`daily.ts`）の修正（別タスクで対応）

---

## 6. 用語集

| 用語 | 定義 |
| ---- | ---- |
| 一時アラート | 取引終了後に自動で無効化される期限付きアラート（`Temporary: true`） |
| 一時失効バッチ | `temporary-alert-expiry` Lambda。毎時起動し期限切れ一時アラートを無効化する |
| 旧形式サブスクリプション | `SubscriptionEndpoint` / `SubscriptionKeysP256dh` / `SubscriptionKeysAuth` のフラット形式（移行中の後方互換） |
| 新形式サブスクリプション | `subscription.endpoint` / `subscription.keys.p256dh` / `subscription.keys.auth` のネスト形式 |
