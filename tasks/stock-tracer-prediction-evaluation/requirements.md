## 1. ビジネス要件

### 1.1 背景・目的

Stock Tracer は OpenAI による日次 AI 解析で銘柄ごとに投資判断シグナル（`BULLISH / NEUTRAL / BEARISH`）を生成しているが、生成された予測が実際に当たったか外れたかを自動的に追跡する仕組みが存在しない。

このため、

- 現在の AI 精度を定量的に把握できない
- AI モデルやプロンプトを改善しても効果測定ができない
- 蓄積される予測データが学習素材として活用されていない

本タスクでは予測 → 翌日実績の自動採点 + 可視化基盤を構築し、後続フェーズでの AI 精度改善（教訓蒸留 / Few-shot RAG / ファインチューニング）に必要なデータと改善ループを成立させる。

### 1.2 対象ユーザー

- **プライマリーユーザー**: Stock Tracer の運用者（精度を確認し、AI 改善方針を意思決定）
- **セカンダリーユーザー**: 一般利用者（自分が見ている銘柄の予測精度を参考情報として確認）

### 1.3 ビジネスゴール

- AI 予測精度の定量的可視化を実現する
- AI 改善施策の効果測定ができる状態にする
- 後続フェーズ（教訓蒸留・RAG・ファインチューニング）の入力となる学習データを蓄積する

## 2. 機能要件

### 2.1 ユースケース

#### UC-001: 自動採点

- **概要**: 既存 AI 解析バッチが生成した予測を、翌営業日終値が確定した時点で自動的に採点する
- **アクター**: システム（採点バッチ Lambda）
- **前提条件**:
    - 対象予測の `AiAnalysisResult` が存在し、`AiAnalysisError` を持たない
    - 対象予測の取引所の翌営業日が引け済み
- **正常フロー**:
    1. EventBridge cron（1 時間毎）が採点バッチ Lambda をトリガー
    2. バッチが「未採点 & 翌営業日引け済」の予測を抽出
    3. 各予測について、当日終値と翌営業日終値を TradingView API から取得
    4. リターン値と Hit/Miss を判定し、対象 `DailySummaryEntity` の Evaluation\* フィールド（`EvaluationDate`、`EvaluationClose`、`ActualReturn`、`Hit`、`EvaluationThresholdPercent`、`EvaluatedAt`）を条件付き UpdateItem で書き込む
- **代替フロー**: 翌営業日が祝日・週末の場合、その次の営業日の終値で判定する
- **例外フロー**:
    - TradingView API 呼び出しが失敗した場合：当該予測は未採点のまま次回再試行
    - 対象予測が `AiAnalysisError` を持つ場合：採点対象から除外（採点フラグを「除外」として記録）

#### UC-002: 精度ダッシュボード閲覧

- **概要**: 利用者がダッシュボードで AI 予測の精度を期間別に確認する
- **アクター**: 認証済みユーザー
- **前提条件**: 既存認証フローでログイン済み
- **正常フロー**:
    1. ユーザーがダッシュボード画面を開く
    2. デフォルトで「直近 30 日」の集計を表示
    3. ページ見出し直下の主要指標テキスト（期間ラベル + 方向精度 + 採点件数 + 総合精度）が更新される
    4. 期間（7/30/90/全期間）を切り替えると主要指標テキスト + 日次推移 + シグナル別の精度が再集計される
- **代替フロー**: 採点済み予測がない場合、主要指標テキストが「期間ラベル: 採点済みの予測がありません」になる
- **例外フロー**: API エラー時は ErrorAlert を表示（再読み込みはブラウザのリロード操作で対応）

### 2.2 機能一覧

| 機能 ID | 機能名 | 説明 | 優先度 |
|---------|--------|------|--------|
| F-001 | 翌営業日終値の取得 | TradingView API から指定銘柄・日付の終値を取得 | 高 |
| F-002 | リターン計算 | 当日終値と翌営業日終値からリターン値（%）を計算 | 高 |
| F-003 | Hit/Miss 判定 | 予測シグナルとリターン値・閾値（0.5%）から Hit/Miss を判定 | 高 |
| F-004 | 採点結果の永続化 | 既存 `DailySummaryEntity` に Evaluation\* フィールドとして DynamoDB に保存（生リターン値含む。条件付き UpdateItem で二重採点を防止） | 高 |
| F-005 | 採点バッチの自動起動 | EventBridge cron で 1 時間毎に採点バッチを起動 | 高 |
| F-006 | 未採点予測の抽出 | 未採点 & 翌営業日引け済の予測を効率的に抽出 | 高 |
| F-007 | 精度集計 API | 期間を指定して集計値（KPI + 日次推移 + シグナル別）を返す API | 高 |
| F-008 | 主要指標テキスト表示 | ページ見出し直下に `期間ラベル + 方向精度 + 採点件数 + 総合精度` の 1 行テキスト。判定済み 0 件のときは「採点済みの予測がありません」 | 高 |
| F-009 | 日次推移グラフ表示 | 日次の方向精度を折れ線、判定済み件数を棒（右軸）で表示。本ダッシュボードの中心指標 | 高 |
| F-010 | シグナル別集計表示 | BULLISH / NEUTRAL / BEARISH ごとのヒット率と件数を棒グラフ + 数値テーブルで表示 | 高 |

## 3. 非機能要件

### 3.1 パフォーマンス要件

| 項目 | 要件 |
|------|------|
| 採点バッチ実行時間 | 全銘柄の採点処理が 5 分以内に完了する |
| 採点遅延 | 翌営業日引け後、最大 1 時間以内に採点が反映される（cron 起動間隔依存） |
| ダッシュボード API 応答時間 | 1 秒以内（直近 90 日分集計時） |
| ダッシュボード初回表示 | 3 秒以内 |

### 3.2 セキュリティ要件

- ダッシュボード閲覧は既存認証フローに準拠（未認証アクセス不可）
- 採点 API も既存 web の認証を共通に使用
- DynamoDB の暗号化は既存テーブル設定を踏襲

### 3.3 可用性要件

| 項目 | 要件 |
|------|------|
| 採点バッチ失敗時 | 1 回の失敗で再試行が次回 cron で自動的に行われる（冪等性） |
| データ整合性 | 同一予測に対して採点が重複しないこと |

### 3.4 テスト要件

- ユニットテストカバレッジ：80% 以上（`coverageThreshold` で自動失敗する設定を維持）
- 純粋関数（判定ロジック）は副作用モックなしで網羅テスト
- TradingView API 呼び出しはモックで分離テスト
- E2E：ダッシュボード基本表示の Fast CI 用シナリオを追加（`chromium-mobile`）

## 4. 採点ルール仕様

### 4.1 判定基準（Phase 1）

| 予測シグナル | 成功条件 |
|-------------|---------|
| BULLISH | 翌営業日終値リターン ≥ +0.5% |
| BEARISH | 翌営業日終値リターン ≤ -0.5% |
| NEUTRAL | 翌営業日終値リターンが -0.5% より大きく、+0.5% より小さい |

- リターン基準点：当日終値（close-to-close）
- 保有期間：1 営業日
- リターン計算式：`(翌営業日終値 - 当日終値) / 当日終値 × 100`（%）
- 境界値（厳密に +0.5% / -0.5% / -0.5% / +0.5%）の扱いは `design.md` で定義

### 4.2 採点対象から除外するケース

- 既存 `DailySummaryEntity.AiAnalysisError` が存在する予測
- 翌営業日終値が取得できない予測（上場廃止・データ欠損など。`design.md` でリトライ戦略を定義）

### 4.3 集計指標

Phase 1 の UI に出すのは以下のみ：

| 指標名 | 定義 | UI 出し方 |
|--------|------|---------|
| 総合精度 | 全予測（BULLISH + NEUTRAL + BEARISH）における Hit 率 | 主要指標テキストの括弧内 |
| 方向精度 | BULLISH + BEARISH のみにおける Hit 率（NEUTRAL を除外） | 主要指標テキスト本体 + 日次推移グラフ + 銘柄別（将来拡張） |
| 判定済み件数 | 採点が完了している予測の件数 | 主要指標テキストの括弧内 + 日次推移グラフ右軸 |
| シグナル別ヒット率 | BULLISH / NEUTRAL / BEARISH ごとの Hit 率と件数 | シグナル別棒グラフ + 数値テーブル |

NEUTRAL 比率はシグナル別の `count` から導出可能なため独立指標としては出さない。AI 失敗件数は精度の話と性質が違うため UI に出さない（将来運用監視で対応）。

## 5. 受け入れ条件

- [ ] 新規生成された予測が、翌営業日終値確定後 1 時間以内に自動採点される
- [ ] 生のリターン値が DB に保存され、後で閾値変更による再集計が可能
- [ ] AI 解析失敗ケース（`AiAnalysisError` あり）が採点対象から除外される（UI 表示はしない）
- [ ] ダッシュボードで以下が確認できる：
    - [ ] 期間切替（7/30/90 日/全期間）。デフォルトは「直近 30 日」
    - [ ] ページ見出し直下の主要指標テキスト（期間ラベル + 方向精度 + 採点件数 + 総合精度。判定済み 0 件のときは空状態文言）
    - [ ] 日次方向精度推移グラフ（折れ線 + 件数棒 + 数値テーブル）
    - [ ] シグナル別ヒット率（棒グラフ + 数値テーブル）
- [ ] 採点バッチ Lambda が EventBridge cron で 1 時間毎に起動する
- [ ] 既存認証フローでダッシュボード閲覧が `stocks:read` 権限ガード経由で制御される
- [ ] ユニットテストカバレッジ 80% 以上を維持
- [ ] dev 環境で動作確認完了

## 6. スコープ外

- 既に蓄積済みの `AiAnalysisResult` への遡及採点（別 Issue）
- ボラティリティ調整による動的閾値（後の Issue）
- 保有期間バリエーション（intraday / 数日先）の追加
- Phase 2 以降の AI 改善機構（教訓蒸留 / RAG / ファインチューニング）

## 7. 補足

### Phase 2 以降での再利用前提

- 生リターン値（`ActualReturn`）の保存：閾値変更による再集計を可能にする
- 永続保持（TTL なし）：古いデータも学習素材として残す
- 採点結果は既存 `DailySummaryEntity` に統合保存するため、銘柄・取引所・日付・パターン合致数・AI 解析結果（`AiAnalysisResult.investmentJudgment.signal`）などのコンテキスト情報も同じレコードから 1 read で取得できる（独立エンティティとの join 不要）
