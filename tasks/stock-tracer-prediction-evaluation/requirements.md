## 1. ビジネス要件

### 1.1 背景・目的

Stock Tracer は OpenAI による日次 AI 解析で銘柄ごとに投資判断シグナル（`BULLISH / NEUTRAL / BEARISH`）を生成しているが、生成された予測が実際に当たったか外れたかを自動的に追跡する仕組みが存在しない。

このため、

- 現在の AI 精度を定量的に把握できない
- AI モデルやプロンプトを改善しても効果測定ができない
- 蓄積される予測データが学習素材として活用されていない

本タスクでは予測 → 翌日実績の自動採点 + 可視化基盤を構築し、後続フェーズでの AI 精度改善（教訓蒸留 / Few-shot RAG / ファインチューニング）に必要なデータと改善ループを成立させる。

### 1.2 対象ユーザー

- **プライマリーユーザー**: Stock Tracer の運用者（精度を確認し、AI 改善方針を意思決定）
- **セカンダリーユーザー**: 一般利用者（自分が見ている銘柄の予測精度を参考情報として確認）

### 1.3 ビジネスゴール

- AI 予測精度の定量的可視化を実現する
- AI 改善施策の効果測定ができる状態にする
- 後続フェーズ（教訓蒸留・RAG・ファインチューニング）の入力となる学習データを蓄積する

## 2. 機能要件

### 2.1 ユースケース

#### UC-001: 自動採点

- **概要**: 既存 AI 解析バッチが生成した予測を、翌営業日終値が確定した時点で自動的に採点する
- **アクター**: システム（採点バッチ Lambda）
- **前提条件**:
    - 対象予測の `AiAnalysisResult` が存在し、`AiAnalysisError` を持たない
    - 対象予測の取引所の翌営業日が引け済み
- **正常フロー**:
    1. EventBridge cron（1 時間毎）が採点バッチ Lambda をトリガー
    2. バッチが「未採点 & 翌営業日引け済」の予測を抽出
    3. 各予測について、当日終値と翌営業日終値を TradingView API から取得
    4. リターン値と Hit/Miss を判定し、`PredictionOutcomeEntity` として保存
- **代替フロー**: 翌営業日が祝日・週末の場合、その次の営業日の終値で判定する
- **例外フロー**:
    - TradingView API 呼び出しが失敗した場合：当該予測は未採点のまま次回再試行
    - 対象予測が `AiAnalysisError` を持つ場合：採点対象から除外（採点フラグを「除外」として記録）

#### UC-002: 精度ダッシュボード閲覧

- **概要**: 利用者がダッシュボードで AI 予測の精度を期間・切り口別に確認する
- **アクター**: 認証済みユーザー
- **前提条件**: 既存認証フローでログイン済み
- **正常フロー**:
    1. ユーザーがダッシュボード画面を開く
    2. デフォルトで「直近 30 日」の集計を表示
    3. 期間（7/30/90/全期間）を切り替えると再集計される
    4. シグナル別・銘柄別・取引所別の切り口で精度を確認できる
- **代替フロー**: 採点済み予測がない場合は「データなし」を表示
- **例外フロー**: API エラー時は再読み込みボタンを表示

## 3. 非機能要件

### 3.1 パフォーマンス要件

| 項目 | 要件 |
|------|------|
| 採点バッチ実行時間 | 全銘柄の採点処理が 5 分以内に完了する |
| 採点遅延 | 翌営業日引け後、最大 1 時間以内に採点が反映される（cron 起動間隔依存） |
| ダッシュボード API 応答時間 | 1 秒以内（直近 90 日分集計時） |
| ダッシュボード初回表示 | 3 秒以内 |

### 3.2 セキュリティ要件

- ダッシュボード閲覧は既存認証フローに準拠（未認証アクセス不可）
- 採点 API も既存 web の認証を共通に使用
- DynamoDB の暗号化は既存テーブル設定を踏襲

### 3.3 可用性要件

| 項目 | 要件 |
|------|------|
| 採点バッチ失敗時 | 1 回の失敗で再試行が次回 cron で自動的に行われる（冪等性） |
| データ整合性 | 同一予測に対して採点が重複しないこと |

### 3.4 テスト要件

- ユニットテストカバレッジ：80% 以上（`coverageThreshold` で自動失敗する設定を維持）
- 純粋関数（判定ロジック）は副作用モックなしで網羅テスト
- TradingView API 呼び出しはモックで分離テスト
- E2E：ダッシュボード基本表示の Fast CI 用シナリオを追加（`chromium-mobile`）

## 4. 採点ルール仕様

### 4.1 判定基準（Phase 1）

| 予測シグナル | 成功条件 |
|-------------|---------|
| BULLISH | 翌営業日終値リターン ≥ +0.5% |
| BEARISH | 翌営業日終値リターン ≤ -0.5% |
| NEUTRAL | 翌営業日終値リターンが -0.5% より大きく、+0.5% より小さい |

- リターン基準点：当日終値（close-to-close）
- 保有期間：1 営業日
- リターン計算式：`(翌営業日終値 - 当日終値) / 当日終値 × 100`（%）
- 境界値（厳密に +0.5% / -0.5% / -0.5% / +0.5%）の扱いは `design.md` で定義

### 4.2 採点対象から除外するケース

- 既存 `DailySummaryEntity.AiAnalysisError` が存在する予測
- 翌営業日終値が取得できない予測（上場廃止・データ欠損など。`design.md` でリトライ戦略を定義）

### 4.3 集計指標

| 指標名 | 定義 |
|--------|------|
| 総合精度 | 全予測（BULLISH + NEUTRAL + BEARISH）における Hit 率 |
| 方向精度 | BULLISH + BEARISH のみにおける Hit 率（NEUTRAL を除外） |
| NEUTRAL 比率 | 全予測に対する NEUTRAL の割合 |
| 判定済み件数 | 採点が完了している予測の件数 |
| AI 失敗件数 | `AiAnalysisError` が存在し採点対象から除外された件数 |

## 5. 受け入れ条件

- [ ] 新規生成された予測が、翌営業日終値確定後 1 時間以内に自動採点される
- [ ] 生のリターン値が DB に保存され、後で閾値変更による再集計が可能
- [ ] AI 解析失敗ケースが採点対象から除外され、件数のみ別途表示される
- [ ] ダッシュボードで以下が確認できる：
    - [ ] 期間切替（7/30/90 日/全期間）
    - [ ] KPI カード 4 種（総合精度・方向精度・NEUTRAL 比率・判定済み件数）
    - [ ] 日次方向精度推移グラフ
    - [ ] シグナル別ヒット率
    - [ ] 銘柄別ヒット率テーブル（ソート・最低件数フィルタ可）
    - [ ] 取引所別ヒット率
    - [ ] AI 失敗件数表示
- [ ] 採点バッチ Lambda が EventBridge cron で 1 時間毎に起動する
- [ ] 既存認証フローでダッシュボード閲覧が制御される
- [ ] ユニットテストカバレッジ 80% 以上を維持
- [ ] dev 環境で動作確認完了

## 6. スコープ外

- 既に蓄積済みの `AiAnalysisResult` への遡及採点（別 Issue）
- ボラティリティ調整による動的閾値（後の Issue）
- 保有期間バリエーション（intraday / 数日先）の追加
- Phase 2 以降の AI 改善機構（教訓蒸留 / RAG / ファインチューニング）

## 7. 補足

### Phase 2 以降での再利用前提

- 生リターン値の保存：閾値変更による再集計を可能にする
- 永続保持（TTL なし）：古いデータも学習素材として残す
- 銘柄・取引所・日付・パターン合致数などのコンテキスト情報は既存 `DailySummaryEntity` を参照することで取得可能（重複保存しない）
