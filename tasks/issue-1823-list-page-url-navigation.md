# リストのページ・URL 遷移の見直し

## 概要

個人リスト・共有リストのページ構成および URL 遷移が複雑になりすぎており、
期待した動作になっていない。
要件を再整理し、必要なページと URL 設計を明確化する。

## 関連情報

- Issue: #1823
- タスクタイプ: サービスタスク（share-together/web）
- マイルストーン: v6.0.0

---

## 現状の問題点

### ページ構成（現状）

| URL | ページ | 備考 |
|-----|--------|------|
| `/` | ホーム（デフォルト個人リスト表示） | ユーザー登録 API も自動実行 |
| `/lists` | リスト（サーバーサイドでデフォルトリストへリダイレクト） | リダイレクト専用ページ |
| `/lists/[listId]` | 個人リスト詳細 | `ListWorkspace` で個人・共有切り替え可能 |
| `/groups` | グループ一覧 | — |
| `/groups/[groupId]` | グループ詳細（メンバー管理、リスト一覧） | — |
| `/groups/[groupId]/lists/[listId]` | 共有リスト詳細 | 独自サイドバー、スコープ切り替えなし |
| `/invitations` | 招待一覧 | — |

### 特定された問題

1. **`/` と `/lists` の役割が重複**
    - ホームページ（`/`）がデフォルト個人リストを表示するが、`/lists` も同じリストへリダイレクトする
    - ナビゲーション「ホーム」と「リスト」が実質的に同じ宛先を指している

2. **URL と状態の非同期**
    - `/lists/[listId]` の `ListWorkspace` で共有リストを選択すると `/groups/[groupId]/lists/[listId]` に遷移する
    - 逆方向（共有リストから個人リストへ）のナビゲーションが一貫していない
    - URL を見てもどのグループの共有リストかが分かる一方、個人リストへの経路が不明確

3. **`/groups/[groupId]/lists/[listId]` の孤立**
    - このページには独立したサイドバーがあるが、`ListWorkspace` のスコープ切り替えがない
    - グループ詳細（`/groups/[groupId]`）と共有リスト詳細が別ページに分離しているが連携が曖昧

4. **モックデータの残存**
    - `ListWorkspace` 内で `MOCK_PERSONAL_LISTS` が一部のパスで使用されており、未完成の状態

5. **`/lists` リダイレクトのコスト**
    - サーバーサイドで `/api/lists` を呼び出してからリダイレクトしており、不要なラウンドトリップが発生
    - デフォルトリストが取得できない場合は 404 を返しユーザー体験が悪い

## 要件

### 機能要件

- **FR1**: 個人リストの一覧と詳細（ToDo）を閲覧・操作できる
- **FR2**: 共有リスト（グループ単位）の一覧と詳細（ToDo）を閲覧・操作できる
- **FR3**: 個人リストと共有リストを切り替えて比較・操作できる
- **FR4**: ナビゲーションメニューからそれぞれの機能へ 1 ステップで到達できる
- **FR5**: URL がリソースの状態（どのリスト・グループ）を一意に表す
- **FR6**: ブラウザの「戻る」ボタンで自然に前の状態に戻れる
- **FR7**: URL を直接入力・共有しても正しいページが表示される

### 非機能要件

- **NFR1**: ユーザー登録の自動実行は初回ログイン時に適切なタイミングで行う（ページ表示に影響しない）
- **NFR2**: サーバーサイドでのリダイレクトは最小限にする
- **NFR3**: モックデータを本番コードから除去する

## 提案する URL 設計

### 基本方針

- **個人リスト**: `/lists` を個人リストのルートとし、リスト選択は URL に反映する
- **グループ・共有リスト**: `/groups` をグループ管理のルートとし、共有リストは `/groups/[groupId]/lists/[listId]` に統一
- **ナビゲーション**: 「リスト」「グループ」の 2 項目に整理

### 提案する URL 構成

| URL | ページ | 説明 |
|-----|--------|------|
| `/` | ホーム | ログイン後ランディング。`/lists` へリダイレクトまたはサービス紹介 |
| `/lists` | 個人リスト一覧 | デフォルトリストを自動選択して表示（SSR 不要、クライアントサイドで解決） |
| `/lists/[listId]` | 個人リスト詳細 | 指定リストの ToDo 一覧（サイドバーで他の個人リストへ切り替え） |
| `/groups` | グループ一覧 | 参加グループの一覧・作成・招待 |
| `/groups/[groupId]` | グループ詳細 | メンバー管理、グループ設定（共有リストへの入口も含む） |
| `/groups/[groupId]/lists` | 共有リスト一覧 | グループ内の共有リスト一覧（最初のリストを自動選択） |
| `/groups/[groupId]/lists/[listId]` | 共有リスト詳細 | 指定共有リストの ToDo 一覧（サイドバーで同グループの他リストへ切り替え） |

### ナビゲーション（提案）

```
[リスト] → /lists
[グループ] → /groups
```

「ホーム」メニューは削除し、アプリ名（"Share Together"）クリックで `/` または `/lists` へ遷移する。

## 実装のヒント

### `/` ホームページ

- ユーザー登録の自動実行を `/` から分離し、レイアウト（`layout.tsx`）または専用の初期化フックで処理する
- `/` は `/lists` へのシンプルなリダイレクト、またはサービス紹介ページにする

### `/lists` 個人リスト一覧

- クライアントサイドでデフォルトリストを取得し、URL を `replace` で `/lists/[listId]` に変更する
    - `router.replace()` を使うことでブラウザ履歴を汚染しない
    - サーバーサイドリダイレクト（現状）から変更することでラウンドトリップを削減

### `/lists/[listId]` 個人リスト詳細

- `ListWorkspace` からスコープ切り替え（個人/共有）を削除し、個人リスト専用にシンプル化
- サイドバーでの個人リスト切り替えは `/lists/[newListId]` への `Link` ナビゲーションに統一
- モックデータ（`MOCK_PERSONAL_LISTS`）を削除し、常に API から取得する

### `/groups/[groupId]/lists/[listId]` 共有リスト詳細

- サイドバーでの共有リスト切り替えは `/groups/[groupId]/lists/[newListId]` への `Link` ナビゲーションに統一
- グループ詳細（`/groups/[groupId]`）へ戻るボタンまたはパンくずリストを追加

### `ListWorkspace` コンポーネント

- スコープ切り替え（個人/共有）の役割を URL ベースのナビゲーションに移行後、削除またはリファクタリングを検討
- 共有リストの状態管理（`selectedGroupId`, `sharedListsByGroup` 等）は各ページコンポーネントに移譲する
