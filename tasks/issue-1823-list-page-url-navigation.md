# リストのページ・URL 遷移の見直し

## 概要

個人リスト・共有リストのページ構成および URL 遷移が複雑になりすぎており、
期待した動作になっていない。
要件を再整理し、必要なページと URL 設計を明確化する。

## 関連情報

- Issue: #1823
- タスクタイプ: サービスタスク（share-together/web）
- マイルストーン: v6.0.0

---

## 現状の問題点

### ページ構成（現状）

| URL | ページ | 備考 |
|-----|--------|------|
| `/` | ホーム（デフォルト個人リスト表示） | ユーザー登録 API も自動実行 |
| `/lists` | リスト（サーバーサイドでデフォルトリストへリダイレクト） | リダイレクト専用ページ |
| `/lists/[listId]` | 個人リスト詳細 | `ListWorkspace` で個人・共有切り替え可能 |
| `/groups` | グループ一覧 | — |
| `/groups/[groupId]` | グループ詳細（メンバー管理、リスト一覧） | — |
| `/groups/[groupId]/lists/[listId]` | 共有リスト詳細 | 独自サイドバー、スコープ切り替えなし |
| `/invitations` | 招待一覧 | — |

### 特定された問題

1. **`/` と `/lists` の役割が重複**
    - ホームページ（`/`）がデフォルト個人リストを表示するが、`/lists` も同じリストへリダイレクトする
    - ナビゲーション「ホーム」と「リスト」が実質的に同じ宛先を指している

2. **URL と状態の非同期**
    - `/lists/[listId]` の `ListWorkspace` で共有リストを選択すると `/groups/[groupId]/lists/[listId]` に遷移する
    - 逆方向（共有リストから個人リストへ）のナビゲーションが一貫していない
    - URL を見てもどのグループの共有リストかが分かる一方、個人リストへの経路が不明確

3. **`/groups/[groupId]/lists/[listId]` の孤立**
    - このページには独立したサイドバーがあるが、`ListWorkspace` のスコープ切り替えがない
    - グループ詳細（`/groups/[groupId]`）と共有リスト詳細が別ページに分離しているが連携が曖昧

4. **モックデータの残存**
    - `ListWorkspace` 内で `MOCK_PERSONAL_LISTS` が一部のパスで使用されており、未完成の状態

5. **`/lists` リダイレクトのコスト**
    - サーバーサイドで `/api/lists` を呼び出してからリダイレクトしており、不要なラウンドトリップが発生
    - デフォルトリストが取得できない場合は 404 を返しユーザー体験が悪い

## 要件

### 機能要件

- **FR1**: 個人リストの一覧と詳細（ToDo）を閲覧・操作できる
- **FR2**: 共有リスト（グループ単位）の一覧と詳細（ToDo）を閲覧・操作できる
- **FR3**: 個人リストと共有リストを `/lists` ページ内で GUI 切り替えにより操作できる
- **FR4**: ナビゲーションメニューからそれぞれの機能へ 1 ステップで到達できる
- **FR5**: 初回表示時にデフォルト個人リストが自動選択される

### 非機能要件

- **NFR1**: ユーザー登録の自動実行は初回ログイン時に適切なタイミングで行う（ページ表示に影響しない）
- **NFR2**: サーバーサイドでのリダイレクトは最小限にする
- **NFR3**: モックデータを本番コードから除去する
- **NFR4**: 各コンポーネントの責務を明確に分離し、肥大化を防ぐ

## 採用する URL 設計

### 基本方針

- **`/lists` に全リスト操作を集約**: 個人リストも共有リストも `/lists` 単一ページで管理し、リスト・グループの切り替えは GUI（ドロップダウン、サイドバー）で行う
    - リスト選択は URL に反映せず、ページ内の状態管理で制御する
    - コンポーネントは各要素の責務ごとに分割する（肥大化への対処）
- **`/groups` はグループ管理専用**: メンバー管理・招待など、グループの設定系操作に限定する
- **ナビゲーション**: 「リスト」「グループ」の 2 項目に整理

### URL 構成

| URL | ページ | 説明 |
|-----|--------|------|
| `/` | サービス紹介 | サービスの概要・機能紹介（ランディングページ） |
| `/lists` | リスト | 個人・共有リストの一覧と ToDo 操作（GUI で切り替え） |
| `/groups` | グループ | 参加グループの選択・メンバー管理・グループ設定（GUI で切り替え） |

※ `/lists/[listId]`、`/groups/[groupId]/lists/[listId]`、`/groups/[groupId]` などの動的ルートは廃止する

### ナビゲーション（確定）

```
[リスト] → /lists
[グループ] → /groups
```

「ホーム」メニューは削除し、アプリ名（"Share Together"）クリックで `/` へ遷移する。

## 実装のヒント

### `/` サービス紹介ページ

- サービスの概要・機能紹介を表示するランディングページにする
- ユーザー登録の自動実行を `/` から分離し、レイアウト（`layout.tsx`）または専用の初期化フックで処理する
- ログイン後も `/` はサービス紹介ページのまま（`/lists` へのリダイレクトは行わない）

### `/lists` リストページ（単一ページで全操作）

- ページ内の状態で個人リスト・共有リストの選択を管理する（URL 変更なし）
- **レイアウト方針（モバイルファースト）**: サイドバーによる横並びレイアウトは採用しない。縦積みの UI で完結させる
  1. **ラジオボタン**で「個人 / 共有」を選択
  2. （共有選択時）グループを選択するセレクトボックスを表示
  3. **セレクトボックス**で ToDo リストを選択
  4. 選択リストの ToDo 一覧をその下に表示
- API から全データを取得し、選択状態に応じて ToDo コンポーネントをレンダリングする
- 初回表示時はデフォルト個人リストを選択状態にする
- モックデータ（`MOCK_PERSONAL_LISTS`）を完全に除去し、常に API から取得する

#### コンポーネント分割指針

`/lists` ページを単一コンポーネントに実装せず、責務ごとに分割する：

- **スコープ選択コンポーネント**: 個人/共有のラジオボタン UI
- **グループ選択コンポーネント**: 共有スコープ時のグループセレクトボックス
- **リスト選択コンポーネント**: ToDo リストのセレクトボックスと CRUD 操作
- **ToDo コンテンツコンポーネント**: 選択リストの ToDo 一覧・操作

### `/groups` グループページ（単一ページで全操作）

- ページ内の状態で選択グループを管理する（URL 変更なし）
- **グループ選択**: サイドバーのグループ項目クリックで選択状態を更新
- 選択したグループのメンバー一覧・招待・設定をメインエリアに表示する
- グループ作成・参加（招待コード入力）も同一ページ内で完結させる
- 共有リストの操作は `/lists` ページに集約するため、このページには含めない
