# リブトーク 知識・記憶の Topic 中心モデル 再設計 要件定義書

---

## 1. ビジネス要件

### 1.1 背景・目的

リブトークの差別化価値は「キャラが同じ時間軸で生活している」体験にあり、その中核が **会話の記憶** と **能動的な勉強（Web リサーチ）** である。しかし現行の知識・記憶は以下の構造的問題を抱え、売りが不安定要素になっている。

- **線形増加・非集約**: 勉強成果（KNOWLEDGE）が生のまま永続追記され、同一話題が繰り返し再調査・重複保存される（dev 実測で「ラーメン最新動向 ×17」等）。
- **非リンク・非抽象化**: 同じ話題について「ユーザーが話したこと（記憶）」と「キャラが調べたこと（知識）」が別サイロに分断され、繋がらない・抽象化されない。
- **陳腐化の粒度を扱えない**: 揮発性（在庫・価格・順位）と安定情報（作品の事実等）を区別せず、一律間隔で再調査するため陳腐な再要約を量産する。
- **忘却・想起の複雑さ**: 会話記憶は Tier A–D＋信頼度＋cooldown＋関連度フィルタで運用しており、目的（誤情報の忘却／些細情報の連呼防止）に対して機構が複雑。

本再設計は、知識・記憶の単位を **Topic（話題）** に置き換え、上記を「集約・リンク・鮮度・関連度想起・決定的忘却」で解く。

### 1.2 対象ユーザー

- **プライマリ**: リブトークのエンドユーザー（キャラと会話する 1 人／`livetalk:chat`）。現状 dev・prod ともに 1 ユーザー規模。
- **セカンダリ**: 運用者（挙動の可観測性・セーフティ）。

### 1.3 ビジネスゴール

- 同一話題で「話したこと」と「調べたこと」が融合した応答ができる（例:「二郎系好きだったよね、新店できたよ」）。
- 勉強成果が無限に溜まらず、話題ごとに畳まれて抽象化される。
- 揮発情報だけ賢く追随し、安定情報の無駄な再調査をやめる。
- 「あなたのために調べた」ギフト体験としてのノート。
- 誤って覚えた情報を確実に忘れさせられる（LLM の抑制に依存しない）。

## 2. 機能要件

### 2.1 ユースケース

#### UC-001: 会話・勉強の集約（consolidation）

- **概要**: 未集約の元データ（会話ログ・Web 取得生）を話題ごとに畳み、Topic の知識を更新する。
- **アクター**: 集約バッチ（毎時）。
- **前提条件**: 集約カーソル以降に未集約の元データが存在する。
- **正常フロー**:
    1. カーソル以降の元データを読む。
    2. 埋め込み近傍で対象 Topic を特定（無ければ新規 Topic）。
    3. SELF/WEB の fact を merge・重複排除・抽象化して更新し、要約・care・stability・nextReview を更新。
    4. カーソルを前進する（元データは削除せず TTL に委ねる）。
- **例外フロー**: ルーティング/要約失敗時は当該断片をスキップし、カーソルは失敗分の手前で止めて次回再処理する（at-least-once）。

#### UC-002: 会話時の想起（関連度 only）

- **概要**: ユーザー発話に関連する Topic のみを文脈に注入する。
- **アクター**: ユーザー、キャラ（チャット）。
- **正常フロー**:
    1. 発話＋直近文脈を埋め込む。
    2. Topic 埋め込みと類似度を計算し、閾値超えの上位 K 件を選ぶ（軽い 1 ホップ関連展開・閾値付き）。
    3. 対象 Topic の SELF/WEB を文脈に注入して応答する。
- **代替フロー**: 関連 Topic が無ければ Topic 知識なしで応答する。

#### UC-003: 記憶の閲覧と忘却

- **概要**: ユーザーが「自分について覚えていること（SELF）」を閲覧し、1 件単位で削除できる。
- **アクター**: ユーザー。
- **正常フロー**:
    1. SELF fact 一覧を表示する。
    2. 削除操作で当該 SELF fact を物理削除する。
    3. 当該 Topic の要約を即時再生成し、削除内容が導出物にも残らないようにする。
- **例外フロー**: 会話での訂正でも SELF は更新されうる（前向き整合）。

#### UC-004: 自発リサーチと依頼リサーチ（acquire）

- **概要**: キャラが興味（care）に基づき自発的に、または会話中の依頼に応じて Web を取得する。
- **アクター**: 取得バッチ（毎時）、ユーザー（依頼）。
- **正常フロー**:
    1. 対象を選ぶ（care 上位の Topic／依頼された話題／鮮度切れの揮発 fact）。
    2. Web を取得し、生結果を元データとして書く。
    3. 集約（UC-001）が Topic に畳む。
- **例外フロー**: 取得失敗はスキップし次回に委ねる。

#### UC-005: 鮮度追随（staleness refresh）

- **概要**: 揮発性の高い WEB fact のみを期限到来時に再取得する。
- **正常フロー**:
    1. 揮発 fact の再確認期限を掃引する。
    2. 期限到来分を acquire 対象にする。
    3. 再取得後、内容に変化があった場合のみ更新する。
- **例外フロー**: 安定 fact は掃引対象外（再確認しない）。

#### UC-006: ギフトとしてのノート

- **概要**: Topic から「なぜ調べたか（SELF フック）＋調べた内容（WEB）」を合成してノート（共有ログ）を作る。
- **正常フロー**:
    1. care・新規性・実在の SELF フックを条件にノート化対象を選ぶ。
    2. SELF を根拠に WEB を贈る文面を合成し、共有ログとして記録する。
    3. チャットで直近の共有に対する感想に反応できる。
- **例外フロー**: 実在フックが弱い場合は自発トーンで控えめにする（捏造しない）。

#### UC-007: 能動通知の優先度

- **概要**: 重要な発見を能動通知する優先度を care で判定する。
- **正常フロー**: care の高い Topic の新規性ある発見を、通知の閾値判定に用いる。

### 2.2 機能一覧

| 機能ID | 機能名 | 説明 | 優先度 |
| ------ | ------ | ---- | ------ |
| FR-1 | Topic 集約 | 未集約元データを Topic 単位に畳み、SELF/WEB fact・要約・care・鮮度を更新 | 高 |
| FR-2 | 関連度 only 想起 | 発話に関連する Topic のみを閾値付き top-K で注入 | 高 |
| FR-3 | 決定的忘却 | SELF fact の物理削除＋当該 Topic 要約の即時再生成 | 高 |
| FR-4 | acquire（取得） | care 自発＋依頼＋鮮度切れを Web 取得し元データ化 | 高 |
| FR-5 | 鮮度追随 | 揮発 fact のみ期限掃引・変化時のみ更新 | 高 |
| FR-6 | ギフトノート | SELF フック＋WEB 合成の共有ログ生成・感想連携 | 中 |
| FR-7 | care 優先度 | 自発リサーチ選定・能動通知閾値を care（回数ベースの興味強度・情動重み付けなし）で駆動 | 中 |
| FR-8 | 旧資産撤去 | Tier / MemorySummary / InterestCategory / 旧 Knowledge・Note / StudyTopic の廃止 | 中 |

### 2.3 想定画面の概要

- 記憶（私について覚えていること）: SELF fact 一覧・削除。
- ノート（調べてくれたこと）: 共有ログ一覧・詳細。

（詳細は external-design.md）

## 3. 非機能要件

### 3.1 パフォーマンス要件

| 項目 | 要件 |
| ---- | ---- |
| 応答時間 | 会話ホットパスで集約 LLM を呼ばない（想起は埋め込み類似度と注入のみ）。想起の追加レイテンシは最小限。 |
| スループット | 集約・取得は毎時バッチで非同期。会話をブロックしない。 |
| データ量 | 1 Topic あたりの item は集約で少数に保つ（無限増加させない）。1 アイテム 400KB 制約を超えない（per-fact 分割で回避）。 |

### 3.2 セキュリティ要件

- 認可は既存 RBAC（`livetalk:chat` / `livetalk:admin`）を踏襲する。
- PII（会話本文・個人事実）は既存方針どおり計測ログに含めない。
- 退会・データ削除（ADR-2.21）と整合させる（`PK=USER#` 配下のハード削除で本再設計の item も消える）。

### 3.3 可用性要件

| 項目 | 要件 |
| ---- | ---- |
| バッチ | 失敗はスキップ／DLQ で継続（fail-warn）。カーソルは at-least-once を守る。 |

### 3.4 保守性・拡張性要件

- DynamoDB Single Table 設計を idiomatic に踏襲（アクセスパターン起点・sparse GSI）。
- グラフ DB・ベクトル DB を導入しない（関連は tags＋埋め込み近傍で AI に委ねる）。
- UI（web）とロジック（core）を分離、libs のパスエイリアス禁止等の既存規約を遵守。

### 3.5 その他の非機能要件

- 律速リスク（Topic ルーティング／マージ品質）に対し、生断片の provenance 保持で可逆化・version 条件付き書込・人手削除を最優先とする。

## 4. ドメインオブジェクト

| エンティティ | 説明 |
| ----------- | ---- |
| Topic | 話題。知識・記憶の集約単位（複数 item の束） |
| SELF fact | 会話由来のユーザー事実（削除単位・ユーザー所有） |
| WEB fact | 勉強由来の知識（鮮度単位・出典付き） |
| Note（共有ログ） | 「調べたよ」の出来事。Topic を参照するギフト記録 |
| 元データ | 未集約の会話ログ（既存 Message）・Web 取得生 |
| 集約カーソル | どこまで集約したかを指すウォーターマーク |
| care | ユーザーがその話題をどれだけ気にしているかの優先度 |
| volatility | WEB fact の揮発性（安定／揮発）。鮮度追随の駆動 |

## 5. スコープ外

- ❌ 会話応答の構造化（JSON 全振り／ストリーム廃止）— 本再設計の前提ではなく別スコープ（旧 #3355）。
- ❌ Memory Tier A–D の互換維持 — 本再設計で廃止する。
- ❌ マルチユーザー規模の最適化（現状 1 ユーザー、病的設計のみ回避）。
- ❌ グラフ DB／ベクトル DB の導入。
- ❌ 旧データの移行（破壊的変更・クリーンスタートを許容）。
- ❌ 親密度（Affection）の体験活用（別ロードマップ R1）。
- ❌ 話題に無関係でも覚えておくべき事項（予定・約束・記念日・情動的な伏線）の能動想起。将来、通知・あいさつ層で care と Topic を用いて実現する余地は残すが、本再設計では扱わない。

## 6. 用語集

| 用語 | 定義 |
| ---- | ---- |
| Topic | 話題を単位とする知識・記憶の集約体。SK プレフィックスで束ねた item 群 |
| SELF / WEB facet | Topic 配下の会話由来 / 勉強由来の事実。前者は削除単位、後者は鮮度単位 |
| episodic / semantic | 生の出来事 / 畳んだ理解。境界は「集約済みか」で決まる |
| consolidation（集約） | 未集約元データを Topic に畳む毎時バッチ処理 |
| acquire（取得） | Web を取ってくるだけの毎時バッチ処理（畳みは consolidation） |
| care | 話題への注目度。自発リサーチ・通知の優先度信号（旧 InterestCategory の Weight を継承） |
| 想起 | 会話時に関連 Topic を文脈へ注入すること（関連度 only） |
