## 背景・目的

大容量動画ファイルのアップロードとバッチ分析処理に時間がかかる問題を解消する。
現状は以下の処理が直列で動いており、ボトルネックになっている。

- クライアント側のマルチパートアップロード PUT（`for...of` による逐次送信）
- Whisper API を使った文字起こしのチャンク処理（`for` ループ）
- OpenAI API を使った感情スコアリングのチャンク処理（`for` ループ）

また、マルチパートアップロードの閾値が 5GB と高すぎるため実運用で恩恵を受けにくく、
失敗時に `AbortMultipartUpload` が呼ばれず S3 に不完全パーツが残るリスクもある。

## 機能要件

1. マルチパートアップロードの閾値を 5GB → **100MB** に引き下げる
2. マルチパートのチャンクサイズを 500MB → **50MB** に変更する
3. クライアント側のチャンク PUT を並列化する（`Promise.all`）
4. アップロードエラー時（PUT 失敗・ETag 取得失敗）に `AbortMultipartUpload` を呼び出す
5. Whisper 文字起こしのチャンク処理を並列化する（`Promise.all`）
6. 感情スコアリングのチャンク処理を並列化する（同時実行数を定数で制御）

## 非機能要件

- 感情スコアリングの同時実行数は定数 `EMOTION_SCORING_CONCURRENCY`（初期値 3）で管理し、後から調整できること
- アップロードの中断・失敗時に S3 の不完全パーツが残らないこと
- 並列化後も既存の動作（ETag 収集・セグメントの時刻オフセット補正・スコア集約）が正しく動作すること

## 受け入れ条件

- [ ] 100MB 以上のファイルでマルチパートアップロードが動作する
- [ ] アップロード失敗時に `AbortMultipartUpload` が呼ばれ、S3 に不完全パーツが残らない
- [ ] Whisper チャンク処理が並列で動作する（複数チャンク時）
- [ ] 感情スコアリングチャンク処理が並列で動作し、同時実行数が `EMOTION_SCORING_CONCURRENCY` で制限される
- [ ] 既存のテストがすべてパスする
- [ ] Lint・型チェックがすべて通過する

## Phase 5: 解析進捗可視化 & モーション+音量+文字起こし並列化

### 背景・目的

`analyzing` ステージ中にモーション分析・音量分析・文字起こしの3種類が直列で実行されており、
UI には「解析中」のテキストしか表示されない。
処理の並列化によってスループットを改善し、各処理の状況を個別に可視化することで
ユーザーが現在地を把握できるようにする。

### 機能要件

7. モーション分析・音量分析・文字起こしを同時に並列実行する
8. 解析ステージ中にモーション分析・音量分析・文字起こし・感情分析それぞれの進捗を
   個別のサブ項目として UI に表示する
   - モーション分析・音量分析: 「実行中」または「完了」の2状態を表示する
   - 文字起こし: チャンク数が 2 以上の場合は「X/Y」形式で進捗を表示する（1の場合は状態のみ）
   - 感情分析: チャンク数が 2 以上の場合は「X/Y」形式で進捗を表示する（1の場合は状態のみ）
9. OpenAI API キーが未設定の場合は文字起こし・感情分析のサブ項目を表示しない

### 非機能要件

- UI のポーリング間隔は既存の 5 秒を維持する
- 感情分析は 3 つの並列処理（モーション・音量・文字起こし）がすべて完了してから開始する
- 進捗情報は DynamoDB の job アイテムに保存し、既存の `/api/jobs/[jobId]` ポーリングで取得する

### 受け入れ条件

- [ ] モーション・音量分析と文字起こしが並列で実行される
- [ ] UI の解析ステップにモーション・音量・文字起こしのサブ項目が表示される
- [ ] モーション・音量は「実行中」→「完了」と切り替わる
- [ ] 文字起こしチャンクが 2 以上の場合「X/Y」の進捗が更新される
- [ ] 感情分析チャンクが 2 以上の場合「X/Y」の進捗が更新される
- [ ] OpenAI API キー未設定時は文字起こし・感情分析のサブ項目が表示されない
- [ ] 既存のテストがすべてパスする
- [ ] Lint・型チェックがすべて通過する
