# 通知アーキテクチャのマルチキャラ対応 — requirements

関連 Issue: #3491（問題洗い出し） / 親 #3470（早瀬アゲハ追加） / Phase 5d #3346（通知基盤）

## WHY

早瀬アゲハ追加（#3470）でマルチキャラ運用になったが、通知まわりが旧来の
「ユーザー単位・ひより前提」設計のまま残り破綻している。dev で以下を踏んだ:

- ひより由来の通知が来た状態で、カレントがアゲハのアプリを起動 → アゲハが第一声を喋った（通知内容はひよりのもの）。
- ひよりに切り替えても、その通知内容は即時消化で既に消えている。

## 解決する問題（#3491）

- **G1**: notify バッチが `characterId = DEFAULT_CHARACTER_ID`（hiyori）固定。アゲハ等は能動通知を生成できない。
- **G2**: `NotificationEvent` に characterId が無く、「どのキャラからの通知か」を保持できない。
- **D1**: push payload に characterId が無く、クリック起動で通知元キャラへ寄せられない（`/` 固定）。
- **C1**: キャラ取り違え。通知元と無関係にカレントキャラが第一声を喋る。
- **C2**: 知識のクロス汚染。通知の knowledgeId（通知元キャラの知識）がカレントキャラの会話 context に混入する。
- **C3**: 即時消化で消える。別キャラに切り替えると通知内容が失われ、複数キャラから溜める概念も無い。

## 決定事項（本セッションの上流ゲートで合意）

- **深さ (2)**: characterId を生成〜配信〜消化まで一貫して通す。G1/G2/D1/C1/C2/C3 を解決する。
- **体験モデル**: 起動経路で出し分け。
  - push クリック起動 = 通知元キャラへ自動切替して第一声。
  - 自前起動 = 「○○から連絡が来てるよ」と提示し、ユーザーが選んだら切替＆第一声。
  - consume は **characterId 単位**。別キャラへ切替えても他キャラの未消化は残す（C3 解消）。
- **生成の調停**: 2 段ゲート。発火判定はキャラ単位の履歴で回し、最後にユーザー総量 cap で normal は 1 体を選抜。critical はキャラ独立 cap。
- **配信**: push サブスクは端末単位のまま。payload に characterId を載せ、sw click → `/?character=<id>`。
- **クロス汚染防止**: knowledgeId は characterId 一致時のみ chat 文脈へ渡す。
- **進め方**: integration/3491-notification-multichar を切り、Phase A→B→C を同一セッション連続。各 Phase 小さい Draft PR。

## 非ゴール（深さ (3)・別フェーズへ切り出し）

- 複数キャラ通知の一覧 UI・バッジ・溜め込み演出の作り込み。
- consume タイミングの高度化（明示操作までの保留など）。
- push サブスクのキャラ単位化（端末単位のまま据え置き）。

## 完了条件

- アゲハ（非デフォルトキャラ）が能動通知を生成・配信できる。
- 通知元キャラと無関係なキャラが第一声を喋らない（C1 解消）。
- 通知の knowledgeId が別キャラの会話に混入しない（C2 解消）。
- 複数キャラから通知が来ても、別キャラへ切替えで他キャラ分が消えない（C3 解消）。
- ユーザーあたりの通知過多が起きない（normal はユーザー総量 cap 内で 1 体選抜）。
- 既存（characterId 欠落）の NotificationEvent が壊れない（後方互換）。
- テストカバレッジ 80% 以上を維持。
