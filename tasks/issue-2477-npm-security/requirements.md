# npm セキュリティ対応 (2026-03-30 週次レポート) 要件定義書

---

## 1. ビジネス要件

### 1.1 背景・目的

2026-03-30 に実行された週次 npm 管理レポート（Issue #2477）において、深刻なセキュリティ脆弱性が検出された。
プラットフォーム全体の安全性を維持するため、早急な対応が求められる。

- **Critical: 1件**（handlebars）
- **High: 2件**（path-to-regexp、picomatch）
- **Moderate: 3件**（handlebars・picomatch の附随脆弱性）

また、Priority 3 として devDependencies の重複（aws-sdk-client-mock）およびパッケージ更新も検出されている。

### 1.2 対象ユーザー

- プラットフォーム開発者（脆弱性が開発ツールチェーン経由で悪用される可能性がある）
- プラットフォーム上で稼働するサービスの利用者（ランタイム依存の場合）

### 1.3 ビジネスゴール

- セキュリティ脆弱性を修正し、プラットフォームの安全性を確保する
- npm audit の Critical/High 件数をゼロにする
- 既存サービスの動作を壊さずにパッケージを安全に更新する

---

## 2. 機能要件

### 2.1 ユースケース

#### UC-001: セキュリティ脆弱性の修正

- **概要**: npm audit で検出された Critical/High 脆弱性をすべて解消する
- **アクター**: 開発者
- **前提条件**: `npm audit` を実行して脆弱性が確認できる状態にある
- **正常フロー**:
    1. 各脆弱パッケージの依存元（どのパッケージが間接依存しているか）を特定する
    2. `npm audit fix` または手動バージョン固定により脆弱バージョンを排除する
    3. `npm audit` を再実行し、Critical/High が 0 件になったことを確認する
    4. 既存のテスト・ビルドが全て通過することを確認する
- **代替フロー**: `npm audit fix` で解決できない場合は、依存元パッケージのバージョンアップまたは `overrides` によるバージョン強制固定を行う
- **例外フロー**: バージョン変更により破壊的変更が発生した場合は、影響箇所を修正してから再検証する

#### UC-002: 重複 devDependency の統合

- **概要**: 3 箇所以上のワークスペースで使用されている `aws-sdk-client-mock` をルートに統合する
- **アクター**: 開発者
- **前提条件**: 各ワークスペースのバージョンが統一されている
- **正常フロー**:
    1. 各ワークスペースで使用されているバージョンを確認する
    2. ルートの `package.json` に `aws-sdk-client-mock` を追加する
    3. 各ワークスペース（admin/core、niconico-mylist-assistant/core、codec-converter/batch）から削除する
    4. `npm install` を実行して動作確認する
- **例外フロー**: バージョン不一致がある場合は最新バージョンに統一する

#### UC-003: パッケージの定期更新

- **概要**: outdated として検出されたパッケージのうち、安全に更新できるものを更新する
- **アクター**: 開発者
- **前提条件**: 各パッケージの更新内容・破壊的変更の有無を把握している
- **正常フロー**:
    1. 各パッケージの changelog やリリースノートを確認する
    2. 破壊的変更がないものから更新する
    3. ビルド・テストが全て通過することを確認する

### 2.2 機能一覧

| 機能ID | 機能名 | 説明 | 優先度 |
| ------ | ------ | ---- | ------ |
| F-001  | セキュリティ脆弱性修正 | handlebars・path-to-regexp・picomatch の脆弱バージョンを安全なバージョンへ更新 | 高 |
| F-002  | devDependency 重複統合 | aws-sdk-client-mock をルートに統合し管理を簡素化 | 低 |
| F-003  | パッケージ更新 | outdated パッケージを最新化（破壊的変更なしのものから優先） | 低 |

## 3. 非機能要件

### 3.1 パフォーマンス要件

| 項目 | 要件 |
| ---- | ---- |
| ビルド時間 | 更新後も既存のビルド時間と同等を維持 |

### 3.2 セキュリティ要件

- `npm audit` の結果で Critical・High 件数がゼロになること
- パッケージ更新後も既存の認証・認可機能が正常に動作すること
- `package-lock.json` が適切に更新され、固定バージョンが脆弱範囲外であること

### 3.3 可用性要件

| 項目 | 要件 |
| ---- | ---- |
| 稼働率 | 更新作業中もサービス停止なし（ビルド・デプロイは既存フローを踏む） |
| テスト通過率 | 全ワークスペースのユニットテスト・E2E テストが 100% 通過 |

### 3.4 保守性・拡張性要件

- `package.json` の変更は最小限にし、意図した変更のみに留める
- 更新記録（どのパッケージをどのバージョンに更新したか）を PR 説明に残す

## 4. ドメインオブジェクト

| エンティティ | 説明 |
| ----------- | ---- |
| 脆弱パッケージ | npm audit で検出された脆弱性を持つ間接依存パッケージ（handlebars・path-to-regexp・picomatch）|
| 依存元パッケージ | 脆弱パッケージを間接依存として持つ直接依存パッケージ |
| package-lock.json | 解決済みバージョンを記録するロックファイル。更新後は CI でも再現できる状態を維持 |

## 5. スコープ外

- ❌ `next-auth` のベータ版 → 安定版移行（破壊的変更が大きいため別 Issue で対応）
- ❌ `typescript` v6 へのメジャーアップデート（別 Issue で対応）
- ❌ `@auth/core` の downgrade（`0.41.1` が必要とされているため保留）
- ❌ セキュリティ脆弱性以外のランタイム機能変更

## 6. 用語集

| 用語 | 定義 |
| ---- | ---- |
| npm audit | npm パッケージの既知脆弱性を検査するコマンド |
| npm audit fix | 自動で安全なバージョンにアップデートするコマンド（semver の互換範囲内） |
| overrides | package.json でネストした依存パッケージのバージョンを強制固定する機能 |
| 間接依存 | 直接インストールしているわけではなく、他のパッケージが依存しているパッケージ |
