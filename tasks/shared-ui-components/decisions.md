## D-1: 共通化の対象範囲

**決定**: アトミックなコンポーネント（Button / TextField 等）はラップ、レイアウトプリミティブ（Box / Container / Grid / Stack / Typography）はラップしない。

**根拠**:
- レイアウトプリミティブは MUI でも他ライブラリでも提供 API がほぼ同じで、ラップしても薄すぎて意味がない。
- アトミック層は意味的なバリアント（`color` / `variant` 等）を持つため、統制価値が高い。
- 「すべてラップ」は工数過多、「何もラップしない」は統制不在。中間が現実解。

---

## D-2: 強制力の担保方法

**決定**: ESLint `no-restricted-imports`（標準ルール）で MUI 直接 import を制限する。

**根拠**:
- 標準ルールのみで実現可能、追加プラグイン不要。
- `importNames` で個別コンポーネント単位の制限が可能。
- レイアウトプリミティブは制限対象から除外できる。
- `sx` prop の中身まで縛るのはコスト過大なので、ここは内部ルール + レビューで運用する。

---

## D-3: API 設計方針

**決定**: 中立な独自 Props（MUI 透過ではない）を採用する。

**根拠**:
- 将来の MUI 差し替え時にサービス側を無修正で維持するためには、Props 層から MUI 依存を排除する必要がある。
- 「時間をかけて堅牢に」「将来の拡張性を見据えた設計」という方針に最も合致する。
- 移行コストは増えるが、本対応は急いでいない。

**却下した代替案**:
- MUI Props 透過 → 移行時にサービス側を全更新する必要が出る。

---

## D-4: ライブラリ選定の前提（市場調査結果）

**決定**: 当面は MUI を継続利用、将来差し替えの可能性を残す設計とする。

**根拠**:
- 2026 年時点のトレンドはヘッドレス + Tailwind + RSC ファースト。
- MUI チームが Base UI v1.0（2026/02）として公式の脱出口を提供している。
- 本リポジトリは MUI X（DataGrid / DatePicker 等）に依存していないため、差し替え自由度が高い。
- shadcn/ui / Mantine も現実的な選択肢として残る。

詳細な市場調査結果は省略（議論時に整理済み、Issue #2900 の背景説明を参照）。

---

## D-5: 設計哲学の 5 軸

**決定**:
- アクセシビリティ: 努力義務（重要部品は WCAG AA 相当）
- 型安全: 厳密 Union（`color: 'primary' | 'secondary' | ...`）
- 構造: ハイブリッド（単純部品は設定駆動、複合部品はコンポジション）
- エスケープハッチ: `className` のみ許容（`sx` / `style` 禁止）
- ライブラリ非依存度: Props 100% 独自定義

**根拠**:
- アクセシビリティ「必須」は工数過大、「最低限」は将来差し替え時に劣化リスク。「努力義務」が中庸。
- 厳密 Union は中立 Props 方針と整合し、IDE 補完も効く。
- `sx` を許容すると MUI 依存が Props 層に漏れ、ライブラリ非依存性が崩れる。
- `className` のみなら将来 Tailwind 移行時にも活用できる。

---

## D-6: API 命名規約

**決定**:
- バリアント: `variant`
- サイズ: `sm` / `md` / `lg`
- 色: `color` + セマンティック値
- boolean: 接頭辞なし（`disabled` / `loading` / `open`）
- イベント: React 標準
- Polymorphic: `asChild`（Radix の Slot パターン）

**根拠**:
- `variant` は MUI / shadcn / Mantine すべてで採用される業界デファクト。
- `sm/md/lg` は Tailwind 系の現代標準で、簡潔・拡張容易。
- セマンティック色は意図ベース（視覚的命名は禁止）で、テーマ切替時にも壊れない。
- `asChild` は型推論が単純で、Next.js の `<Link>` のような外部コンポーネント統合が容易。

---

## D-7: デザイントークンの実装方式

**決定**: CSS 変数（`:root`）+ TS 型定義の組み合わせ。

**根拠**:
- MUI / shadcn / Tailwind すべてで CSS 変数を消費可能。ライブラリ非依存性を構造的に担保する唯一の方法。
- 動的テーマ切替（ライト・ダーク・サービス別）が CSS のみで可能。
- 将来どのライブラリに移っても、トークン定義は無変更で持ち越せる。

**却下した代替案**:
- 純粋な TS オブジェクト → JS 経由でしか使えず、CSS から参照できない。
- Tailwind config 直書き → Tailwind ロックイン。

### 実装時に判明した制約（PR 0-1）

MUI の `palette` には CSS 変数（`var(--...)`）を渡せない。MUI 内部の
`alpha()` / `decomposeColor()` が色値の文字列パースを行うため、リテラル
の色形式（`#xxxxxx` / `rgb()` 等）が要求される。

このため `libs/ui/src/styles/theme.ts` の `palette` は具体的な色値を保持し、
`tokens.css` の Primitive と一致させる方針を採った。borderRadius / boxShadow /
transition 等の非カラー値は MUI 内部での色解析が走らないため、CSS 変数を
そのまま参照できる。

将来 Phase 1 以降で実装する独自ラッパーコンポーネント（`Button` 等）は MUI
Theme を経由せず、`tokens.css` の CSS 変数を直接参照することで、ライト/
ダーク・サービス別アクセント等のテーマ切替に追従させる。

## D-8: Storybook の採用と配信

**決定**: Storybook v8（Vite ビルダー）を採用、CDK で `storybook.nagiyu.com` に dev デプロイ。

**根拠**:
- 共通 UI ライブラリの試行錯誤には視覚プレビューが事実上必須。
- ライト・ダーク・サービス別アクセントの動作確認が一画面でできる。
- AI（Claude）の生成相性が最も良い（文献量が圧倒的）。
- Claude が PR を出した後、人が dev 環境の Storybook で視覚レビューする運用が、既存の対応フローと整合する。
- prod 環境への配信は不要（社内開発確認用途のみ）。

---

## D-9: テスト戦略

**決定**:
- ユニットテスト: `tests/` 配下、Testing Library
- a11y: `jest-axe` 新規導入、ユニットテスト内に同居
- ビジュアルリグレッション: Playwright + Storybook（既存 Playwright を活用）
- カバレッジ閾値: 90%（既定 80% より厳しめ）
- Stories 配置: `src/` 配下のコンポーネント隣接

**根拠**:
- `jest-axe` で a11y を継続的に担保（A-1 努力義務の実装手段）。
- Playwright が既に導入済みのため、Chromatic（有料 SaaS）の代替として活用可能。
- 共通 UI は性質上カバレッジを上げやすく、品質要求も高いため閾値を引き上げる。
- Stories はテストではなくカタログ用途のため、CLAUDE.md のテスト分離方針の対象外として扱う。

---

## D-10: 段階移行計画

**決定**: コンポーネント単位（縦切り）で Phase 1〜4 を順次実装。各 Phase は 3 PR 構造（実装 / 置換 / ESLint 強制）。

**根拠**:
- E で決めた Phase 構造（Button → Select → Card → 残り）と完全に一致し、自然な流れになる。
- 各 Phase の PR を分割することで、レビュー観点が明確になり、問題発生時の切り戻しが容易。
- ビッグバン（一括置換）は PR が巨大になりレビュー不能。
- サービス単位（横切り）は複数コンポーネントを同時対応する必要があり負担が重い。

---

## D-11: ガバナンス

**決定**:
- 新規追加基準: 2 サービス以上で使う見込み
- MUI 直接利用例外: ESLint disable + 理由コメント必須
- 破壊的変更: major bump、同 PR で全サービス更新
- 改修フロー: サービス側で実装止めず ESLint disable で逃がし、Issue で後追い拡張

**根拠**:
- 個人開発の小規模性を考慮し、過剰なプロセスは避ける。
- ESLint disable に理由を必須化することで、grep で例外箇所を後追いできる。
- モノレポ前提のため、破壊的変更を同 PR で全更新するのが最も簡素。
