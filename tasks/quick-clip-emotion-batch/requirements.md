# さくっとクリップ 感情分析バッチ分割対応 - 要件定義

---

## 1. ビジネス要件

### 1.1 背景・目的

さくっとクリップ（Quick Clip）のハイライト抽出処理において、感情分析（Emotion Analysis）が長尺動画で正常に完了しない問題が発生している。

文字起こし（Transcription）は音声ファイルを 24MB のチャンクに分割して Whisper API に送信しているが、感情分析は文字起こし済みの全セグメントを一括で GPT に送信している。長尺動画では数百件のセグメントが生成されるため、1つのリクエストに詰め込むと GPT の出力トークン上限に抵触し、結果が空または途中で切れる事象が発生する。

本対応では、感情分析においても文字起こしと同様にセグメントをチャンク分割して順次処理することで、長尺動画でも安定して感情スコアを取得できるようにする。

### 1.2 対象ユーザー

- さくっとクリップを利用する動画投稿者

### 1.3 ビジネスゴール

- 長尺動画（目安: 文字起こしセグメントが 50 件超）でも感情分析が正常に完了すること

---

## 2. 機能要件

### 2.1 ユースケース

#### UC-001: 感情分析のチャンク分割処理

- **概要**: 文字起こしセグメント数が上限を超える場合、複数回に分けて感情分析 API を呼び出す
- **アクター**: バッチ処理（Quick Clip Batch Runner）
- **前提条件**: 文字起こしが正常に完了し、1件以上のセグメントが存在する
- **正常フロー**:
  1. 全セグメントを `SEGMENTS_PER_CHUNK`（= 50）件ごとのチャンクに分割する
  2. 各チャンクに対して感情分析 API を順次呼び出す
  3. 各チャンクの結果を結合して返す
- **代替フロー**: セグメントが 50 件以下の場合は従来通り 1 回の API 呼び出しで処理する（チャンク数が 1 になるだけで同一のロジックで動作する）
- **例外フロー**: チャンク単位でリトライ（最大 3 回）。タイムアウト（600 秒）時はリトライせずエラーをスロー。感情分析全体の失敗は graceful degradation（motion・volume スコアのみでハイライト抽出を継続）

### 2.2 機能一覧

| 機能ID | 機能名 | 説明 | 優先度 |
| ------ | ------ | ---- | ------ |
| F-001 | 感情分析チャンク分割 | セグメントを一定件数ごとに分割して順次 API 呼び出しを行う | 高 |

---

## 3. 非機能要件

### 3.1 パフォーマンス要件

| 項目 | 要件 |
| ---- | ---- |
| 処理方式 | チャンクごとに順次実行（並列不可。API レート制限・コスト考慮） |
| チャンクサイズ | 50 セグメント/チャンク（日本語 50 件 ≈ 3,500 入力トークン） |

### 3.2 信頼性要件

- チャンクごとに既存のリトライ（最大 3 回）・タイムアウト（600 秒）ロジックを適用する
- 1チャンクの失敗は感情分析全体の失敗として扱い、graceful degradation を維持する

---

## 4. 受け入れ条件

- [ ] 50 件以下のセグメントでは API が 1 回だけ呼ばれる
- [ ] 51 件のセグメントでは API が 2 回呼ばれ、結果が結合されて返る
- [ ] チャンク単位でリトライが動作する（2チャンク目の失敗時にリトライが走ること）
- [ ] 既存のすべての単体テストが引き続きパスする
- [ ] Lint・型チェックが通過する

---

## 5. スコープ外

- ❌ 並列チャンク処理（コスト・レート制限の観点から対象外）
- ❌ チャンクサイズの動的調整
- ❌ 感情分析のタイムアウト値の変更
