# Admin エラー検知 - 要件定義

## 1. ビジネス要件

### 1.1 背景・目的

Stock Tracker サービスでは CloudWatch Logs にアラームを設定し、SNS トピックまで連携している。
しかし、SNS サブスクリプションは手動でメールのみを想定しており、管理者がリアルタイムにエラーを検知する手段として不十分。

Admin サービスにプッシュ通知エンドポイントを新設し、Admin をインストールした端末（PWA）に対してリアルタイムにアラームを配信できる仕組みを構築する。

また、今後新たに CloudWatch Alarms を追加する際は Admin 用の SNS トピックへ通知する設計に統一し、既存のアラームも段階的に移行する。

### 1.2 対象ユーザー

- **プライマリーユーザー**: プラットフォーム管理者（Admin アプリをインストールした端末を持つ人）

### 1.3 ビジネスゴール

- CloudWatch Alarms が発火した際に、Admin サービスをインストールしている管理者端末にプッシュ通知が届く
- SNS → Admin サービス経由のプッシュ通知フローを確立し、メール以外の通知手段を持つ
- 将来の新規 CloudWatch Alarms 追加時に Admin 通知トピックへの接続が標準化される
- 既存 Stock Tracker の CloudWatch Alarms を新トピックへ段階的に移行できる

## 2. 機能要件

### 2.1 ユースケース

#### UC-001: SNS 通知を受信してプッシュ通知を送信する

- **概要**: CloudWatch Alarm が SNS トピックに通知し、Admin サービスがそれを受け取ってプッシュ通知を送信する
- **アクター**: AWS CloudWatch Alarms（自動）
- **前提条件**: Admin SNS トピックが作成されており、Admin サービスの HTTP エンドポイントがサブスクリプションとして登録されている
- **正常フロー**:
    1. CloudWatch Alarm がアラーム状態に遷移し、SNS トピックに通知する
    2. SNS が Admin サービスの `/api/notify/sns` エンドポイントに POST する
    3. Admin サービスが SNS サブスクリプション確認（初回）または通知本文を検証する
    4. Admin サービスが DynamoDB から有効なプッシュサブスクリプションを取得する
    5. Web Push API 経由で各サブスクリプションにプッシュ通知を送信する
- **代替フロー**: 有効なサブスクリプションが存在しない場合はプッシュ通知スキップ
- **例外フロー**: SNS 署名検証失敗 → 401 を返しログに記録する

#### UC-002: 管理者端末がプッシュ通知を購読する

- **概要**: Admin サービスにアクセスした管理者が、プッシュ通知の許可を与えてサブスクリプションを登録する
- **アクター**: 管理者ユーザー
- **前提条件**: Admin サービスにログイン済み、ブラウザが Web Push をサポートしている
- **正常フロー**:
    1. ダッシュボードに「通知を有効にする」ボタンが表示される
    2. ユーザーがボタンをクリックするとブラウザの通知許可ダイアログが表示される
    3. 許可後、サービスワーカーが PushSubscription を取得する
    4. Admin サービスの `/api/notify/subscribe` に POST してサブスクリプションを登録する
    5. DynamoDB にサブスクリプション情報が保存される
- **代替フロー**: 通知を拒否した場合は登録せずにスキップ
- **例外フロー**: サービスワーカー登録失敗 → ユーザーにエラー表示

#### UC-003: プッシュ通知の受信

- **概要**: CloudWatch Alarm 発火時にブラウザ（またはホーム画面追加済みの Admin PWA）にプッシュ通知が届く
- **アクター**: 管理者ユーザー
- **前提条件**: UC-002 が完了し、サービスワーカーが登録されている
- **正常フロー**:
    1. サーバーからプッシュイベントが配信される
    2. サービスワーカーが `push` イベントを受け取る
    3. サービスワーカーがシステム通知を表示する（タイトル・本文・アイコン）
    4. 通知タップで Admin ダッシュボードを開く

## 3. 非機能要件

### 3.1 パフォーマンス要件

| 項目             | 要件                                       |
| ---------------- | ------------------------------------------ |
| 通知到達時間     | SNS 発火から 10 秒以内にプッシュ通知を送信 |
| Web Push 送信数  | 同時 100 サブスクリプション以内を想定      |

### 3.2 セキュリティ要件

- **SNS 署名検証**: AWS SNS の証明書を使って HTTP 通知の署名を必ず検証する（偽造通知を拒否）
- **VAPID 認証**: Web Push Protocol の VAPID 鍵ペアを使用し、サーバーの正当性を証明する
- **VAPID 秘密鍵管理**: AWS Secrets Manager に保存し、Lambda 環境変数で参照する
- **サブスクリプション所有者確認**: プッシュサブスクリプション登録は認証済みユーザーのみ可能

### 3.3 可用性要件

| 項目             | 要件                                                 |
| ---------------- | ---------------------------------------------------- |
| SNS サブスクリプション | SNS が HTTP エンドポイントに 3 回失敗した場合の DLQ を設定する（任意） |

### 3.4 保守性・拡張性要件

- **新規アラーム追加**: 将来 CloudWatch Alarms を追加する際は Admin SNS トピック ARN を使用する設計に統一する
- **既存アラームの段階移行**: 既存 Stock Tracker アラームは既存トピックを維持しつつ Admin トピックへも通知を追加する

## 4. ドメインオブジェクト

| エンティティ           | 説明                                                                    |
| ---------------------- | ----------------------------------------------------------------------- |
| SNS トピック（Admin 用）| Admin エラー通知専用の SNS トピック                                     |
| プッシュサブスクリプション | ブラウザの Web Push API が生成する `PushSubscription` 情報              |
| CloudWatch Alarm       | AWS リソース異常を検知してアラーム状態に遷移するルール                  |
| VAPID 鍵ペア           | Web Push Protocol で使用するサーバー認証用の公開鍵・秘密鍵              |

---

## 5. スコープ外

- ❌ Admin サービス以外のサービスへのプッシュ通知（例: Stock Tracker ユーザーへの通知）
- ❌ メール通知の廃止・変更（既存の SNS サブスクリプションはそのまま維持）
- ❌ 通知履歴の閲覧 UI
- ❌ 通知フィルタリング設定 UI（どのアラームを通知するか選択する機能）
- ❌ Stock Tracker 以外の既存サービスのアラーム移行（段階移行は Stock Tracker から開始）

## 6. 用語集

| 用語                  | 定義                                                                              |
| --------------------- | --------------------------------------------------------------------------------- |
| Web Push              | W3C 標準のブラウザプッシュ通知プロトコル。サービスワーカー経由で動作する          |
| VAPID                 | Voluntary Application Server Identification。Web Push サーバー認証の仕組み        |
| PushSubscription      | ブラウザが生成するプッシュ通知の宛先情報（endpoint・keys）                        |
| SNS サブスクリプション | SNS トピックの配信先として登録された HTTP/HTTPS エンドポイントまたはメールアドレス |
| DLQ                   | Dead Letter Queue。処理失敗したメッセージを保存する SQS キュー                    |
