# NPM パッケージ管理 (2026年第13週) - 要件定義書

---

## 1. ビジネス要件

### 1.1 背景・目的

2026年第13週の週次 npm 管理レポート（Issue #2477）において、モノレポ全体に影響する npm パッケージの問題が検出された。
セキュリティ脆弱性（Critical 1件・High 2件）が存在するため、早急な対応が必要である。
また、パッケージの更新対応・重複解消によって保守性を向上させる。

### 1.2 対象ユーザー

- プラットフォーム開発者・保守担当者

### 1.3 ビジネスゴール

- セキュリティ脆弱性（Critical/High）をゼロにする
- `npm audit` の出力で Priority 1 の脆弱性が残らない状態にする
- パッケージ管理の健全性を維持する

---

## 2. 機能要件

### 2.1 ユースケース

#### UC-001: セキュリティ脆弱性の修正

- **概要**: `npm audit` で検出された Critical/High 脆弱性を修正する
- **アクター**: 開発者
- **前提条件**: ルートで `npm audit` を実行すると脆弱性が検出される
- **正常フロー**:
    1. 各脆弱性の依存チェーンを確認する
    2. 修正方法（パッケージ更新 / overrides 設定）を決定する
    3. 修正を適用し `npm audit` で Critical/High がゼロになることを確認する
    4. ビルド・テストが通過することを確認する
- **代替フロー**: `npm audit fix` で自動修正が可能な場合はこれを利用する
- **例外フロー**: 直接修正できない場合は `overrides` / `resolutions` を package.json に設定する

#### UC-002: パッケージ更新の適用

- **概要**: `npm outdated` で検出されたパッケージを更新する
- **アクター**: 開発者
- **前提条件**: 更新可能なパッケージが存在する
- **正常フロー**:
    1. 更新対象パッケージをリストアップする
    2. Breaking Change がないか確認する
    3. 順次更新し、ビルド・テストを確認する
- **例外フロー**: Breaking Change がある場合は対応コストを見積もり、スコープ外とする

#### UC-003: 重複 devDependencies のルート統合

- **概要**: 複数ワークスペースで重複している `devDependencies` をルートに統合する
- **アクター**: 開発者
- **前提条件**: `aws-sdk-client-mock` が3ワークスペース以上で重複して宣言されている
- **正常フロー**:
    1. 重複パッケージとバージョンを確認する
    2. ルート `package.json` の `devDependencies` に移行する
    3. 各ワークスペースから削除し、テストが通過することを確認する

### 2.2 機能一覧

| 機能ID | 機能名 | 説明 | 優先度 |
| ------ | ------ | ---- | ------ |
| F-001 | handlebars 脆弱性修正 | ts-jest 経由の handlebars critical 脆弱性を解消する | 高 |
| F-002 | path-to-regexp 脆弱性修正 | aws-sdk-client-mock → sinon → nise 経由の high 脆弱性を解消する | 高 |
| F-003 | picomatch 脆弱性修正 | jest 内部の picomatch high 脆弱性を解消する | 高 |
| F-004 | パッケージ更新 | AWS SDK・Next.js 等の定期更新を適用する | 中 |
| F-005 | devDependencies 重複解消 | aws-sdk-client-mock をルートに統合する | 低 |

UI 変更なし。

## 3. 非機能要件

### 3.1 パフォーマンス要件

| 項目 | 要件 |
| ---- | ---- |
| 応答時間 | 既存の CI パイプラインの実行時間を維持する |

### 3.2 セキュリティ要件

- `npm audit` の Critical / High 件数がゼロになること
- 修正後に新たな Critical / High 脆弱性が導入されないこと

### 3.3 可用性要件

| 項目 | 要件 |
| ---- | ---- |
| 稼働率 | 既存サービスの稼働率を維持する（パッケージ更新に伴うデグレを起こさない） |

### 3.4 保守性・拡張性要件

- 修正後も既存のビルド・Lint・テストがすべて通過すること
- 週次 npm チェックで検出件数が増加しないこと

## 4. ドメインオブジェクト

| エンティティ | 説明 |
| ----------- | ---- |
| 脆弱性 | npm audit で検出されるセキュリティ問題（severity・依存チェーン情報を持つ） |
| パッケージ | npm ワークスペース内で管理される依存パッケージ |
| ワークスペース | モノレポ内の各サービス・ライブラリ・インフラ単位 |

---

## 5. スコープ外

- ❌ next-auth のメジャーバージョン変更（v5-beta は意図的に使用中）
- ❌ TypeScript v6 へのアップグレード（Breaking Change の影響調査が別途必要）
- ❌ @auth/core のダウングレード（next-auth v5 beta が要求するバージョンに依存）
- ❌ Moderate / Low 脆弱性の完全解消（Priority 1 のみ本タスクのスコープ）
