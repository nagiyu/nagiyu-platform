## 1. 背景・目的

既存の さくっとクリップ に対して、以下の問題を解決するリファクタリングを行う。

1. **アップロード後の進捗フィードバック不足**: アップロード完了後、解析・切り出しがどの段階にあるか全くわからない
2. **ジョブ失敗の検知漏れ**: AWS Batch コンテナが強制終了（OOM・スポット中断等）した場合、DynamoDB のステータスが PROCESSING のまま永続化し、クライアントが解析中を表示し続ける
3. **DynamoDB TTL の不整合**: Job レコードは 24 時間で自動削除されるが、Highlight レコードには TTL が設定されておらず永続化される
4. **ユーザー制約の未通知**: アップロード中はタブを閉じてはいけない、データが 24 時間で失効するといった制約がユーザーに伝わっていない

---

## 2. 機能要件

### 2.1 UX 改善: アップロード後の進捗フィードバック

#### F-R-001: アップロード進捗表示

- アップロード中、バイト単位の進捗をプログレスバーで表示する
- 小ファイル（< 5 GB）: `XMLHttpRequest` の `upload.onprogress` イベントを使用してバイト進捗を取得する
- 大ファイル（≥ 5 GB）: マルチパートアップロードの各チャンク（500 MB）完了時に進捗を更新する
- 進捗は 0〜100% で表示し、アップロードボタンのラベルも動的に更新する
- アップロード完了後（SCR-002 遷移前）に 100% を表示する

#### F-R-002: 解析ステージ表示

- 処理中画面（SCR-002）に 3 段階のステッパー（アップロード → 解析 → 切り出し）を表示する
- 解析ステージでは、現在のサブステージを補足テキストとして表示する
    - `downloading`: 「ダウンロード中」
    - `analyzing`: 「解析中」
    - `aggregating`: 「集計中」
- ポーリング間隔を 10 秒から **5 秒** に短縮する
- Batch コンテナは各ステップ開始前に `batchStage` フィールドを DynamoDB に書き込む

#### F-R-003: クリップ生成進捗表示

- 見どころ確認画面（SCR-003）にクリップ生成の進捗を「X / Y 件生成済」として表示する
- X = `clipStatus === 'GENERATED'` の件数、Y = ハイライト総件数
- クリップ生成中（`hasPendingOrGenerating === true`）のときのみ表示する

---

### 2.2 ジョブ失敗検知の改善

#### F-R-004: AWS Batch を正とした状態管理

- AWS Batch の job ID（以下 `batchJobId`）を Job レコードに保存する
- Web API は DynamoDB の `status` フィールドに依存せず、`DescribeJobsCommand` で AWS Batch の実状態を取得してジョブ状態を返す
- Batch コンテナは `status` フィールドの更新を行わない。代わりに以下のみを行う:
    - 各処理ステップ開始前に `batchStage` を更新する
    - 失敗時に `errorMessage` を DynamoDB に書き込む
- Web API の GET リクエストは DynamoDB への書き込みを行わない（read-only）
- DynamoDB に残る PROCESSING 放置レコードは 24 時間の TTL で自動削除されるため、修正不要

#### F-R-005: Batch 状態のマッピング

Web API は AWS Batch の状態を以下のようにマッピングしてクライアントに返す:

| `batchJobId` | AWS Batch 状態 | クライアントへの status |
|-------------|--------------|----------------------|
| 未設定（null/undefined） | N/A | `PENDING` |
| 設定済み | SUBMITTED / PENDING / RUNNABLE / STARTING | `PENDING` |
| 設定済み | RUNNING | `PROCESSING`（+ DynamoDB の `batchStage` を添付） |
| 設定済み | SUCCEEDED | `COMPLETED` |
| 設定済み | FAILED | `FAILED`（+ DynamoDB の `errorMessage` を添付） |

---

### 2.3 DynamoDB TTL 整合

#### F-R-006: Highlight TTL の設定

- Batch コンテナが Highlight を作成する際、親 Job の `expiresAt`（Unix 秒）を各 Highlight レコードに設定する
- DynamoDB テーブルの TTL 属性（`expiresAt`）は既存のまま変更しない
- これにより Job と Highlight が同じタイミングで自動削除される

---

### 2.4 ユーザー制約の通知

#### F-R-007: アップロード画面での制約通知（SCR-001）

- ページ内注意書きとして以下を常時表示する:
    - 「アップロード中はタブを閉じないでください」
    - 「データは 24 時間で自動削除されます」

#### F-R-008: 処理中画面での制約通知（SCR-002）

- ページ内注意書きとして以下を表示する:
    - 「タブを閉じても処理は続きます。URLを控えておくと後で確認できます」
    - 「データの有効期限: ◯◯◯◯/◯◯/◯◯ ◯◯:◯◯」（`expiresAt` から算出した実際の日時）

#### F-R-009: 見どころ確認画面での有効期限表示（SCR-003）

- ページ内に以下を表示する:
    - 「データの有効期限: ◯◯◯◯/◯◯/◯◯ ◯◯:◯◯」（`expiresAt` から算出した実際の日時）

## 3. 非機能要件

### 3.1 ポーリングコスト

- ステータスページのポーリング間隔を 5 秒にしても、DynamoDB の read コストは無視できるレベル（On-Demand pricing で月数円〜数十円/100 同時ユーザー）
- `DescribeJobsCommand` の呼び出しはポーリングごとに 1 回。AWS Batch API の rate limit（100 req/s）に対して問題なし

### 3.2 後方互換性

- クライアントへの API レスポンスの `status` フィールドの値（`PENDING` / `PROCESSING` / `COMPLETED` / `FAILED`）は変更しない
- DynamoDB の内部スキーマは変更するが、外部 API の型は維持する

---

## 4. 受け入れ条件

| ID | 条件 |
|----|------|
| AC-001 | 小ファイルアップロード時、プログレスバーが 0→100% まで更新される |
| AC-002 | 大ファイルアップロード時、チャンクごとに進捗が更新される |
| AC-003 | SCR-002 にステッパーが表示され、ジョブが PROCESSING のときステージ名が表示される |
| AC-004 | SCR-002 のポーリング間隔が 5 秒になっている |
| AC-005 | SCR-003 でクリップ生成中に「X / Y 件生成済」が表示される |
| AC-006 | Batch コンテナが OOM 等で強制終了されてもクライアントは適切に FAILED を表示する |
| AC-007 | Job 作成後、DynamoDB の Job レコードに `batchJobId` が保存される |
| AC-008 | Highlight レコードに `expiresAt` が設定され、Job と同じタイミングで自動削除される |
| AC-009 | SCR-001 に制約通知（タブ閉じ禁止・24時間失効）が表示される |
| AC-010 | SCR-002 に有効期限の具体的な日時が表示される |
| AC-011 | SCR-003 に有効期限の具体的な日時が表示される |
| AC-012 | Lint・型チェックがすべて通過する |
