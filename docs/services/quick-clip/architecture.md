# アーキテクチャ

## システム概要

さくっとクリップ (QuickClip) はサーバーレスアーキテクチャを採用した動画見どころ抽出サービスです。
フロントエンドは Next.js を Lambda で実行し、動画の解析・見どころ抽出は AWS Batch (Fargate) で非同期処理します。
クリップ生成と ZIP 生成はそれぞれ専用の Lambda に委譲し、Web API から必要に応じて非同期 Invoke する構成です。
動画ファイルは Next.js / Lambda を経由せず S3 へ直接アップロードされます。

## コンポーネント

| コンポーネント | 役割 | 技術 |
| -------------- | ---- | ---- |
| **CloudFront** | CDN、SSL/TLS終端 | Amazon CloudFront |
| **Lambda (Web)** | Next.js SSR・API Routes | AWS Lambda + Lambda Web Adapter |
| **Lambda (clip)** | クリップ切り出し・S3保存・DynamoDB更新 | AWS Lambda + FFmpeg |
| **Lambda (zip)** | クリップ収集・ZIP組み立て・S3保存 | AWS Lambda |
| **S3** | 動画ファイル・クリップ・ZIPのストレージ | Amazon S3 |
| **DynamoDB** | ジョブ・見どころ管理（シングルテーブル設計） | Amazon DynamoDB |
| **Batch** | 動画解析・見どころ抽出 | AWS Batch (Fargate) + FFmpeg |
| **ECR** | コンテナイメージレジストリ | Amazon ECR |

## 設計決定記録（ADR）

### ADR-001: 見どころ抽出アルゴリズム（貪欲クリップ選択）の採用

**背景・問題**

Phase 1 では AI 非依存で動作する見どころ抽出が必要。映像フレームの変化量（ピクセル差分）と音量の大きさそれぞれから得られる生スコアリストを、重複なく最大 20 件のクリップにまとめる方法を決定する必要があった。

**決定**

Motion スコアリストと Volume スコアリストをスコア降順にソートし、両者を交互に 1 件ずつ取り出しながら以下の手順で貪欲選択を行う。

1. 取り出した peak 時刻 `t` に対して理想クリップ `[t - 10, t + 10]` を計算し、動画端でクランプする
2. 採用済みクリップと区間が重複する場合は両区間のユニオンで既存クリップを拡張統合する（連鎖統合も許容）
3. 重複がない場合は新規クリップとして追加する
4. 採用数が 20 件に達したら終了する
5. 最後に開始時間昇順にソートして順序番号を付与する

同一時間帯が両手法で検出された場合は 1 件にマージし、抽出根拠を「両方（both）」とする。各クリップは抽出根拠（`motion` / `volume` / `both`）を持ち、UI 上で「モーション」「音量」「両方」として表示する。

**根拠・トレードオフ**

- AI 不使用のため実装コストが低く処理速度が速い
- 貪欲選択により時間的に近いピークを 1 つのクリップに統合でき、過剰な分割を防げる
- 意味的な見どころは検出できないが、Phase 2 の AI 統合（文字起こしベースの見どころ抽出）で補完する方針
- 抽出ロジックはサービスとして抽象化されており、将来的な手法の追加（AI 解析など）は具象クラスを追加するだけで対応できる

### ADR-002: クリップ生成を専用 Lambda に分離する

**背景・問題**

見どころ抽出バッチはクリップ生成を含まず、ハイライトを `clipStatus: PENDING` で保存するのみとした。クリップ生成（FFmpeg による切り出し）をどのコンポーネントに担わせるかを決定する必要があった。バッチに含める案と Lambda に分離する案が候補として挙がった。

**決定**

クリップ生成は専用の clip-regenerate Lambda に委譲し、Web API（見どころ一覧取得・時間調整・リトライ）から非同期 Invoke する。ZIP 生成も同様に専用の zip-generator Lambda に委譲する。

**根拠・トレードオフ**

- バッチの責務を「動画解析（抽出）のみ」に絞り、単一責任を保つ
- 初回表示時・時間調整時・リトライ時のいずれからも個別に Invoke できるため、UX 上のクリップ再生成フローを柔軟に実現できる
- Web API から Lambda を直接 Invoke することで、S3 ポーリングや Batch 再投入より低遅延かつシンプルな連携が可能
- 単体テストを FFmpeg 環境なしで実行できる（コアロジックと FFmpeg 依存実装を分離）
- トレードオフ: Lambda のタイムアウト上限（15 分）があるため、非常に長い区間のクリップ生成は注意が必要

### ADR-003: 大容量ファイル対応（マルチパートアップロード）

**背景・問題**

アップロード上限を 20 GB と定めたため、S3 の単一 PUT リクエスト上限（5 GB）を超えるファイルに対応する必要があった。Next.js / Lambda を経由させると Lambda のメモリ・タイムアウト制約に抵触するため、直接 S3 へアップロードする方式が前提となる。

**決定**

ファイルサイズに応じてアップロード方式を切り替える。

- **5 GB 未満**: `PutObject` の Presigned URL を生成してブラウザから単一 PUT を行い、ジョブ作成と同時に Batch ジョブをサブミットする
- **5 GB 以上**: `CreateMultipartUpload` で uploadId を取得し、500 MB チャンク単位の `UploadPart` Presigned URL 群を返す。アップロード完了後にブラウザが別途 complete-upload API を呼び出し、`CompleteMultipartUpload` と Batch ジョブサブミットを行う

**根拠・トレードオフ**

- 5 GB 未満は既存のシンプルな単一 PUT フローを維持し、実装複雑度を最小化できる
- マルチパートにより 20 GB ファイルを 40 パーツ（500 MB × 40）に分割して転送でき、S3 の制約に適合する
- Batch の S3 ダウンロードはストリーミング処理（メモリに全量展開しない）とすることで OOM を回避する
- トレードオフ: マルチパートフローはブラウザ側とサーバー側の連携が複雑になる（ETag 管理・complete 呼び出しが必要）

### ADR-004: Batch Job Definition のファイルサイズ別選択

**背景・問題**

アップロード上限が 20 GB と大きく、ファイルサイズによって必要なメモリ・CPU・タイムアウト・エフェメラルストレージが大きく異なる。単一の Job Definition では小さいファイルに過剰なリソースを割り当てるか、大きいファイルで OOM が発生するかのいずれかになる。

**決定**

ファイルサイズに応じて 3 種類の Job Definition を使い分ける。ジョブ投入時にファイルサイズでティアを自動選択する。

| サイズ | 対象ファイルサイズ | vCPU | メモリ | タイムアウト | エフェメラルストレージ |
|-------|----------------|------|------|----------|----------------|
| small | 1 GiB 未満 | 1 | 4 GB | 1 時間 | 20 GB（デフォルト） |
| large | 1 GiB 以上 8 GiB 未満 | 2 | 8 GB | 3 時間 | 30 GB |
| xlarge | 8 GiB 以上 | 4 | 16 GB | 8 時間 | 60 GB |

**根拠・トレードオフ**

- ファイルサイズに見合ったリソースを割り当てることでコスト効率と処理速度を両立する
- small ジョブは低リソースで多数並列実行でき、全体的なスループットを向上させる
- xlarge の `maxvCpus: 8` 設定により同時 2 ジョブまで対応する
- トレードオフ: Job Definition が 3 つに増えるため CDK での管理コストが増加する。ティア境界値の妥当性は実運用で検証が必要

### ADR-005: Google IMA SDK による動画広告の採用

**背景・問題**

処理中画面（SCR-002）の待ち時間（数十秒〜数分）を活用して広告収入を確保したい。動画広告の配信方式として、Google IMA SDK（Interactive Media Ads）を使う方式と、独自広告サーバーを立てる方式が候補として挙がった。

**決定**

Google IMA SDK（HTML5）を CDN から動的ロードして使用する。VAST タグ URL は `NEXT_PUBLIC_VAST_TAG_URL` 環境変数でビルド時に注入する。

**根拠・トレードオフ**

- Google IMA SDK は npm パッケージとして提供されていないため、CDN ロード（`https://imasdk.googleapis.com/js/sdkloader/ima3.js`）が公式の唯一の方法
- Google Ad Manager と統合することで広告の在庫管理・ターゲティング・レポートを Google のインフラに委託でき、独自サーバーの構築・運用コストを削減できる
- `NEXT_PUBLIC_VAST_TAG_URL` を未設定にするだけで広告を無効化できるため、開発・E2E テスト環境への影響を完全に排除できる
- トレードオフ: IMA SDK の読み込みは外部 CDN に依存する。広告ブロッカーや SDK ロード失敗時はフォールバック処理（即座にボタン有効化）でサービス継続を保証する

### ADR-006: 広告完了ゲートの設計（AND 条件）

**背景・問題**

動画広告の再生と見どころ抽出処理はそれぞれ独立して完了するため、どちらが先に完了してもユーザーが見どころ確認に進めてしまう問題がある。また、処理完了前に広告が終わった場合・広告完了前に処理が終わった場合の挙動を明確に定義する必要があった。

**決定**

見どころ確認ボタンの表示条件を `job.status === 'COMPLETED' && adFinished` の AND 条件とする。`VideoAd` コンポーネントは `job.status !== 'FAILED' && !adFinished` の条件で表示し続けることで、ステータス変化によるアンマウント・再マウントを防ぐ。

**根拠・トレードオフ**

- AND 条件により「広告が先に終わっても処理完了まではボタンを出さない」「処理が先に完了しても広告終了まではボタンを出さない」の両方を自然に実現できる
- `VideoAd` を `PENDING`/`PROCESSING`/`COMPLETED` すべての状態で継続マウントすることで、ポーリングによるステータス変化時に広告が途中でリセットされることを防ぐ
- FAILED 時は広告を表示しない（失敗時にさらに広告を見せるのはユーザー体験を著しく損なうため）
- 広告エラー・SDK ロード失敗・no fill 時はすべて `onAdFinished()` を即時呼び出すフォールバックにより、広告に起因したサービス利用不能を防ぐ
- トレードオフ: `VideoAd` の `active` フラグによるべき等な `finish()` 実装が必要になる

### ADR-007: 感情スコアによる見どころ抽出（gpt-4o-mini-transcribe + gpt-5-mini Responses API）

**背景・問題**

既存の見どころ抽出は映像変化量（motion）と音量（volume）の2要素のみで、発言内容の感情的な盛り上がりを考慮できなかった。ゲーム実況アーカイブのような長時間配信から「笑いが多い場面」「テンションが上がる場面」を切り出したいニーズに対応するため、感情的強度を第3の見どころソースとして追加する方法を決定する必要があった。

**決定**

OpenAI の文字起こし API（gpt-4o-mini-transcribe）と感情分析（gpt-5-mini Responses API）を組み合わせ、発言内容の感情的強度を見どころスコアの第3ソースとして追加する。

- **文字起こし**: `gpt-4o-mini-transcribe` でセグメント単位のタイムスタンプ付き文字起こしを取得する
- **感情分析**: `gpt-5-mini` Responses API（`zodTextFormat`）でセグメント単位の感情カテゴリ別スコア（laugh / excite / touch / tension: 各 0.0〜1.0）を算出する
- **3ソース round-robin**: motion → volume → emotion → motion → ... の順番で交互にピークを選択し、最大 20 件のハイライトを抽出する
- **感情フィルタ**: ジョブ作成時に `emotionFilter`（laugh / excite / touch / tension / any）を指定できる。`any` は全カテゴリの最大値でスコアリングする
- **graceful degradation**: `OPENAI_API_KEY` 未設定時・API 呼び出し失敗時は感情分析をスキップし、motion・volume のみで既存の動作を継続する（ジョブ全体は FAILED にしない）
- **音声ファイルサイズ対応**: 音声を WAV ではなく MP3（32kbps mono）に変換することで OpenAI API の 25 MB 上限に適合させる。25 MB を超える場合はチャンク分割して順次 transcribe し、タイムスタンプにオフセットを加算して結合する
- **ハルシネーションフィルタリング**: Whisper が無音区間などで繰り返しパターンを出力するハルシネーションをフィルタリングし、信頼性の低いセグメントを除外する

**根拠・トレードオフ**

- gpt-4o-mini-transcribe はコスト効率が高く、セグメント単位のタイムスタンプを取得できるため感情スコアとの時刻マッピングに適している
- Responses API（`zodTextFormat`）により感情スコアを型安全に構造化出力できる
- 3ソース round-robin により既存の motion・volume ソースと感情ソースが均等に選出され、特定ソースへの偏りを防げる
- graceful degradation により `OPENAI_API_KEY` を設定しない環境（ローカル・テスト）では既存動作と完全に互換性を保つ
- トレードオフ: 感情分析が有効な場合、処理時間が +60 秒程度増加する（文字起こし + 感情分析 API 呼び出しのオーバーヘッド）
