# Portal 要件定義書

---

## 1. ビジネス要件

### 1.1 背景・目的

nagiyu.com（ルートドメイン）で Google AdSense 承認を取得し、SEO トラフィックを獲得するためのサイト。

**Portal は「AWS・Next.js を中心とした個人開発の実運用経験を一次情報として発信する技術メディア」として単一目的に運営する。** かつてはルートドメインで Tools（ブラウザ完結型ユーティリティ）を提供し、その後「サービス紹介・ドキュメント＋技術ブログのハイブリッド」構成を採っていたが、AdSense は「サイトの主目的が一つに明確であること」を重視するため、サービス紹介・ドキュメント機能を撤去して技術メディアへ一本化した。経緯と判断根拠は `architecture.md` ADR-001 / ADR-004 を参照。

各サービス（Tools など）はそれぞれのサブドメインで提供・案内し、Portal からは扱わない。運営者が各サービスを実際に設計・運用している事実は、記事の一次情報性を裏づける E-E-A-T シグナルとして位置づける（宣伝・カタログとしては扱わない）。

### 1.2 対象ユーザー

- **プライマリ**: AWS・Next.js でフルスタック開発・個人開発を行うエンジニア（技術記事の読者）
- **セカンダリ**: Google AdSense 審査担当者（コンテンツの有用性・E-E-A-T を評価）

### 1.3 ビジネスゴール

1. **Google AdSense 承認取得**（nagiyu.com ドメインで承認後、各サブドメインへ展開）
2. **SEO トラフィック獲得**（技術記事がロングテールキーワードで流入を生む）
3. **技術メディアとしての信頼性確立**（一次情報・継続更新・著者の実運用実績で E-E-A-T を担保）

## 2. 機能要件

### 2.1 ユースケース

#### UC-001: 技術記事の閲覧

- **概要**: エンジニアがサイトを訪れ、AWS・Next.js などの実運用知見を解説した記事を読む
- **アクター**: エンジニア（読者）
- **前提条件**: なし（認証不要）
- **正常フロー**:
    1. ユーザーがトップページまたは技術記事一覧にアクセスする
    2. 記事タイトル・概要・タグ・公開日が一覧表示される
    3. ユーザーが記事を選択する
    4. 記事本文（コード・図を含む）が表示される
- **代替フロー**: 記事下部の関連記事から別の記事へ回遊する

#### UC-002: 運営者・運営方針の確認（E-E-A-T）

- **概要**: 読者・審査担当者が、誰が・どのような方針で記事を書いているかを確認する
- **アクター**: 読者 / AdSense 審査担当者
- **前提条件**: なし（認証不要）
- **正常フロー**:
    1. ユーザーが About ページにアクセスする
    2. 運営者プロフィール・運営方針・編集ポリシーが表示される
    3. 必要に応じてお問い合わせ手段（記事フィードバック中心）を確認する

### 2.2 機能一覧

| 機能 ID | 機能名 | 説明 | 優先度 |
| ------- | ------ | ---- | ------ |
| F-001 | トップページ | ヒーロー（技術メディアであることの提示）＋特集記事＋最新記事 | 高 |
| F-002 | 技術記事一覧・詳細ページ | Markdown ベースの技術記事（目標 25 本以上） | 高 |
| F-003 | 関連記事・タグ動線 | 記事下部の関連記事、記事カードのタグ／カテゴリラベルによる内部リンク強化 | 中 |
| F-004 | About / プロフィールページ | 運営者情報・運営方針・編集ポリシー・お問い合わせ導線（E-E-A-T） | 高 |
| F-005 | お問い合わせページ | Google フォームを主手段とし、記事フィードバック（誤り指摘・質問）を中心に受け付ける | 高 |
| F-006 | 利用規約・プライバシーポリシー | 各種法的ページ | 高 |
| F-007 | Google AdSense 組み込み | 本番環境での広告スクリプト読み込み | 高 |
| F-008 | Google Analytics 組み込み | アクセス解析スクリプト読み込み | 中 |
| F-009 | 構造化データ（JSON-LD） | `WebSite` / `Organization` / `BlogPosting` / `BreadcrumbList` などを各ページに埋め込む | 高 |
| F-010 | OGP / Twitter カード | デフォルト OGP 画像（1200×630）＋ページ別 metadata | 高 |
| F-011 | sitemap / robots | `/sitemap.xml`・robots.txt の動的生成 | 高 |

## 3. 非機能要件

### 3.1 パフォーマンス要件

| 項目 | 要件 |
| ---- | ---- |
| Lighthouse Performance スコア | 90 以上（AdSense 審査・SEO 要件） |
| Lighthouse Accessibility スコア | 90 以上 |
| Core Web Vitals | LCP < 2.5s、FID < 100ms、CLS < 0.1 |

### 3.2 セキュリティ要件

- 認証不要（全ページ公開）
- 外部スクリプト（AdSense / Analytics）は本番環境のみ読み込む
- セキュリティヘッダーを CloudFront レベルで付与（既存インフラの設定を継承）

### 3.3 可用性要件

| 項目 | 要件 |
| ---- | ---- |
| 稼働率 | 99.9%（既存インフラ共通） |
| RTO | 30 分 |
| RPO | N/A（ステートレス・静的コンテンツ） |

### 3.4 保守性・拡張性要件

- コンテンツ追加・更新は Markdown ファイルの編集のみで完結すること
- 既存サービスパターンに準拠（Next.js + MUI + TypeScript strict）

### 3.5 SEO・クローラビリティ要件

- SSG（静的サイト生成）による完全クローラブルなページ生成
- 各ページに適切な `<title>` / `<meta description>` / OGP タグ / canonical URL を設定
- サイトマップ（`/sitemap.xml`）・robots.txt を動的生成し、実在するルートのみを掲載する
- 構造化データ（JSON-LD）を埋め込み、Rich Results Test で検証可能な状態を維持
- E-E-A-T シグナル（著者情報・編集ポリシー・お問い合わせ手段・更新日表示）を About と各記事に配置
- **単一目的の堅持**: サイト内に技術メディアと無関係な導線（自サービスのカタログ・誘導ページ＝doorway）を持たない

## 4. ドメインオブジェクト

| エンティティ | 説明 |
| ----------- | ---- |
| Article | 技術記事（タイトル・概要・タグ・カテゴリ・公開日・更新日・著者・特集フラグ付き） |

> カテゴリは記事カードのラベル（非リンク）としてのみ用い、カテゴリ別の集約ページ（ハブ）は持たない（理由は `architecture.md` ADR-004）。

---

## 5. スコープ外

- 各サービスの紹介・使い方ガイド・FAQ（各サービスのサブドメイン側で対応する）
- コンテンツ管理システム（CMS）の導入
- ユーザー認証・会員機能
- コメント・いいね機能
- タグ別・カテゴリ別の集約ページ（一覧の網羅性より単一目的の明確さを優先）
- 動的なコンテンツ生成（全ページ静的）

## 6. 用語集

| 用語 | 定義 |
| ---- | ---- |
| Portal | nagiyu.com のルートドメインで稼働する技術メディアサイト |
| SSG | Static Site Generation。ビルド時に静的 HTML を生成する方式 |
| E-E-A-T | Experience（経験）・Expertise（専門性）・Authoritativeness（権威性）・Trustworthiness（信頼性）。Google の品質評価基準 |
| doorway | 検索流入の入口だけを目的にした、本来のコンテンツと無関係な誘導ページ。AdSense・検索品質の観点で避ける |
| JSON-LD | JSON-LD 形式の構造化データ。検索エンジンがページ内容を機械的に理解するためのメタデータ |
| AdSense | Google の広告配信プログラム。サイトオーナーが承認を受けることで広告収益を得られる |
