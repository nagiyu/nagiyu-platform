# リストページ・URL 遷移設計（Issue #1823）

## 背景

個人リスト・共有リストのページ構成と URL 遷移が複雑化し、期待した動作になっていない課題がある。
本ドキュメントは、実装前に要件を再整理し、必要なページと URL 遷移を明確化することを目的とする。

---

## 現状課題

1. `/` と `/lists` の責務が重複している
2. URL と画面状態（個人/共有リスト選択）が同期していない
3. `/groups/[groupId]/lists/[listId]` が独立ページとして孤立している
4. `MOCK_PERSONAL_LISTS` が残存している
5. `/lists` がリダイレクト専用ページになっている

---

## 採用 URL 設計

### 基本方針

- リスト操作は `/lists` に集約し、個人/共有の切り替えは GUI で行う
- グループ管理は `/groups` に集約する
- ナビゲーションは「リスト」「グループ」の 2 項目とする

### URL 構成

| URL       | ページ       | 説明                                                     |
| --------- | ------------ | -------------------------------------------------------- |
| `/`       | サービス紹介 | サービス概要・機能紹介を表示するランディングページ       |
| `/lists`  | リスト       | 個人/共有リストの選択と ToDo 操作を単一ページで提供      |
| `/groups` | グループ     | グループ選択、メンバー管理、招待・設定を単一ページで提供 |

### 廃止対象ルート

- `/lists/[listId]`
- `/groups/[groupId]`
- `/groups/[groupId]/lists/[listId]`

---

## 画面遷移方針

- ナビゲーション:
  - `[リスト]` → `/lists`
  - `[グループ]` → `/groups`
- 「ホーム」メニューは廃止し、アプリ名クリック時のみ `/` へ遷移
- `/lists`・`/groups` はページ内状態で選択を切り替え、選択のたびに URL は変更しない
