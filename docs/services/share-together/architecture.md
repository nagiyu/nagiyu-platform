# みんなでシェアリスト (Share Together) アーキテクチャ

## 1. システム概要

みんなでシェアリスト (Share Together) は、nagiyu プラットフォームの Auth サービスと連携した ToDo 共有ツールです。個人の ToDo リスト管理と、複数ユーザーによるグループ共有リストの両方を提供します。

サービスは **core** と **web** の2パッケージで構成されます。core はフレームワーク非依存のビジネスロジック層、web は Next.js による UI および API 層です。バッチ処理は MVP では含まず、将来のプッシュ通知実装時に追加します。

## 2. 重要なアーキテクチャ決定

### 2.1 DynamoDB シングルテーブル設計の採用

**すべてのエンティティを単一テーブルで管理するシングルテーブル設計を採用する。**

- nagiyu プラットフォームの既存サービスが同パターンを採用しており、コードベース全体での一貫性が確保される
- 本サービスで必要なすべてのアクセスパターンが、2つの GSI（ユーザー→グループ逆引き用、メールアドレス検索用）で網羅可能であることを確認した
- DynamoDB のコスト効率と運用の一貫性を両立できる

**検討した代替案**: エンティティ別マルチテーブル設計はシンプルで直感的だが、プラットフォーム標準パターンからの逸脱となるため却下した。GSI を3つ以上設ける案は、PK/SK の工夫で代替可能なため不要と判断した。

### 2.2 Auth Consumer パターンの採用

**Auth サービスが発行した JWT を、`.nagiyu.com` ドメイン配下の Cookie 共有で受け取る Auth Consumer パターンを採用する。**

- Auth サービスとのインテグレーションを JWT 検証のみとする（自前の OAuth 実装なし）
- `.nagiyu.com` ドメインの Cookie 共有による SSO は nagiyu プラットフォームで確立されたアプローチであり、追加のログイン操作なしに利用できる
- セッション戦略は JWT（ステートレス）とし、DB 保存のセッション管理に比べてインフラ複雑度を抑える

**検討した代替案**: Share Together が直接 Google OAuth を実装する方式は Auth サービスとの二重管理になりプラットフォーム方針に反するため却下した。

### 2.3 グループ招待通知をサービス内通知とした理由

**グループへの招待通知はメール送信ではなく、サービス内通知（In-app notification）で実装する。**

- 仕様書の制約として「グループへの招待通知はサービス内通知で対応し、メール通知は対象外」と明示されている
- GroupMembership の `status = PENDING` を招待情報として活用することで、別の通知エンティティを設けずに済み、二重管理を回避できる
- SES / SNS などの追加インフラが不要で、MVP 実装をシンプルに保てる

招待通知の表示に必要なグループ名・招待者名は、GroupMembership に持たせず都度 GetItem で取得する。これはグループ名が変更された際の二重管理を避けるためである。招待件数は少ないため追加取得コストは許容範囲とする。

**検討した代替案**: 当初は招待情報を独立エンティティとして設計したが、GroupMembership の GSI と機能が重複するため削除した。WebSocket によるリアルタイム通知は MVP スコープ外として明示されている。

### 2.4 ページリフレッシュによる最新化

**リアルタイム同期は採用せず、手動の「更新」操作によって最新状態を取得する設計とする。**

- 仕様書の制約として「リアルタイム同期（WebSocket 等）は対象外とし、ページ更新時に最新状態を表示する形で十分」と明示されている
- 成功基準（SC-003）は「ページを更新することで5秒以内に確認できる」であり、手動リフレッシュで達成可能
- WebSocket / Server-Sent Events は MVP スコープ外

**検討した代替案**: 定期ポーリングは Lambda のコールドスタートコストと不要なリクエスト増加をもたらすため MVP では採用しない。

### 2.5 グループ削除時のアプリケーション層カスケード削除

**グループ削除時は、グループに紐づくメンバーシップ・共有リスト・ToDo のすべてをアプリケーション層でトランザクション削除する。**

- DynamoDB にはリレーショナル DB のような外部キー制約がないため、整合性の維持はアプリケーション層で担保する必要がある
- DynamoDB Streams + Lambda によるイベント駆動カスケード削除は MVP には過剰な複雑度のため却下した
- TTL による自動削除は即時削除でないためデータ整合性リスクがあり、採用しない

DynamoDB のトランザクション書き込みは100アイテムの上限があるため、アイテム数が多い場合は複数バッチに分割して逐次実行する。途中で失敗した場合は、それ以前のバッチで削除済みのアイテムを元に戻す補償トランザクションは行わず、グループ削除を不完全な状態とみなさない（孤立アイテムは後続の参照不可により実害がない）。

### 2.6 ユーザー検索をメールアドレス完全一致とした理由

**グループ招待時のユーザー検索は、メールアドレスによる完全一致検索を主とする。**

- Auth サービスは現在 Google OAuth のみをサポートしており、Google アカウントのメールアドレスでユーザーを一意に識別できる（仕様の前提条件）
- メールアドレス完全一致はインデックス設計がシンプルで、GSI2（EmailIndex）による効率的なクエリが可能
- 名前による部分一致検索は DynamoDB では scan が必要となり、MVP には不適切

### 2.7 PWA 実装方針

**MVP では manifest と service worker シェルのみを実装し、プッシュ通知は将来対応とする。**

- 仕様書の追加要件として「PWA 必須（モバイルファースト）」「将来のプッシュ通知フレームワーク（シェルのみ、MVP では未実装）」と明記されている
- シェルを事前に用意することで、将来のプッシュ通知実装時の変更差分を最小化できる
- VAPID キー・SNS・EventBridge を含む完全なプッシュ通知実装は MVP スコープ外

### 2.8 スコープ切り替え時のリスト選択リセット

**表示スコープ（個人／共有）を切り替えた際、選択中のリストは毎回初期状態にリセットされる。**

- 個人と共有で選択中のリストを単一の状態として共用しているため、スコープを切り替えると以前のスコープの選択状態は引き継がれない
- スコープを繰り返し切り替えると個人側の選択状態も毎回リセットされるが、現状のユースケースでは許容範囲と判断した

**検討した代替案**: 個人用と共有用で選択状態をそれぞれ独立して保持し、スコープ復帰時に前回の選択を維持する方式も検討したが、変更範囲が大きくなるため現時点では採用しない。

## 3. データモデル概要

DynamoDB シングルテーブルで以下のエンティティを管理します。

| エンティティ | 説明 |
| ------------ | ---- |
| User | Auth サービスで識別されるアカウント。初回ログイン時に自動生成 |
| PersonalList | ユーザー専用の ToDo リスト。各ユーザーはデフォルトリストを必ず持つ |
| Group | 複数ユーザーが所属する共有単位。オーナー1名と最大4名のメンバーで構成 |
| GroupMembership | ユーザーとグループの紐付け。招待ステータス（招待中・承認済み・拒否済み）を管理 |
| GroupList | グループに紐づく共有 ToDo リスト。メンバー全員が操作可能 |
| TodoItem | 個人リストまたはグループ共有リストに属するタスク単位 |

### GSI 設計の目的

シングルテーブルではパーティションキーが同一のアイテムしか一度に取得できないため、異なる方向からのアクセスパターンを満たすために2つの GSI を設けています。

- **GSI1（UserGroupIndex）**: ユーザーが所属するグループを逆引きするために使用します。グループメンバーシップはグループをパーティションキーとして格納されているため、ユーザー起点での一覧取得には逆引き用インデックスが必要です。保留中の招待の取得も同じ GSI をユーザー ID で検索し、ステータスで絞り込むことで実現します
- **GSI2（EmailIndex）**: グループ招待時にメールアドレスからユーザーを一意に特定するために使用します

## 4. インフラ構成

CDK スタックは以下の構成で管理します。

| スタック | 役割 |
| -------- | ---- |
| DynamoDBStack | シングルテーブル（2 GSI、オンデマンド課金、PITR 有効） |
| EcrStack | Web コンテナイメージリポジトリ |
| LambdaStack | Lambda（コンテナイメージで web を実行） |
| IAMStack | 開発用 IAM ユーザー（dev 環境のみ） |
| CloudFrontStack | CDN（`CloudFrontStackBase` 継承） |

**バッチスタックを含まない理由**: MVP では定期的なバッチ処理が不要です。将来のプッシュ通知実装時に BatchStack・SNS・EventBridge を追加します。`CloudFrontStackBase` を継承することで CDN 設定のメンテナンスコストを削減しています。

**検討した代替案**: ECS/Fargate や App Runner はコスト・管理コストともに Lambda が有利なため採用しない。

## 5. 技術スタック

### 5.1 フロントエンド

- Next.js 15（App Router）+ React 19 + Material-UI v6
- TypeScript strict mode
- PWA: manifest.json + service worker シェル

### 5.2 バックエンド

- Node.js 22 + TypeScript
- Next.js API Routes
- NextAuth v5（Auth Consumer パターン、JWT 検証のみ）

### 5.3 インフラ

- AWS Lambda（コンテナイメージ）
- Amazon DynamoDB（シングルテーブル、オンデマンド課金）
- Amazon CloudFront（CDN）
- Amazon ECR（コンテナリポジトリ）
- AWS CDK（IaC、TypeScript）
