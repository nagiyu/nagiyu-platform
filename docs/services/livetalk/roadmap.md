- **設計判断の永続化**: ADR は本ドキュメントではなく [architecture.md](./architecture.md) に集約する
- **MVP 仕様ドキュメント**: [requirements.md](./requirements.md) / [external-design.md](./external-design.md) / [architecture.md](./architecture.md) に永続化済み（旧 `tasks/livetalk/` から移行・削除済み）

## A. ビジュアル拡張（画像生成）

### 背景・狙い
「キャラが同じ時間軸で生活している」体験を、季節・イベント・ユーザーの嗜好に応じた背景・小物・静止演出（「今日の一枚」）で強化する。Phase 4/5 のコンセプトの自然な延長。

### 妥当性評価

**メリット**:
- 「世界が動いてる」感の強化
- Phase 5 の能動アクション（通知・ノート）に視覚要素を加えられる
- バッチ生成 + S3 + CloudFront のワークフローは既存資産（Phase 5a の勉強バッチパターン）で実装可能

**デメリット / 課題**:
- **Live2D キャラ本体への画像生成適用はリギングの壁で実質不可**（L1 非現実的）。テクスチャ差し替え（L2）も色変え程度が限界
- → **現実的な着地点（L3）**: キャラ本体は Live2D で固定し、**背景・小物・静止演出のみ画像生成で能動更新**
- ⚠️ ライセンス: 桃瀬ひより（Live2D サンプル）の画像改変は規約違反リスク → **オリジナルキャラ移行後に解放**
- コスト（1 枚 数¢〜）、品質ばらつき → 事前生成 + キュレーション前提

### 着手の前提条件 / トリガー
- オリジナルキャラへの移行が完了している（権利問題回避）
- AI 画像生成 API（GPT-image / SDXL 等）の選定・コスト評価
- Phase 5 の周辺演出として組み込む価値が明確になったタイミング

### 位置づけ
Live2D の代替ではなく **周辺を彩る補助**。MVP コア体験を壊さない範囲で導入。

## B. モーション拡張（お辞儀・頷き等）

### 背景・狙い
半目（sleeping）の演出（Phase 4b で実装済み）の延長として、応答内容に応じたモーション（頷き・首かしげ・お辞儀・手の動き等）でキャラの「生きてる感」を強化する。

### 妥当性評価

**メリット**:
- **画像生成と真逆で実装が素直**: 新しい絵が要らない（既存パラメータの時系列を動かすだけ）
- **ライセンスがクリーン**: 絵を改変しないため、桃瀬ひよりでも問題が起きにくい
- 既存の motion 切替（Phase 2c）+ Phase 4 の状態演出の自然な延長
- Live2D のパラメータがある限り、**コードまたは LLM で素直に作れる**

**デメリット / 課題**:
- **モデルのパラメータ依存**: 腕パラメータがなければ手は動かせない
- **2D の表現限界**: 複雑な全身動作・ダンスは不向き（除外妥当）
- 生成したカーブの不自然さ（ガクガク）→ イージング調整が要る

### 実装アプローチ候補

| 方式 | 内容 | 難易度 |
|---|---|---|
| **A1. プログラマティック生成** | コード/数式でキーフレーム生成（お辞儀 = 体の角度を sin カーブ）| ◎ 最も簡単、AI 不要 |
| **A2. LLM にキーフレーム生成** | 「お辞儀モーション作って」→ LLM が Structured Outputs で motion JSON 出力 | ○ 実現性高い |
| **A3. モーション生成専用 AI** | 拡散モデル等 | △ Live2D 向けは未成熟、オーバースペック |

→ **A1 → A2 の段階的拡張**が現実的。第一歩はコードで基礎モーション数種、次に LLM が応答に応じて選択 → 将来的に生成。

### 可否マトリクス（桃瀬ひよりの場合）

| 動作 | 必要パラメータ | 可能性 |
|---|---|---|
| 半目（実装済み）| eye_open | ✅ |
| 頷き・首かしげ | 顔の角度 X/Z | ✅ 標準装備 |
| お辞儀 | 体の角度（BodyAngle）| ✅ おそらく可能（要確認）|
| 手を振る・手を動かす | 腕パラメータ | △ モデル依存（要確認）|
| ダンス・全身協調 | 多数の協調パラメータ | ❌ 2D 表現限界 |

### 着手の前提条件 / トリガー
- 桃瀬ひよりのパラメータ調査（腕パラメータの有無確認）
- 基礎モーション数種で UX 改善効果が見込めると判断したとき
- B はオリジナルキャラ移行を待たず**現行の桃瀬ひよりでも着手可能**

## C. 簡易モデル / マルチ Renderer（拡張の本命）

### 背景・狙い
Live2D 一本に依存しているとモデル制作コスト・権利・拡張性の3点で詰まる。
「顔の素体 + 瞬き + 口パク」レベルの簡易モデル（Inochi2D Puppet / 静止画レイヤー方式 等）を選択肢に加えることで、AI 画像生成と直結する量産パスを開く。

### 妥当性評価

**メリット（強い）**:
- **権利問題の根本対策**: 桃瀬ひより（Live2D サンプル）は規模拡大に制約（年商 1,000 万超で別途契約）。オリジナル画像 + 簡易モデルなら自分の権利で完結 → **スケール時の保険**
- **AI 画像生成との直結**: 拡張 A で「Live2D は AI 生成と相性が悪い」と結論したが、簡易モデル（1 枚絵 + 瞬き/口パクパーツ）なら**素直に接続**できる
- **キャラ多様性の量産**: Live2D は 1 体に数十時間 / 数万円。簡易モデルなら数時間レベル → 拡張 D（マルチキャラ）と直結
- **モーション議論との接続**: 拡張 B の「Live2D 限界」とは別の限界 / 別の可能性
- core 層（`CharacterDefinition`）は既に Renderer 抽象化済み（Phase 2c / #3249）

**デメリット**:
- 「生きてる感」の低下: リブトーク本体のコアコンセプトに直結 → **メインキャラ置き換えは推奨しない、サブキャラ位置づけ**
- モーション表現が更に狭まる: 顔の素体 + 瞬き + 口パクのみ
- Renderer 抽象化のリファクタコスト: web 層（`Live2DCanvas`）が Live2D ハードコード → 抽象化レイヤーが必要
- 複数モデル並走の運用負荷: メンテ対象 2 倍

### 現状のアーキテクチャ

| 層 | 抽象化 | 詳細 |
|---|---|---|
| **core**（性格・声・キャラ定義）| ✅ 抽象化済 | `CharacterDefinition` 型で Renderer 非依存 |
| **web**（描画コンポーネント）| ❌ Live2D ハードコード | `page.tsx` が `Live2DCanvas` を直接 import |

### 拡張アーキテクチャ（参考）

```
CharacterDefinition  ← 既存、抽象化済
  ├─ renderer: 'live2d' | 'puppet' | 'sprite'  ← 追加
  └─ rendererConfig: Live2DConfig | PuppetConfig | ...

libs/character-renderer/（新規、design.md に予定があったが未着手）
  ├─ types.ts: ICharacterRenderer インターフェース
  ├─ live2d/Live2DRenderer
  └─ puppet/PuppetRenderer  ← 新規追加点

services/livetalk/web/src/components/
  CharacterCanvas.tsx  ← 抽象化レイヤー
    └─ rendererType に応じて Live2DCanvas or PuppetCanvas をマウント
```

### 位置づけ
- **メインキャラ（桃瀬ひより / 将来オリジナル）は Live2D を維持**（生きてる感の核）
- **簡易モデルはサブキャラ・量産・実験用**として位置づけ
- **拡張 D（マルチキャラ）と統合して進める**（独立 Issue より統合が筋）

### 着手の前提条件 / トリガー
- 権利スケール到達（年商 1,000 万円接近 等）
- マルチキャラ展開のニーズ顕在化
- AI 画像生成パイプライン（拡張 A）との接続検討

## D. マルチキャラ展開

### 背景・狙い
桃瀬ひより 1 キャラから、複数キャラを切替・併存可能にする。

### 妥当性評価

**メリット**:
- ユーザーの嗜好・気分に応じたキャラ選択
- 「キャラごとに別人格・別記憶」で世界観の幅が広がる

**デメリット**:
- 1 ユーザー × N キャラの記憶・親密度・通知設定の管理コスト
- データモデル（PK = `USER#<gid>` / SK = `CHAR#<charId>#...`）は既にキャラ単位で分離されているが、UI / 切替体験は未設計

### 着手の前提条件 / トリガー
- 拡張 C（簡易モデル / マルチ Renderer）とセットで検討する価値が高い
- コア体験の熟成後（MVP 完了 + 一定期間の運用）

## 戦略的束（拡張同士の関係）

### 「キャラ表現基盤の拡張」3 点セット

A（画像生成）/ B（モーション）/ C（簡易モデル）は独立ではなく、**戦略的に直結する 3 つ**：

| 拡張 | Live2D との関係 | 簡易モデルとの関係 |
|---|---|---|
| A. 画像生成 | 周辺演出のみ | **キャラ自体も生成可能** |
| B. モーション生成 | パラメータ次第で可 | 顔のみで限界 |
| C. 簡易モデル / マルチ Renderer | — | 本命 |

**C は「画像生成（A）」「マルチキャラ展開（D）」「権利スケール」の 3 つを同時に解放する鍵**になる。

### 推奨される段階的展開

```
[現在] Phase 0〜5 完了（MVP コア）
   ↓ 桃瀬ひより + Live2D
[Phase 6] B（モーション拡張） — Live2D で素直に伸ばせる、コンセプト保持
   ↓
[Phase 7] C（マルチ Renderer 基盤） — メインキャラは Live2D 維持、サブ枠を解放
   ↓
[Phase 8] A（画像生成）+ D（マルチキャラ） — オリジナルキャラ移行 + 量産
   ↓
[必要時] E（UGC）/ F（STT）/ G（マネタイズ）
```

これは**現時点の構想**。実際の優先度は MVP 公開後のユーザー反応 / 自身の関心 / 権利スケールのタイミングで再評価する。

## 公開後の運用品質課題（Not Planned から移行）

MVP では実害が限定的なため Not Planned としてクローズしたが、公開フェーズの品質強化で再検討する候補。いずれも「会話による記憶訂正」の精度に関わり、相補的に効く。

### Q1. 暗黙訂正キーワード辞書の取りこぼし（旧 #3322）

**課題**: 暗黙訂正検出（`correction-detector.ts`）の 2 段判定（キーワードフィルタ → LLM 判定）で、ステップ 1 のキーワード辞書に比較・否定の言い換えが不足し、一部の訂正が LLM 判定に到達せず弾かれる。

- 例: 「実は ◯◯」→ 通過 / 「本当は ほうじ茶の方が好き」→ 弾かれる（「本当は」「〜の方が好き」が辞書にない）
- 編集 UI を撤去（ADR-007 補足）した以上、会話訂正は正規フローであり、取りこぼしは誤情報残存に直結

**対応案**:
- 案 A: 辞書に言い換え追加（「本当は」「〜の方が好き」「〜じゃなくて」「〜ではなく」「やっぱり」等）— 最小変更
- 案 B: キーワードフィルタの感度を上げる — LLM 呼び出しコスト増
- 案 C: フィルタ廃止して全件 LLM 判定 — 精度最高だがコスト大
- false positive（訂正でないのに訂正判定）にも注意

**Not Planned 判断理由**: MVP では会話訂正が大半のケースで機能しており、公開後に実トラフィックを見て辞書を育成するのが現実的。リリースブロッカーではない。

### Q2. 記憶削除時の MemorySummary 即時反映（旧 #3324）

**課題**: `/memory` UI で記憶を削除しても、② MemorySummary と ③ Messages には古い情報が残存し、Summary 再生成（日次バッチ）までの最大 24 時間、AI が削除前の情報を言及しうる。

**現状の対処（実施済み）**: 削除ダイアログに「直近会話・圧縮要約には即時反映されません」と注釈（#3308）。翌日の日次バッチで Summary 再生成 → 自然に消える。透明性は確保済み。

**対応案**:
- 案 A: 削除時に Summary へ stale フラグ → 次 chat 時に再生成（整合性◎ / 次応答が遅延）
- 案 B: 削除時に Summary 文字列から該当 content を機械除去（軽い / 言い換え表現は残存しうる）
- 案 C: 現状維持（注釈のみ、コスト 0）
- Messages は案問わず触らない（履歴改変感を避ける、design.md 議論済み）

**Not Planned 判断理由**: 注釈で透明性を確保済みで、24 時間で自然解消する。Q1（会話訂正の精度向上）と組み合わせれば窓も実質緩和される。公開フェーズの品質強化で判断。

### Q3. LLM 応答の構造化（非ストリーミング JSON 化、旧 #3355）

**課題・狙い**: 現状の NDJSON ストリーミング応答を Structured Outputs（JSON）に切り替え、emotion / motion / affectionDelta 等のメタデータを 1 応答に統合する案。

**再計測の結論（#3354 マージ後）**: TTFB は 14秒→5秒に改善したが、`llmTtfb / llmTotal` は依然 75% で TTFB 支配。token ストリーミングの「逐次表示」効果は体感的に薄いまま。構造化のメリット（メタデータ統合）は有効だが、リリースブロッカーではない。

**Not Planned 判断理由**: 構造化はモーション拡張（B）・能動演出と直結する**戦略的布石**であり、それらに着手するタイミングで一緒に再検討するのが筋。単独で今やる必要はない。本ロードマップ B（モーション拡張）の LLM 生成と接続する形で将来判断。
