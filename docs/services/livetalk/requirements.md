# リブトーク（LiveTalk）要件定義書

## 1. ビジネス要件

### 1.1 背景・目的

Live2D アバターと AI（LLM + TTS）を組み合わせたコンパニオン PWA。

既存サービス（ChatGPT Pulse、Replika、Character.AI、Open-LLM-VTuber、Grok の Ani 等）の事前調査により、AI チャット × キャラクター × 能動アクションの組み合わせ自体は既存領域であることを確認済み。本サービスの差別化は機能数や個別機能の優劣ではなく、**「キャラが同じ時間軸で生活している」体験の総合演出**に置く。

参考としたコンセプト軸：

- ヤチヨ（超かぐや姫）：AI ライバー
- エネ（カゲロウプロジェクト）：端末を行き来する電脳生命体
- リツ（暗殺教室）：スマホに出入りする AI
- デスクトップメイト：常駐キャラクターの癒し

「機能ではなく存在感」を提供する。

### 1.2 対象ユーザー

| フェーズ | 対象 |
|---|---|
| MVP（Phase 0〜2 完了時） | 開発者自身のみ |
| 限定公開（Phase 3〜5 完了時） | 招待制で数名 |
| 公開（Phase 6+） | 国内一般ユーザー（モバイル中心） |

優先プラットフォーム：モバイル PWA（iOS / Android）。

### 1.3 ビジネスゴール

- 自分用 AI コンパニオンとして実用に堪える状態を作る（Phase 2 完了時）
- 「同じ時間軸で生きてるキャラ」の体験を完成させる（Phase 5 完了時）
- 長期保持率を既存 AI コンパニオンより高くする（公開後）
- ライセンス・法的観点で破綻しない運用を最初から成立させる

数値目標は MVP 段階では設けない（PMF 検証フェーズに入った段階で設定）。

## 2. 機能要件

### 2.1 ユースケース

#### UC-001: テキストでキャラと会話する

- **概要**: ユーザーがテキストを入力し、キャラから自然な応答を受け取る
- **アクター**: ユーザー
- **前提条件**: ログイン済み、利用規約同意済み
- **正常フロー**:
    1. ユーザーがチャット画面でテキストを入力
    2. 応答テキストがストリーミングで表示開始
    3. 文単位で音声合成が始まり、口パクと共に音声が再生される
    4. 応答完了後、待機モーションに戻る
- **例外フロー**: 危険発言検出時は UC-006 に遷移

#### UC-002: キャラが Web リサーチした内容を能動的に知らせる

- **概要**: ユーザーの興味とキャラの嗜好に基づいて Web を定期巡回し、見つけた情報を通知する
- **アクター**: ユーザー、システム（バッチ）
- **前提条件**: 通知許可済み、ホーム画面追加済み（PWA）
- **正常フロー**:
    1. システムが定期バッチで Web 取得（acquire）→ Topic への集約（consolidation）を行う
    2. care の高い Topic の新規性ある WEB fact を通知候補にする
    3. ユーザーの活動時間帯（朝・夕方）に通知配信
    4. ユーザーが通知をタップしてアプリを起動
    5. キャラからの第一声で該当情報を共有

#### UC-003: キャラが「勉強しておくね」と返す

- **概要**: 真に知らないトピックを聞かれた時、キャラは即答せず後で調べて答える。既に調べてある話題（WEB fact を持つ Topic）は勉強に倒さず会話でそのまま答える
- **アクター**: ユーザー、システム（バッチ）
- **前提条件**: 会話中
- **正常フロー**:
    1. ユーザーがトピックを質問する
    2. 想起で関連 Topic を探し、WEB fact があれば会話でそのまま回答する
    3. 関連 Topic が無い（真に未知の）場合のみ「えーっと…知らない、勉強しておくね」と返し、取得（acquire）対象に予約する
    4. 次回の取得バッチが Web を取得し、集約が Topic に畳む。次回の会話 or 通知でキャラが該当情報を持ち出す
- **補足**: 知識ゲートは想起の後段に置く（既知は会話で回答・真に未知のみ勉強予約）。専用の勉強候補キューは持たず、取得対象は care と鮮度で選ぶ。詳細は architecture.md「2.12」「2.24」を参照

#### UC-004: キャラの記憶を閲覧・削除する

- **概要**: ユーザーがキャラに覚えられている「自分についての事実（SELF fact）」を閲覧・削除する
- **アクター**: ユーザー
- **前提条件**: ログイン済み
- **正常フロー**:
    1. ユーザーが記憶ページを開く
    2. SELF fact が話題（Topic）ごとに一覧表示される（Tier 別タブ・ピン留めは持たない）
    3. ユーザーが不要な項目を 1 件単位で削除する
    4. 削除は物理削除で、当該 Topic の要約を即時再生成し、削除内容が導出物にも残らないようにする
- **補足**: キャラが調べた知識（WEB fact）はこの画面に出さない・削除対象にしない。記憶の**編集機能は提供しない**。誤情報の訂正は「会話で直接伝える」か削除が正規フロー。削除が確実に応答へ反映される仕組みは architecture.md「2.24 決定的忘却」を参照

#### UC-005: キャラの生活サイクルを体験する

- **概要**: 時間帯によってキャラの応答スタイルが変わる
- **アクター**: ユーザー
- **前提条件**: なし
- **正常フロー**:
    1. ユーザーが深夜にアプリを起動して話しかける
    2. キャラが寝ぼけた口調・半目で応答する
    3. 起きてる時間帯に話しかけると通常応答に戻る

#### UC-006: 危険発言時に安全フローへ誘導される

- **概要**: 自殺・自傷を含む発言を検出した場合、専用フローでリソースを案内する
- **アクター**: ユーザー、システム（セーフティ）
- **前提条件**: なし
- **正常フロー**:
    1. ユーザーが該当発言を入力
    2. システムがキーワード検出 → LLM をバイパス
    3. キャラの口調で心配を表明 + リソース案内（いのちの電話等）
    4. 検出ログを別領域に保管

#### UC-007: 退会してデータを削除する（忘れてもらう権利）

- **概要**: ユーザーが退会し、自分の livetalk データを一括削除する
- **アクター**: ユーザー
- **前提条件**: ログイン済み
- **正常フロー**:
    1. ユーザーが退会・データ削除の導線を開く
    2. 削除範囲と不可逆である旨の確認を表示
    3. ユーザーが確認すると、`PK=USER#<googleId>` 配下の全データを即時ハード削除する
    4. セーフティ検出ログ（SafetyEvent）のみ匿名化して保持される（ADR-2.21）
- **補足**: データエクスポートはセルフサービスでは提供しない。開示請求は運用（問い合わせ対応）で対応する。詳細は architecture.md「2.21」を参照

#### UC-008: セーフティ検出を横断レビューする（運用者）

- **概要**: 運用者が全ユーザーのセーフティ検出ログを時系列で確認する
- **アクター**: 運用者（`livetalk:admin`）
- **前提条件**: `livetalk:admin` 権限保持
- **正常フロー**:
    1. 運用者が管理画面のセーフティ横断レビューを開く
    2. 全ユーザーの SafetyEvent が検出時刻の降順で一覧表示される
    3. 運用者が検出内容を確認し、辞書メンテ・対応妥当性の判断に用いる
- **補足**: requirements 3.2 の「人間レビュー可能」を実態として満たすための手段。実現方式は architecture.md「2.22」を参照

## 3. 非機能要件

### 3.1 パフォーマンス要件

| 項目 | 要件 |
|---|---|
| 応答開始までの時間 | テキスト 1〜2 秒、音声 2〜3 秒（最初の文） |
| 音声再生方式 | 文単位の逐次合成（Web Audio API ベース再生） |
| Live2D 描画 fps | 30 fps 以上（モバイル含む） |
| 同時接続数 | MVP では考慮不要（自分用） |

### 3.2 セキュリティ要件

- **認証**: Google OAuth（既存 Auth サービス経由）
- **認可**: 既存 RBAC、`livetalk:chat` permission を新設
- **データ保護**: DynamoDB 暗号化、Secrets Manager で API キー管理
- **コンテンツセーフティ**:
    - 自殺・自傷・希死念慮発言の検出と強制エスカレーション
    - OpenAI Moderation API による応答後の二段チェック
    - 検出ログの保管（別領域、人間レビュー可能）。横断レビューは sparse GSI + `livetalk:admin` 管理画面で実現する（architecture.md「2.22」）
- **法的対応**:
    - 18 歳以上同意フロー
    - 利用規約・プライバシーポリシーの提示と同意取得
    - 医療目的でない旨の明示
    - 退会・データ削除（忘れてもらう権利）の導線。開示請求は運用（手動開示）で対応する（architecture.md「2.21」）

### 3.3 可用性要件

| 項目 | 要件 |
|---|---|
| 稼働率 | MVP では SLA 規定なし（自分用） |
| RTO | 障害時は dev / prod 再デプロイで復旧 |
| RPO | DynamoDB のバックアップ機能（PITR）に準拠 |

### 3.4 保守性・拡張性要件

- **抽象化レイヤー**:
    - Live2D 描画は `CharacterDefinition` で抽象化（将来の他キャラ対応）
    - LLM は `ILLMClient` で抽象化（OpenAI / Anthropic 切替）
    - TTS は `IVoiceClient` で抽象化（VOICEVOX → 別 TTS 切替）
- **コーディング規約**: 既存プラットフォーム規約に準拠（TypeScript strict、テストカバレッジ 80% 以上）
- **共通基盤の活用**: 認証・Push 通知・CDK・CloudFront・DynamoDB は既存プラットフォームに乗る
- **将来拡張**: キャラ表現基盤（画像生成・モーション・簡易モデル）等の構想は [roadmap.md](./roadmap.md) を参照

### 3.5 その他の非機能要件

- **ライセンス遵守**:
    - Live2D Free Material License（年商 1,000 万円未満で商用 OK）
    - 桃瀬ひより 個別利用規約
    - VOICEVOX エンジン LGPL（HTTP API 経由のため影響なし）
    - VOICEVOX キャラ別ライセンス（クレジット表記必須）
- **コンプライアンス**:
    - 国内法準拠
    - 将来的に CA SB 243（AI コンパニオン規制法、2027 施行）への対応も視野

### 3.6 リージョン

us-east-1 固定。CloudFront との連携（ACM 証明書、Lambda@Edge 等が us-east-1 必須）の都合による。

## 4. ドメインオブジェクト

| エンティティ | 説明 |
|---|---|
| User | サービスを利用するユーザー（Google アカウント単位） |
| Character | キャラクター定義（運営が用意する複数キャラを切替・併存。例: 桃瀬ひより / 早瀬アゲハ。ユーザー作成キャラはスコープ外） |
| CharacterState | ユーザーごとのキャラ状態（親密度、最終起動時刻等） |
| LifecycleState | キャラの生活サイクル状態（起きてる / 寝てる、就寝時刻、ユーザー活動プロファイル） |
| Message | 会話メッセージ 1 件（ユーザー発話 / キャラ応答）。集約前の元データを兼ねる |
| Topic | 話題。知識・記憶の集約単位（SELF/WEB fact を束ねる。ADR-2.24） |
| SELF fact | 会話由来のユーザー事実（削除単位・ユーザー所有） |
| WEB fact | 勉強由来の知識（鮮度単位・出典付き。volatility を持つ） |
| Note（共有ログ） | 「調べたよ」の出来事。Topic を参照するギフト記録 |
| 集約カーソル | どこまで集約したかを指すウォーターマーク（ストリーム別） |
| care | ユーザーがその話題をどれだけ気にしているかの優先度（回数ベース） |
| Affection | 親密度の現在値 + 履歴 |
| NotificationEvent | プッシュ通知配信履歴 |
| SafetyEvent | セーフティ検出イベントログ |

Topic 中心モデルへの再設計で、旧 Memory（Tier）/ MemorySummary / InterestCategory / Knowledge / StudyTopic は撤去した（ADR-2.24）。詳細な属性・関係は [external-design.md](./external-design.md) を参照する。

## 5. スコープ外

本プロジェクトでは**意図的に実装しない**機能を明示する（将来拡張候補は [roadmap.md](./roadmap.md) を参照）：

- ❌ **音声入力（STT）**: Phase 6+ で検討、MVP は文字入力のみ
- ❌ **ユーザー作成キャラ（UGC）**: 将来拡張（運営が用意する複数キャラの切替・併存は対応済み。ADR 2.15 / 2.16 / 2.17 を参照）
- ❌ **オリジナル Live2D モデル**: 公式サンプル使用、本格公開時に検討
- ❌ **本格的な声優録音**: VOICEVOX 使用、本格公開時に検討
- ❌ **ネイティブアプリ**: PWA のみ、iOS Push 限界に直面したら検討
- ❌ **多言語対応**: 日本語のみ
- ❌ **マネタイズ機構**: 将来拡張
- ❌ **ToDo / カレンダー / メール連携**: コンパニオン体験を散漫にするため不採用
- ❌ **親密度の降下**: Replika 教訓に基づき意図的に実装しない
- ❌ **キャラからの NSFW / 性的応答**: 法的・倫理的リスク回避のため不採用
- ❌ **応答ゲートとしての睡眠状態**: 寝ていてもスタイル変調で応答する
- ❌ **記憶の編集**: 誤情報訂正は会話で行う（編集 UI は撤去済み、削除のみ提供）
- ❌ **セルフサービスのデータエクスポート（ポータビリティ）**: 開示請求は運用で対応。セルフサービス化は将来検討（roadmap）

## 6. 用語集

| 用語 | 定義 |
|---|---|
| Live2D | 2D イラストをメッシュ変形でアニメーションさせる技術。Live2D Inc. の登録商標 |
| Cubism | Live2D のモデルファイル形式・SDK |

| 冥鳴ひまり | VOICEVOX の話者キャラクター（speaker=14） |
| PWA | Progressive Web App。Web 技術で作るアプリ |
| RBAC | Role-Based Access Control。既存プラットフォームの認可方式 |
| Single Table Design | DynamoDB の単一テーブル設計パターン |
| Topic | 話題を単位とする知識・記憶の集約体（SELF fact ＋ WEB fact を束ねる） |
| SELF fact / WEB fact | Topic 配下の会話由来 / 勉強由来の事実。前者は削除単位、後者は鮮度単位 |
| consolidation（集約） | 未集約の元データを Topic に畳む毎時バッチ処理 |
| acquire（取得） | Web を取ってくる毎時バッチ処理（畳みは consolidation が行う） |
| care | 話題への注目度。自発リサーチ・通知の優先度信号（回数ベース・ユーザー起点で上昇） |
| volatility | WEB fact の揮発性（安定 / 揮発）。鮮度追随の駆動 |
| 想起 | 会話時に関連 Topic を文脈へ注入すること（埋め込み類似度の関連度 only） |
| 親密度 | ユーザーとキャラの関係性の深さを表す数値（上昇のみ） |
| 生活サイクル | キャラの時間軸（起床・就寝・勉強等）の状態モデル |
| クリティカル escalation | 通常時間帯外でも push 通知する例外条件 |
| LLM ストリーミング | LLM の応答 token を逐次返す仕組み（NDJSON ストリーム） |
| Structured Outputs | OpenAI の構造化出力（Zod スキーマで型保証された JSON 応答） |
| ADR | Architectural Decision Record。重要な設計判断の記録 |
