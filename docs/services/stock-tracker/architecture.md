# Stock Tracker アーキテクチャ設計書（設計決定記録）

## 1. システム概要

Stock Tracker は、株価のリアルタイム監視と条件ベースのアラート通知を提供するサービスです。TradingView API から株価データを取得し、ユーザーが設定した条件（価格閾値など）を満たした場合に Web Push 通知を送信します。また、保有株式の管理と日次サマリー・AI 解析により、投資判断を支援します。

Next.js アプリケーションをサーバーレスで実行し、アラートチェックと日次サマリー生成を独立したバッチ Lambda で処理する構成です。データはすべて DynamoDB のシングルテーブル設計で管理し、認証には nagiyu プラットフォーム共通の NextAuth 基盤を使用します。

---

## 2. 重要なアーキテクチャ決定

### 2.1 DynamoDB シングルテーブル設計の採用

**背景・問題**

複数エンティティ（取引所・ティッカー・保有株式・アラート・日次サマリー）を効率的に管理するデータベース設計が必要だった。

**決定**

単一の DynamoDB テーブルで全エンティティを管理するシングルテーブル設計を採用する。

**根拠・トレードオフ**

- アクセスパターンが「ユーザー中心」に集中しており、GSI 設計でほぼ全クエリを網羅できる
- nagiyu プラットフォームの既存サービスと同じパターンで一貫性がある
- RDB に比べ設計の複雑性は高いが、DynamoDB の低レイテンシ・オンデマンドスケールとの親和性を優先した
- 複数テーブル設計はクロステーブルクエリが頻発し、コスト・パフォーマンス上不利

---

### 2.2 Repository Factory パターンによるテスタビリティの確保

**背景・問題**

E2E テストで実際の DynamoDB に接続せずに動作確認したい。一方で、本番環境では DynamoDB リポジトリを使用する必要がある。

**決定**

環境変数 `USE_IN_MEMORY_REPOSITORY` により DynamoDB 実装とインメモリ実装を切り替える Repository Factory パターンを採用する。

**根拠・トレードオフ**

- テスト時は InMemory 実装でデータを分離でき、実行速度と独立性を確保できる
- API エンドポイントはリポジトリのインターフェースに依存するため、実装の変更が容易
- シングルトン管理によりリポジトリのインスタンスを効率的に使い回せる

---

### 2.3 バッチ処理の分離（アラート処理 vs 日次サマリー生成）

**背景・問題**

アラートチェック（1分・1時間・日次）と日次サマリー生成（1時間間隔）は処理目的が異なるが、同じスケジュールで実行されることがある。

**決定**

日次サマリー生成バッチを既存のアラート処理バッチ（hourly）とは独立した別 Lambda・別 EventBridge ルールとして運用する。

**根拠・トレードオフ**

- 一方の処理が失敗しても他方に影響しない障害分離が実現できる
- バッチごとにメモリ・タイムアウト設定を最適化できる
- Lambda 関数が増えるが、責務の明確化と保守性向上を優先した

---

### 2.4 Web Push 通知の採用

**背景・問題**

アラート条件達成時にユーザーへ通知する手段が必要だった。スマートフォン・デスクトップ両方で動作し、追加アプリなしで利用できる方式が求められた。

**決定**

VAPID 認証を用いた Web Push 通知を採用する。通知クリック時は `exchangeId`/`tickerId` パラメータ付きのチャート画面 URL へ遷移する。

**根拠・トレードオフ**

- ブラウザネイティブ機能であり、追加アプリが不要
- Service Worker によりブラウザを閉じた状態でも通知受信が可能
- VAPID 認証により通知送信元の検証が可能
- iOS Safari の対応が限定的であることは既知のトレードオフ

---

### 2.5 TradingView API の採用

**背景・問題**

リアルタイム株価データをブラウザとバッチ処理の両方で取得する手段が必要だった。

**決定**

`@mathieuc/tradingview` パッケージを通じて TradingView API を利用する。

**根拠・トレードオフ**

- 既存の finance リポジトリでの動作実績がある
- WebSocket 接続による効率的なリアルタイム更新（ポーリング不要）
- 複数タイムフレーム・多数の取引所・ティッカーに対応
- 非公式パッケージであり、API 仕様変更のリスクがある

---

### 2.6 AI 解析の非同期生成と障害分離

**背景・問題**

日次サマリーに AI 解析（投資判断・パターン分析など）を追加したいが、AI API の失敗がサマリー機能全体に影響しないようにしたい。

**決定**

AI 解析は日次サマリー生成フローの拡張として実行し、失敗しても日次サマリー自体の保存・表示は継続する。AI 解析の状態を「生成成功」「未生成」「生成失敗」の3状態で明示的に管理する。

**根拠・トレードオフ**

- AI API（OpenAI）の障害を局所化し、既存機能への波及を防ぐ
- ユーザー操作時の追加待機が不要（バッチ処理内で非同期処理）
- OpenAI Responses API の Structured Output を利用し、固定フォーマットで結果を返す

---

### 2.7 日足パターン分析の設計方針

**背景・問題**

株価チャートのパターン分析を提供したいが、ユーザー定義の条件式は複雑度が高い。

**決定**

初版は固定パターン（買い1種・売り1種）を採用し、判定結果を「成立・不成立・判定不能」の3状態で管理する。

**根拠・トレードオフ**

- 初版実装の複雑度を抑え、確実に動作するパターンから提供する
- 取得日足が所定本数に満たない場合は「判定不能」として成立判定を行わず、精度を担保する
- 将来的なパターン追加・カスタム条件式は Phase 2 以降で対応する

---

### 2.8 バルク取得メソッドのアイテム単位エラー耐性

**背景・問題**

DynamoDB に格納された一部アラートのサブスクリプションデータが不正な形式で残存しており、バッチがアラートを一括取得する際に個別アイテムのマッピング失敗がバッチ全体を停止させていた。旧形式から新形式へのサブスクリプション移行期に発生したデータ不整合が原因。

**決定**

repository 層のバルク取得メソッドは、個別アイテムのマッピング失敗（エンティティデータ不正エラー）をアイテム単位で吸収し、不正アイテムをスキップして有効アイテムのみを返す。スキップ時は識別情報をワーニングとしてログ出力する。

**根拠・トレードオフ**

- 一部の不正データが全件処理を停止させることを防ぎ、バッチの可用性を高める
- スキップログにより、後続の不正データ修正作業をサポートできる
- ログに個人情報（サブスクリプション endpoint など）を含めないことを原則とする
- マッピング失敗以外の DynamoDB SDK エラーは従来通りエラーとして伝播する

## 3. データモデル概要

DynamoDB シングルテーブルで以下のエンティティを管理します。

| エンティティ | 説明 | 主要な属性 |
| ------------ | ---- | ---------- |
| Exchange（取引所） | 株式市場の取引所 | ID、名前、タイムゾーン |
| Ticker（ティッカー） | 個別銘柄 | ID、シンボル、銘柄名、所属取引所 |
| Holding（保有株式） | ユーザーの保有情報 | ユーザーID、ティッカー、数量、平均取得価格 |
| Alert（アラート） | 価格条件通知設定 | ユーザーID、ティッカー、条件タイプ、目標価格、通知頻度 |
| PushSubscription | Web Push 購読情報 | ユーザーID、エンドポイント |
| DailySummary（日次サマリー） | 日次 OHLCV データと AI 解析 | ティッカー、日付、OHLCV、パターン分析、AI 解析結果 |

エンティティ関係:

- ユーザーは複数の Holding・Alert・PushSubscription を持つ
- Ticker は1つの Exchange に属する
- DailySummary は1つの Ticker に対して日付ごとに1件生成される

詳細な PK/SK・GSI 設計はコードを参照してください。
