# Stock Tracker アーキテクチャ設計書（設計決定記録）

## 1. システム概要

Stock Tracker は、株価のリアルタイム監視と条件ベースのアラート通知を提供するサービスです。TradingView API から株価データを取得し、ユーザーが設定した条件（価格閾値など）を満たした場合に Web Push 通知を送信します。また、保有株式の管理と日次サマリー・AI 解析により、投資判断を支援します。

Next.js アプリケーションをサーバーレスで実行し、アラートチェックと日次サマリー生成を独立したバッチ Lambda で処理する構成です。データはすべて DynamoDB のシングルテーブル設計で管理し、認証には nagiyu プラットフォーム共通の NextAuth 基盤を使用します。

---

## 2. 重要なアーキテクチャ決定

### 2.1 DynamoDB シングルテーブル設計の採用

**背景・問題**

複数エンティティ（取引所・ティッカー・保有株式・アラート・日次サマリー）を効率的に管理するデータベース設計が必要だった。

**決定**

単一の DynamoDB テーブルで全エンティティを管理するシングルテーブル設計を採用する。

**根拠・トレードオフ**

- アクセスパターンが「ユーザー中心」に集中しており、GSI 設計でほぼ全クエリを網羅できる
- nagiyu プラットフォームの既存サービスと同じパターンで一貫性がある
- RDB に比べ設計の複雑性は高いが、DynamoDB の低レイテンシ・オンデマンドスケールとの親和性を優先した
- 複数テーブル設計はクロステーブルクエリが頻発し、コスト・パフォーマンス上不利

---

### 2.2 Repository Factory パターンによるテスタビリティの確保

**背景・問題**

E2E テストで実際の DynamoDB に接続せずに動作確認したい。一方で、本番環境では DynamoDB リポジトリを使用する必要がある。

**決定**

環境変数 `USE_IN_MEMORY_REPOSITORY` により DynamoDB 実装とインメモリ実装を切り替える Repository Factory パターンを採用する。

**根拠・トレードオフ**

- テスト時は InMemory 実装でデータを分離でき、実行速度と独立性を確保できる
- API エンドポイントはリポジトリのインターフェースに依存するため、実装の変更が容易
- シングルトン管理によりリポジトリのインスタンスを効率的に使い回せる

---

### 2.3 バッチ処理の分離（アラート処理 vs 日次サマリー生成）

**背景・問題**

アラートチェック（1分・1時間・日次）と日次サマリー生成（1時間間隔）は処理目的が異なるが、同じスケジュールで実行されることがある。

**決定**

日次サマリー生成バッチを既存のアラート処理バッチ（hourly）とは独立した別 Lambda・別 EventBridge ルールとして運用する。

**根拠・トレードオフ**

- 一方の処理が失敗しても他方に影響しない障害分離が実現できる
- バッチごとにメモリ・タイムアウト設定を最適化できる
- Lambda 関数が増えるが、責務の明確化と保守性向上を優先した

---

### 2.4 Web Push 通知の採用

**背景・問題**

アラート条件達成時にユーザーへ通知する手段が必要だった。スマートフォン・デスクトップ両方で動作し、追加アプリなしで利用できる方式が求められた。

**決定**

VAPID 認証を用いた Web Push 通知を採用する。通知クリック時は `exchangeId`/`tickerId` パラメータ付きのチャート画面 URL へ遷移する。

**根拠・トレードオフ**

- ブラウザネイティブ機能であり、追加アプリが不要
- Service Worker によりブラウザを閉じた状態でも通知受信が可能
- VAPID 認証により通知送信元の検証が可能
- iOS Safari の対応が限定的であることは既知のトレードオフ

---

### 2.5 TradingView API の採用

**背景・問題**

リアルタイム株価データをブラウザとバッチ処理の両方で取得する手段が必要だった。

**決定**

`@mathieuc/tradingview` パッケージを通じて TradingView API を利用する。

**根拠・トレードオフ**

- 既存の finance リポジトリでの動作実績がある
- WebSocket 接続による効率的なリアルタイム更新（ポーリング不要）
- 複数タイムフレーム・多数の取引所・ティッカーに対応
- 非公式パッケージであり、API 仕様変更のリスクがある

---

### 2.6 AI 解析の非同期生成と障害分離

**背景・問題**

日次サマリーに AI 解析（投資判断・パターン分析など）を追加したいが、AI API の失敗がサマリー機能全体に影響しないようにしたい。

**決定**

AI 解析は日次サマリー生成フローの拡張として実行し、失敗しても日次サマリー自体の保存・表示は継続する。AI 解析の状態を「生成成功」「未生成」「生成失敗」の3状態で明示的に管理する。

**根拠・トレードオフ**

- AI API（OpenAI）の障害を局所化し、既存機能への波及を防ぐ
- ユーザー操作時の追加待機が不要（バッチ処理内で非同期処理）
- OpenAI Responses API の Structured Output を利用し、固定フォーマットで結果を返す

---

### 2.7 日足パターン分析の設計方針

**背景・問題**

株価チャートのパターン分析を提供したいが、ユーザー定義の条件式は複雑度が高い。

**決定**

初版は固定パターン（買い1種・売り1種）を採用し、判定結果を「成立・不成立・判定不能」の3状態で管理する。

**根拠・トレードオフ**

- 初版実装の複雑度を抑え、確実に動作するパターンから提供する
- 取得日足が所定本数に満たない場合は「判定不能」として成立判定を行わず、精度を担保する
- 将来的なパターン追加・カスタム条件式は Phase 2 以降で対応する

---

### 2.8 バルク取得メソッドのアイテム単位エラー耐性

**背景・問題**

DynamoDB に格納された一部アラートのサブスクリプションデータが不正な形式で残存しており、バッチがアラートを一括取得する際に個別アイテムのマッピング失敗がバッチ全体を停止させていた。旧形式から新形式へのサブスクリプション移行期に発生したデータ不整合が原因。

**決定**

repository 層のバルク取得メソッドは、個別アイテムのマッピング失敗（エンティティデータ不正エラー）をアイテム単位で吸収し、不正アイテムをスキップして有効アイテムのみを返す。スキップ時は識別情報をワーニングとしてログ出力する。

**根拠・トレードオフ**

- 一部の不正データが全件処理を停止させることを防ぎ、バッチの可用性を高める
- スキップログにより、後続の不正データ修正作業をサポートできる
- ログに個人情報（サブスクリプション endpoint など）を含めないことを原則とする
- マッピング失敗以外の DynamoDB SDK エラーは従来通りエラーとして伝播する

---

### 2.9 ダイアログコンポーネントのトップ画面への流用

**背景・問題**

トップ画面（SCR-001）に保有株式・アラート・サマリーの CRUD 操作を追加するにあたり、各専用画面（SCR-002, SCR-003, SCR-004）で既に利用しているダイアログコンポーネントの再利用が求められた。

**決定**

各専用画面のダイアログコンポーネントをトップ画面からも流用する。

**根拠・トレードオフ**

- コンポーネントの重複実装を避け、専用画面とトップ画面で同一の操作 UI を提供できる
- ダイアログの仕様変更は一箇所の修正で全画面に反映される
- 既存コンポーネントの props インターフェースをそのまま利用するため、既存設計を変えずに再利用できる

---

### 2.10 InvalidEntityDataError 判定のモジュール差異耐性

**背景・問題**

一時アラート失効バッチで、DynamoDB の不正レコード（`SubscriptionEndpoint` 欠損など）をスキップする実装があるにもかかわらず、実行環境差異により `instanceof InvalidEntityDataError` が一致せず、バッチ全体が失敗する事象が発生した。

**決定**

`DynamoDBAlertRepository` の不正データ判定は `instanceof` のみではなく、`error.name === 'InvalidEntityDataError'` とメッセージ内容も併用して判定する。

**根拠・トレードオフ**

- モジュール解決経路が異なる実行環境でも、不正データを安定してスキップできる
- 既存の `instanceof` 判定を維持しつつ後方互換で拡張できる
- `name` 判定のみの過検知を抑えるため、メッセージ先頭（`エンティティデータが無効です`）も併用する
- DynamoDB SDK エラーなど他種別エラーは従来どおり伝播させ、障害検知性を維持する

---

### 2.11 一時アラート失効バッチの subscription 非依存化と「無効化 + TTL」方式

**背景・問題**

ADR 2.8 / 2.10 でアイテム単位エラースキップ・モジュール差異耐性を入れてもなお、一時アラート失効バッチが `subscription` フィールドの整合性に依存して失敗するケースが残っていた。原因は、削除判定そのものに本来不要な `subscription` データを読み込んでいる構造にある。さらに、物理削除を伴うバッチは失敗時の冪等性が低く、リトライ運用が難しかった。

**決定**

一時アラート失効バッチを、対症療法ではなく構造的に組み替える。

1. **バッチ用の軽量取得経路を独立**: `AlertRepository.getTemporaryCandidatesByFrequency` を新設し、DynamoDB Query の `ProjectionExpression` で失効判定に必要な属性（`AlertID` / `UserID` / `ExchangeID` / `Frequency` / `Enabled` / `Temporary` / `TemporaryExpireDate`）のみを取得する。`subscription` は読み込まない。`FilterExpression` で `Temporary = true AND Enabled = true` のアラートのみに絞る。
2. **物理削除 → 「無効化 + DynamoDB TTL」**: `AlertRepository.markTemporaryAsExpired(userId, alertId, ttlSeconds)` を新設。`Enabled = false` への更新と `TTL` 属性のセットを 1 回の `UpdateItem` で行い、物理削除は DynamoDB TTL に委ねる（猶予 7 日）。DynamoDB テーブル定義は既に `timeToLiveAttribute: 'TTL'` で TTL 有効化済み。
3. **Web 一覧 API の `Enabled` フィルタ**: `GET /api/alerts` のリポジトリ呼び出しに `enabledOnly: true` を渡し、無効化された一時アラートを Web 一覧から即時非表示にする。これにより、TTL 発火を待たずユーザー視点では「期限切れ＝即削除」と同等の UX を維持する。

**根拠・トレードオフ**

- 削除判定に subscription を一切触らない構造により、subscription データの不整合がバッチをブロックしない（ADR 2.8 / 2.10 の補完ではなく、その必要性自体を縮減する）
- バッチが「フラグ更新 + TTL 設定」だけになるため冪等で、リトライしても安全
- TTL 発火は最大 48 時間遅延し得るが、Web 一覧の `Enabled` フィルタにより UX 影響は無い
- バッチ専用の軽量経路を独立させたことで、通知送信側で利用する `getByFrequency` / `mapper.toEntity` の挙動には影響を与えない
- `Subscription` を `Alert` に埋め込んでいる現データモデル自体の見直し（独立エンティティ化、新旧形式撲滅）は別途中期対応として計画する

### 2.12 採点結果を独立エンティティ化せず DailySummary に統合した判断

**背景・問題**

AI 予測採点機能（Phase 1）で採点結果（Hit/Miss・実績リターン・採点閾値等）を保存するにあたり、(a) 独立エンティティとして別レコードに保存 / (b) 既存 `DailySummaryEntity` に optional フィールドとして同居、の選択肢があった。

**決定**

採点結果は独立エンティティ化せず、既存 `DailySummaryEntity` に Evaluation\* フィールド（`EvaluationDate`、`EvaluationClose`、`ActualReturn`、`Hit`、`EvaluationThresholdPercent`、`EvaluatedAt`）として統合する。

**根拠・トレードオフ**

- 紐づけキー（TickerID + Date）が `DailySummary` と完全一致するため、分離することで得られる利点がほぼない
- 既存 `AiAnalysisResult` が「後から付与される派生属性」として同レコードに存在するパターンと一致している
- 採点結果と銘柄・取引所・日付・AI 解析結果のコンテキストを 1 read で取得でき、集計クエリの効率が高い
- 独立エンティティにした場合、採点集計クエリで 2 テーブル join が必要になるコストを回避できる

### 2.13 採点バッチを既存バッチに相乗りせず独立 Lambda にした判断

**背景・問題**

採点バッチを実装するにあたり、既存の日次サマリー生成バッチ（`summary Lambda`）に処理を追加する案と、新規独立 Lambda として切り出す案を比較した。

**決定**

採点バッチは既存バッチに相乗りせず、独立した Lambda（`batch-evaluation`）と EventBridge cron（1 時間毎）として運用する。

**根拠・トレードオフ**

- 日次サマリー生成バッチは「当日 OHLCV 取得 → 保存」を担う。採点バッチは「翌営業日終値の確認 → 採点」であり、タイミングも責務も異なる
- 一方が失敗しても他方に影響しない障害分離が実現できる
- Lambda ごとにメモリ・タイムアウト設定を最適化できる（2.3 と同じ判断パターン）
- Lambda 関数が増えるが、責務の明確化と保守性向上を優先した

---

### 2.14 集計用 GSI を新設せず既存 GSI4（ExchangeID × Date）を流用した判断

**背景・問題**

採点バッチで「未採点 & 翌営業日引け済の予測を効率的に抽出」するためのクエリ設計において、新規 GSI の追加と既存 GSI の流用を比較した。

**決定**

新規 GSI を追加せず、既存 GSI4（ExchangeID × Date）を流用して未採点予測を抽出する。

**根拠・トレードオフ**

- GSI を新設すると DynamoDB への書き込みコストが上がり、バックフィル処理も必要になる
- GSI4 は「取引所ごとに日付順で DailySummary を取得する」クエリに使われており、採点バッチが必要とする「取引所の引け日時を考慮した絞り込み」と親和性が高い
- Phase 1 の着手コストを最小化し、実運用で問題が出た場合に後から GSI を追加する判断を留保できる

---

### 2.15 UI 先行 PoC 方式を採用した判断

**背景・問題**

AI 予測精度ダッシュボード（Phase 1）の設計において、(a) 要件→設計→実装の従来型フロー / (b) モックデータを使った UI PoC を先行して実物を見てから要件を確定するフロー、の選択肢があった。

**決定**

UI 先行 PoC 方式を採用し、モックデータで動く UI を最初に作ってから要件・設計を再確定する（作業 1 → 作業 2 のループを設計に組み込んだ）。

**根拠・トレードオフ**

- 精度ダッシュボードのように「実物を見るまで指標の取捨選択や見せ方の微調整が判断しきれない」ケースでは、PoC フィードバックを得てから実装する方がやり直しコストが低い
- 実際に PoC フィードバックを経て、KPI カード廃止・銘柄別/取引所別の UI 削除・NEUTRAL 比率の扱い変更等の設計変更が行われた（`external-design.md` ADR-004〜ADR-006 参照）
- 追加のコミュニケーションラウンドが 1 回発生するが、後工程の手戻りを大幅に削減できる

### 2.16 祝日カレンダー対応を Phase 1 で実施しない判断

**背景・問題**

AI 予測採点機能（Phase 1）で「翌営業日終値」を取得するにあたり、(a) 祝日カレンダーを導入して厳密に翌取引日を判定する / (b) 既存 Stock Tracker と同じく土日のみスキップ（祝日は無視）で判定する、の選択肢があった。GW のような連休クラスタでは「翌平日が祝日」となり、TradingView API に終値が存在しないため未採点のまま retry が継続するという課題が dev 環境で観測されていた。

**決定**

採点バッチでも `getNextWeekday` をそのまま流用し、**土日のみスキップ・祝日は考慮しない** を Phase 1 の仕様として確定する。祝日カレンダー対応は Phase 2 以降のスコープとする。

**根拠・トレードオフ**

- Stock Tracker 全体（`getLastTradingDate`、`isTradingHours`、`calculateTemporaryExpireDate` 等）が既に「平日 = 取引日」前提で統一されており、採点バッチだけ祝日対応を入れると **上流の日次サマリー生成バッチが祝日にも DailySummary を作っている**（= 採点対象として存在する）状況との不整合が広がる
- 祝日マスターを導入する場合、JP / US 双方の祝日カレンダーが必要で、毎年更新の運用負担も生じる。Phase 1 の最小スコープを超える
- 祝日にあたる未採点予測は ADR 2.17 のリトライ上限で自動卒業させるため、永続的なゴミは発生しない
- 将来全体一括で祝日対応を入れる際に、採点バッチだけ先行対応していると逆に整合性のリファクタが必要になるため、足並みを揃える方が安全

### 2.17 採点リトライ上限を N 営業日とした判断

**背景・問題**

ADR 2.16 の通り Phase 1 では祝日を考慮しないため、祝日にあたる翌平日には TradingView 終値が存在しない予測が発生する。`findPendingEvaluations` の走査ウィンドウ（`windowDays = 30`）内に残る限り、これらの予測は毎 cron で TradingView を叩き続ける（GW クラスタで約 10 件 × 24 cron/day × 30 日 ≒ **約 7,200 回 / クラスタ** の無駄リクエスト）。TradingView API は非公式かつ rate limit が明示されていないため、無駄打ちはサービス継続性リスクとなる。

**決定**

`isCandidate` 判定に「予測日からの経過営業日数（土日除く）が `MAX_EVALUATION_BUSINESS_DAYS` を超えていれば候補から外す」ロジックを追加し、上限値を **5 営業日** に設定する。

**根拠・トレードオフ**

- 5 営業日は JP の GW・年末年始・盆休みなど主要連休クラスタ（最大連続休業日数）をほぼ吸収できる長さで、正常時の採点には影響しない
- 上限超過後の予測は採点候補から外れるだけで、DB 上は `EvaluatedAt = undefined` のまま残る（履歴を消さない / 卒業フラグを立てない）。後で再評価したくなった場合の選択肢を残しつつ、現時点では集計クエリで `EvaluatedAt` の有無で十分にフィルタできる
- dev 環境で本判断を反映後、未採点候補が 10 件 → 1 件 に減少することを確認済み（残 1 件も 1〜2 営業日で自動卒業する想定）

## 3. データモデル概要

DynamoDB シングルテーブルで以下のエンティティを管理します。

| エンティティ                 | 説明                        | 主要な属性                                             |
| ---------------------------- | --------------------------- | ------------------------------------------------------ |
| Exchange（取引所）           | 株式市場の取引所            | ID、名前、タイムゾーン                                 |
| Ticker（ティッカー）         | 個別銘柄                    | ID、シンボル、銘柄名、所属取引所                       |
| Holding（保有株式）          | ユーザーの保有情報          | ユーザーID、ティッカー、数量、平均取得価格             |
| Alert（アラート）            | 価格条件通知設定            | ユーザーID、ティッカー、条件タイプ、目標価格、通知頻度 |
| PushSubscription             | Web Push 購読情報           | ユーザーID、エンドポイント                             |
| DailySummary（日次サマリー） | 日次 OHLCV データと AI 解析・採点結果 | ティッカー、日付、OHLCV、パターン分析、AI 解析結果、採点結果（Evaluation\* フィールド） |

エンティティ関係:

- ユーザーは複数の Holding・Alert・PushSubscription を持つ
- Ticker は1つの Exchange に属する
- DailySummary は1つの Ticker に対して日付ごとに1件生成される

詳細な PK/SK・GSI 設計はコードを参照してください。
