# Stock Tracker 要件定義書

## 1. ビジネス要件

### 1.1 背景・目的

**背景**:
- 株式投資において、タイムリーな価格情報と適切なタイミングでの売買判断が重要
- 既存の finance リポジトリで提供していた株価追跡・通知機能を、本プラットフォームで刷新
- モノレポ構成による開発効率向上、共通インフラ活用によるコスト削減を目指す

**目的**:
- リアルタイムな株価データの可視化（チャート表示）
- 条件ベースのアラート・通知による売買タイミングの最適化
- 保有株式の管理と目標価格の算出による投資判断支援

### 1.2 対象ユーザー

**プライマリーユーザー**:
- 個人投資家（株式トレーダー）
- 日々の株価変動をチェックし、売買判断を行うユーザー

**ユーザー特性**:
- 複数の銘柄を監視している
- 価格条件に基づいた通知を受け取りたい
- 保有株式の平均取得価格と目標価格を把握したい
- モバイル/デスクトップの両方から利用する

### 1.3 ビジネスゴール

**実装済み機能**:
1. **株価の可視化**: リアルタイムな株価データをチャートで表示
2. **通知機能**: 価格条件に基づくアラート通知
3. **保有株式管理**: 平均取得価格、目標価格の算出

**将来の拡張候補**:
- テクニカル指標の追加（移動平均、ボリンジャーバンドなど）
- ポートフォリオ分析機能

## 2. 機能要件

### 2.1 ユースケース

#### UC-001: 株価チャート表示

**概要**: ユーザーが指定した取引所・ティッカーの株価チャートを表示する

**アクター**: ログインユーザー

**前提条件**:
- ユーザーがログイン済み
- 取引所・ティッカーが登録されている

**正常フロー**:
1. ユーザーがトップ画面にアクセス
2. 取引所を選択
3. ティッカーを選択
4. 時間枠を選択（5分足、1時間足、日足など）
5. システムが TradingView API から株価データを取得
6. ECharts でインタラクティブなチャートを表示

**URL パラメータによる初期選択**:
- URL クエリパラメータ `exchangeId` および `tickerId` を指定すると、画面ロード時に対応する取引所・ティッカーが自動選択された状態で表示される
- Web Push 通知クリック時はこの URL を使用し、通知元アラートの取引所・ティッカーが選択済みの状態でチャートを表示する

#### UC-002: アラート設定

**概要**: ユーザーが株価の条件を設定し、条件達成時に通知を受け取る。アラートの作成・修正・削除は、アラート一覧画面（SCR-003）に加えて、トップ画面（SCR-001）のアラートパネルからも行える。

**アクター**: ログインユーザー

**前提条件**:
- ユーザーがログイン済み
- ブラウザで通知許可が有効

**正常フロー**:
1. ユーザーがアラート設定画面にアクセス
2. 取引所とティッカーを選択
3. 条件タイプを選択（価格が指定値を上回る/下回る）
4. 目標価格を入力（入力に応じてチャートオーバーレイが即時更新される）
5. 通知頻度を選択
6. 通知設定ダイアログを開き、通知タイトル・通知本文を入力する
    - デフォルトは入力内容をもとに自動生成された値が設定される
    - 条件や価格を変更した場合は本文の上書き確認ダイアログが表示される
7. システムがアラート設定を保存

#### UC-003: 保有株式管理

**概要**: ユーザーが保有株式を登録し、平均取得価格と目標価格を確認する。保有株式の作成・修正・削除は、保有株式管理画面（SCR-002）に加えて、トップ画面（SCR-001）の保有株式パネルからも行える。

**アクター**: ログインユーザー

**正常フロー**:
1. ユーザーが保有株式管理画面にアクセス
2. 新規登録ボタンをクリック
3. 取引所、ティッカー、保有数、平均取得価格を入力
4. システムが保有株式情報を保存
5. 目標価格を自動算出して表示

#### UC-004: 日次サマリー閲覧

**概要**: ユーザーが各取引所に登録されたティッカーの直近の日次サマリーを一覧形式で確認する

**アクター**: ログインユーザー（`stocks:read` 権限保持者）

**前提条件**:
- ユーザーがログイン済み
- 取引所・ティッカーが登録されている

**正常フロー**:
1. ユーザーがサマリー一覧画面にアクセス
2. システムが各取引所の最新保存済みサマリーおよびユーザーのアラート情報を取得する（当日分が未生成の場合は直近保存済み日付のデータを表示し、空表示にしない）
3. 取引所ごとにグループ化されたティッカー一覧と、各ティッカーの投資判断に関する情報が表示される
4. サマリーデータが存在しない取引所は空状態メッセージを表示する

#### UC-005: 日次サマリー自動生成

**概要**: システムが 1 時間間隔の専用バッチを実行し、取引時間が終了した取引所の全ティッカーについて日次サマリーを自動生成して保存する

**アクター**: システム（EventBridge Scheduler）

**正常フロー**:
1. バッチが 1 時間間隔で実行される
2. 全取引所に対して取引時間終了を判定する（取引所のタイムゾーンを考慮）
3. 取引時間が終了した取引所について、取引所に紐づく全ティッカーの当日 OHLC データを TradingView API から取得する
4. 取得した OHLC データを日次サマリーとして DynamoDB に保存する（同一ティッカー・同一日付は上書き更新で冪等性を保証）
5. エラーが発生した場合はログに記録し、当該ティッカー・取引所の処理をスキップして他の処理を継続する

**スコープ**:
- サマリーデータ項目は OHLCV（始値・高値・安値・終値・出来高）とする
- 本バッチはアラート用の既存 1 時間バッチ（`hourly.ts`）とは独立した別バッチとして運用する

#### UC-006: 日次サマリーの AI 解析閲覧

**概要**: ユーザーがサマリー詳細で、価格推移・パターン分析・関連市場情報をまとめた AI 解析を閲覧する。AI 解析生成時は Web 検索を必ず実施し、決算発表・重要経済指標など直近のニュースを根拠として含める。

**アクター**: ログインユーザー（`stocks:read` 権限保持者）

**前提条件**:
- ユーザーがログイン済み
- 日次サマリーが生成済み

**正常フロー**:
1. システムが日次サマリー生成時に AI 解析を作成する
2. ユーザーがサマリー画面の詳細ダイアログを開く
3. システムが保存済みの AI 解析を表示する

**例外フロー**:
- AI 解析が未生成の場合は「AI 解析はまだ生成されていません」を表示する
- AI 解析生成失敗が記録されている場合は「AI 解析の取得に失敗しました」を表示する

#### UC-007: アラート絞り込み

**概要**: ユーザーがアラート一覧画面で取引所またはモードを指定してアラートを絞り込む

**アクター**: ログインユーザー

**前提条件**:
- ユーザーがログイン済み
- アラートが1件以上登録されている

**正常フロー**:
1. ユーザーがアラート一覧画面にアクセス
2. 取引所フィルタまたはモードフィルタを選択する
3. システムが選択条件に一致するアラートのみを一覧に表示する
4. ユーザーがクリアボタンを押すと全件表示に戻る

**スコープ**:
- フィルタリングはクライアントサイドで処理する
- 取引所・モードフィルタは同時に適用できる
- フィルタ状態は URL パラメータには反映しない（ページリロード時にフィルタ状態は保持されない。URL 同期が必要な場合は後続 Issue で対応する）

#### UC-008: チャート自動更新

**概要**: ユーザーがチャート画面で自動更新を有効にすると、10秒ごとにチャートデータが再取得・再描画される

**アクター**: ログインユーザー

**前提条件**:
- ユーザーがログイン済み
- 取引所・ティッカーが選択済み

**正常フロー**:
1. ユーザーがチャート表示エリアの自動更新ボタンをオンにする
2. システムが 10 秒ごとにチャートデータを再取得・再描画する
3. ユーザーが自動更新ボタンをオフにすると、自動更新が停止する

**例外フロー**:
- ティッカーが未選択の場合、自動更新を開始しない
- 取引所またはティッカーを変更しても、自動更新の有効/無効状態を維持する

**スコープ**:
- 自動更新のデフォルト値はオフ
- 自動更新中はボタンの状態変化でユーザーにフィードバックする
- コンポーネントのアンマウント時にタイマーをクリアし、メモリリークを防ぐ

#### UC-011: 予測の自動採点

**概要**: 既存 AI 解析バッチが生成した予測を、翌営業日終値が確定した時点で自動的に採点する

**アクター**: システム（採点バッチ Lambda）

**前提条件**:
- 対象予測の `AiAnalysisResult` が存在し、`AiAnalysisError` を持たない
- 対象予測の取引所の翌営業日が引け済み

**正常フロー**:
1. EventBridge cron（1 時間毎）が採点バッチ Lambda をトリガー
2. バッチが「未採点 & 翌営業日引け済」の予測を抽出
3. 各予測について、当日終値と翌営業日終値を TradingView API から取得
4. リターン値と Hit/Miss を判定し、対象 `DailySummaryEntity` の Evaluation\* フィールド（`EvaluationDate`、`EvaluationClose`、`ActualReturn`、`Hit`、`EvaluationThresholdPercent`、`EvaluatedAt`）を条件付き UpdateItem で書き込む

**代替フロー**: 翌営業日が週末（土日）の場合、その次の平日の終値で判定する。祝日カレンダーは Phase 1 では考慮しない（Stock Tracker 全体が土日のみスキップで統一されているため。判断の詳細は `architecture.md` § 2.16 を参照）

**例外フロー**:
- TradingView API 呼び出しが失敗した場合：当該予測は未採点のまま次回再試行（採点リトライ上限の挙動は `architecture.md` § 2.17 を参照）
- 対象予測が `AiAnalysisError` を持つ場合：採点対象から除外
- 翌平日が祝日にあたり TradingView API に終値が存在しない場合：採点リトライ上限（予測日から N 営業日）まで未採点のまま再試行し、上限超過後は採点候補から外れる

#### UC-012: 予測精度ダッシュボード閲覧

**概要**: 利用者がダッシュボードで AI 予測の精度を期間別に確認する

**アクター**: 認証済みユーザー（`stocks:read-evaluation` 権限保持者）

**前提条件**: 既存認証フローでログイン済み

**正常フロー**:
1. ユーザーがダッシュボード画面を開く
2. デフォルトで「直近 30 日」の集計を表示
3. ページ見出し直下の主要指標テキスト（期間ラベル + 方向精度 + 採点件数 + 総合精度）が表示される
4. 期間（7/30/90 日/全期間）を切り替えると主要指標テキスト・日次推移グラフ・シグナル別精度が再集計される

**代替フロー**: 採点済み予測がない場合、主要指標テキストが「期間ラベル: 採点済みの予測がありません」になる

**例外フロー**: API エラー時は ErrorAlert を表示（再読み込みはブラウザのリロードで対応）

#### UC-010: 一時アラート失効処理

**概要**: システムが毎時バッチを実行し、取引終了後に期限切れとなった一時アラートを自動削除する

**アクター**: システム（EventBridge Scheduler）

**前提条件**:
- DynamoDB に一時アラートが存在する

**正常フロー**:
1. バッチが毎時起動し、通知頻度ごとに全件の一時アラートを取得する（ページネーション対応）
2. 各アラートを評価し、期限切れ・取引時間外かを確認する
3. 条件を満たすアラートを DynamoDB から削除する
4. 処理統計をログ出力して正常終了する

**代替フロー**: 対象アラートがなければスキップして正常終了する

**例外フロー**:
- マッピング不能な不正データアイテムを検出した場合、該当アイテムの識別情報（AlertID・UserID）をワーニングとしてログ出力し、処理をスキップして後続アラートの処理を継続する
- 個別アラートの削除に失敗した場合はログ出力してスキップし、後続アラートの処理を継続する

**スコープ**:
- 不正なサブスクリプションデータを持つアイテムが存在しても、バッチ全体は停止しない
- ログに個人情報（サブスクリプションの endpoint 等）を含めない

#### UC-009: チャート画面統合表示

**概要**: ユーザーが取引所・ティッカーを選択すると、チャートに加えてサマリー・保有株式・アラート情報をチャート画面上に表示する。サマリーパネルには投資判断・買いシグナル・売りシグナル・AI 判定・更新時間・サポートレベル・レジスタンスレベルを表示し、詳細ダイアログボタンから AI 解析詳細を参照できる。保有株式パネルおよびアラートパネルからは、トップ画面上で直接 CRUD 操作が可能。

**アクター**: ログインユーザー

**前提条件**:
- ユーザーがログイン済み
- 取引所・ティッカーが選択済み

**正常フロー**:
1. ユーザーが取引所とティッカーを選択する
2. システムがサマリー・保有株式・アラート情報を並行取得する
3. サマリーパネルに投資判断・買いシグナル・売りシグナル・AI 判定・更新時間・サポートレベル・レジスタンスレベルを表示する。詳細ダイアログボタンをクリックすると詳細ダイアログが開く
4. 保有株式情報およびアラート一覧をチャート画面に表示する
5. 保有平均価格ラインとアラートしきい値ラインをチャートにオーバーレイ表示する
6. 保有株式パネル・アラートパネルの追加・編集・削除ボタンから CRUD 操作を行うと、操作完了後にパネルの表示が更新される

**例外フロー**:
- 取引所またはティッカーが未選択の場合、統合情報エリアを表示しない
- 保有なしの場合は「保有なし」を表示し、保有価格ラインは表示しない
- データ取得中はローディング表示し、エラー時はエラーメッセージを表示する

### 2.3 想定画面の概要

- トップ画面（チャート表示） - `/`：取引所・ティッカー選択、株価チャート・サマリー・保有株式・アラート情報の統合表示
- 保有株式管理画面 - `/holdings`：保有株式の登録・編集・削除
- アラート一覧画面 - `/alerts`：アラートの一覧・作成・編集・削除・絞り込み
- 取引所管理画面 - `/exchanges`（stock-admin のみ）：取引所マスタの管理
- ティッカー管理画面 - `/tickers`（stock-admin のみ）：ティッカーマスタの管理
- サマリー一覧画面 - `/summaries`：取引所ごとの日次サマリー一覧・投資判断表示
- 予測精度ダッシュボード - `/prediction-evaluation`（`stocks:read-evaluation` 権限必須）：AI 予測の採点結果を期間別に可視化

### 2.4 採点ルール仕様（Phase 1）

AI 予測（BULLISH / NEUTRAL / BEARISH）の Hit/Miss 判定基準を定義する。

#### 判定基準

| 予測シグナル | 成功条件 |
|-------------|---------|
| BULLISH | 翌営業日終値リターン ≥ +0.5% |
| BEARISH | 翌営業日終値リターン ≤ -0.5% |
| NEUTRAL | 翌営業日終値リターンが -0.5% より大きく +0.5% より小さい |

- リターン基準点：当日終値（close-to-close）
- 保有期間：1 営業日

#### 採点対象から除外するケース

- 既存 `DailySummaryEntity.AiAnalysisError` が存在する予測
- 翌営業日終値が取得できない予測（上場廃止・データ欠損など。次回バッチで再試行）

#### 集計指標

Phase 1 の UI に出す指標：

| 指標名 | 定義 | UI での出し方 |
|--------|------|---------|
| 総合精度 | 全予測（BULLISH + NEUTRAL + BEARISH）における Hit 率 | 主要指標テキストの括弧内 |
| 方向精度 | BULLISH + BEARISH のみにおける Hit 率（NEUTRAL を除外） | 主要指標テキスト本体・日次推移グラフ |
| 判定済み件数 | 採点が完了している予測の件数 | 主要指標テキストの括弧内・日次推移グラフ右軸 |
| NEUTRAL 比率 | 全判定済み件数に対する NEUTRAL 予測の件数の割合 (%) | 主要指標テキストの括弧内 |
| シグナル別ヒット率 | BULLISH / NEUTRAL / BEARISH ごとの Hit 率と件数 | シグナル別棒グラフ + 数値テーブル |
| 市場ベースレート（ベースライン） | 全件を母数とした実績の上昇率・下落率・フラット率 (%)。閾値 t で分類：upRate = ActualReturn >= +t の割合、downRate = ActualReturn <= -t の割合、flatRate = それ以外 | シグナル別数値テーブルの「ベースライン」列（BULLISH→upRate、BEARISH→downRate、NEUTRAL→flatRate） |
| エッジ（予測力） | シグナル別精度 − 対応するベースライン (%)。正ならばランダムより優れた予測力があることを示す | シグナル別数値テーブルの「エッジ」列（符号付き表示、例: +7.1pt） |

## 3. 非機能要件

### 3.1 パフォーマンス要件

| 項目                   | 要件                                    |
| ---------------------- | --------------------------------------- |
| チャート表示           | 初回ロード 2秒以内                      |
| 株価データ取得         | 1秒以内                                 |
| 通知遅延               | 条件達成から 1分以内                    |
| 同時接続数             | 想定 100ユーザー                       |
| アラート設定入力       | 通知テキスト入力時にアラート設定モーダルの不要な再描画が発生しないこと（通知編集は別ダイアログで独立して管理する） |

### 3.2 セキュリティ要件

**認証・認可**:
- Auth サービス（プラットフォーム共通）による JWT 認証
- ロールベースアクセス制御（RBAC）

**Stock Tracker 用のロール定義**:

| ロール名       | 権限（Permissions）                                                                   | 想定ユーザー                |
| -------------- | ------------------------------------------------------------------------------------- | --------------------------- |
| `stock-viewer` | `stocks:read`                                                                         | ゲストユーザー（デフォルト） |
| `stock-user`   | `stocks:read`, `stocks:write-own`                                                     | 一般ユーザー                 |
| `stock-admin`  | `stocks:read`, `stocks:write-own`, `stocks:manage-data`, `stocks:read-evaluation`     | 管理者                       |

**Permission 定義**:

| Permission                  | 説明                                                                 |
| --------------------------- | -------------------------------------------------------------------- |
| `stocks:read`               | チャート閲覧、取引所一覧取得、ティッカー一覧取得                       |
| `stocks:write-own`          | 自分のアラート設定・保有株式管理                                      |
| `stocks:manage-data`        | 取引所・ティッカーの CRUD（マスタデータ管理）                         |
| `stocks:read-evaluation`    | AI 予測精度ダッシュボードの閲覧・精度集計 API の呼び出し（Phase 1 では `stock-admin` ロールにのみ付与） |

**データ保護**:
- HTTPS 通信の強制
- DynamoDB のデータは保存時暗号化
- Web Push サブスクリプション情報の暗号化

**セキュリティ対策**:
- XSS 対策: React の自動エスケープ、CSP ヘッダー
- CSRF 対策: SameSite Cookie
- 入力バリデーション: クライアント・サーバー両方で実施
- 機密情報管理: OpenAI API キーは Secrets Manager で管理し、実行環境へ安全に注入する

### 3.3 可用性要件

| 項目               | 要件                           |
| ------------------ | ------------------------------ |
| 稼働率             | 99.5%（ベストエフォート）      |
| RTO (復旧目標時間) | 4時間以内                      |
| RPO (復旧目標時点) | 1時間以内（DynamoDB PITR有効） |

**バックアップ戦略**:
- DynamoDB Point-in-Time Recovery (PITR) 有効化
- 保持期間: 35日間

### 3.4 保守性・拡張性要件

**コード品質**:
- TypeScript strict mode 有効
- ESLint による静的解析
- テストカバレッジ 80%以上（ビジネスロジック）

**モジュール化**:
- core パッケージ: ビジネスロジック（フレームワーク非依存）
- web パッケージ: Next.js UI
- batch パッケージ: Lambda 関数（通知バッチ）

**拡張ポイント**:
- 新しい条件タイプの追加（パターン認識など）
- 新しいチャートライブラリへの切り替え
- 複数の株価データソースへの対応

### 3.5 その他の非機能要件

**スケーラビリティ**:
- Lambda 自動スケール（Web）
- DynamoDB オンデマンドキャパシティ
- EventBridge Scheduler による定期実行（バッチ）

**インフラ**:
- **Web**: Lambda (Web Adapter) + CloudFront
- **Batch**: Lambda + EventBridge Scheduler
- **データストア**: DynamoDB
- **認証**: Auth サービス（本プラットフォーム共通）

**モニタリング**:
- CloudWatch Logs による統一ログ管理
- CloudWatch Alarms によるエラー監視

**障害分離**:
- AI 解析の生成に失敗しても、日次サマリー自体の保存と表示は継続する
- 外部依存（AI API・Web 検索）の障害は局所化し、既存のサマリー機能へ波及させない

## 4. ドメインオブジェクト

| エンティティ | 説明 |
| ----------- | ---- |
| Exchange（取引所） | 株式が取引される市場（例: NYSE, NASDAQ） |
| Ticker（ティッカー） | 個別銘柄を識別するシンボル |
| Holding（保有株式） | ユーザーが保有する銘柄・数量・平均取得価格 |

| PushSubscription | Web Push 通知の購読情報 |
| DailySummary（日次サマリー） | 取引所ごとのティッカー日次 OHLCV データ・パターン分析・AI 解析結果・採点結果（Evaluation\* フィールド）を同一レコードで管理 |
