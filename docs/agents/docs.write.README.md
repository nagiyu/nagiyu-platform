# Docs Write Agent - 使用ガイド

## 概要

`docs.write.agent.md` は、`docs/` 配下のドキュメントを作成・更新する専属エージェントです。

**重要**: このエージェントは**ドキュメント作成専用**です。コード実装や `tasks/` 配下のドキュメント作成は行いません。

## 動作原則

1. **対象限定**: 変更対象は `docs/` 配下のみ
2. **最小限のルール**: 必須事項のみ記述し、コードを見れば分かる内容は書かない
3. **単体完結**: ドキュメント単体で理解できる内容にする
4. **コード参照禁止**: 実装断片やコードパスに依存した説明をしない
5. **4スペースインデント**: Markdownのインデントはスペース4つを厳守

## 使用方法

### 基本的な使い方

1. **GitHub Issueを作成**:
    - 何を、なぜ更新するかを明記
    - 対象のドキュメントカテゴリを明記

2. **エージェントをIssueにAssign**:
    - GitHubのUIで `@docs.write` をAssign

3. **エージェントが実行すること**:
    - 対象ドキュメントと関連ドキュメントを確認
    - 既存記述との整合性を検証
    - ルール準拠で作成・更新
    - 変更ファイルと要点を報告

### Issue記載例

```markdown
# ドキュメント更新: サービス運用方針の整理

## 目的
運用時の判断基準を統一し、更新時の迷いを減らす

## 対象
- docs/services/{サービス名}/README.md
- docs/services/{サービス名}/deployment.md

## 依頼内容
- 重複している説明を統合
- 手順の前提条件を明確化
- ドキュメント単体で理解できる構成に調整
```

## 対応範囲

- プラットフォーム全体の方針ドキュメント
- サービス別ドキュメント
- 共通ライブラリ関連ドキュメント
- インフラ関連ドキュメント
- エージェント運用ドキュメント

## 非対応範囲

- コード実装・リファクタリング
- `tasks/` 配下のドキュメント作成
- テストコードの追加や修正

## 既存エージェントとの連携

| エージェント | 役割 | 主な成果物 |
|------|------|------|
| `task.proposal` | 実装前タスクの整理 | `tasks/` 配下のタスク文書 |
| `docs.write` | ドキュメント整備 | `docs/` 配下の更新 |
| `task.implement` | 実装作業 | コードと関連更新 |

### 推奨フロー

1. `task.proposal` で実装計画を整理
2. `docs.write` でドキュメントを整備
3. `task.implement` で実装を進める

## 品質チェック観点

- [ ] `docs/` 配下のみ変更されている
- [ ] 最小限のルールに従っている
- [ ] コード参照に依存していない
- [ ] Markdownインデントが4スペースで統一されている
- [ ] 関連ドキュメントと矛盾しない

## トラブルシューティング

### Q: どこまで詳細に書くべきですか？

A: 実装の挙動説明ではなく、目的・前提・判断基準に集中します。コードを見れば分かる内容は省略します。

### Q: 既存ドキュメントに矛盾がある場合は？

A: 更新前に矛盾点を報告し、解決方針の合意後に編集します。

### Q: docs以外のファイル更新も依頼できますか？

A: できません。`docs.write` は `docs/` 配下専属です。
