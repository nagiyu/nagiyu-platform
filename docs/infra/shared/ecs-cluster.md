# 共通 ECS Cluster

## 目的

nagiyu プラットフォームで ECS Fargate を使用するすべての新規サービスが乗る、プラットフォーム共通の ECS Cluster を管理します。既存 Portal サービス専用の Cluster とは別に、新規 ECS サービスの受け皿として整備した共通基盤です。

---

## 設計方針

### Cluster は共有、ALB は個別

ECS Cluster はコンテナのオーケストレーション単位であり、複数のサービスを同一 Cluster で動かすことによる副作用はなく、Cluster 自体のコストも発生しません。そのため、プラットフォーム全体で一つの Cluster を共有します。

ALB はサービスごとに独立して管理します。アイドルタイムアウト・TLS 設定・ログ設定などはサービスの性質によってチューニング要件が異なるため、共有には馴染みません。既存 Portal も個別 ALB を持つため、この方針はプラットフォーム全体で一貫しています。

結果として：

- 共通基盤は Cluster と VPC に集約され、管理範囲がシンプルになる
- 各サービスは ALB 以下を自律管理し、他サービスへの影響なく調整できる

### 既存 Portal との関係

既存 Portal サービスは、root ドメイン提供専用の独自 Cluster を持っています。この Cluster は意味的に「Portal 専用」であるため、新規サービスはここに相乗りしません。

新規 ECS サービスはすべて共通 Cluster を使用します。将来的に Portal を共通 Cluster へ移行する選択肢は残しますが、現在はスコープ外です。

---

## リソースの責任分担

| リソース | 管理主体 | 理由 |
|---|---|---|
| ECS Cluster | 共通インフラ | 全サービス共有、コストゼロ |
| VPC / サブネット | 共通インフラ | 全サービス共有 |
| ALB | サービス個別 | チューニング要件がサービスにより異なる |
| Target Group / Listener Rule | サービス個別 | ALB に従う |
| ECS Service / Task Definition | サービス個別 | サービスごとのワークロード定義 |
| CloudFront / Route53 連携 | サービス個別 | サービスごとのドメイン・配信設定 |

新規 ECS サービスは共通 Cluster と共通 VPC を参照した上で、ALB 以下のリソースを自身のインフラスタック内で定義します。共通リソースへの参照は SSM Parameter Store を介して行い、スタック間の直接依存を避けます。

---

## タグ戦略

共通インフラのリソースには `Component: shared` タグを付与します。サービス固有のリソースには `Component: {サービス名}` タグを付与します。コスト配賦や障害調査でのリソース識別を一貫して行えるよう、タグは省略しないことが原則です。

---

## 環境別運用

dev と prod でそれぞれ独立した共通 Cluster を持ちます。ECS Cluster 自体のコストはゼロのため、環境分離によるコスト増加はありません。

dev 環境では、ALB がない状態でも共通 Cluster の構成として成立します。ALB は各サービスのインフラスタックが初めて登場させるリソースであり、共通基盤の責任範囲外です。
