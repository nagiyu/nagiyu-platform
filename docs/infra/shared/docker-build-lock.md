# Docker ビルド排他制御

## 概要

複数のワークフローが並列実行される際、`public.ecr.aws` の公開 ECR イメージへのアクセスが集中し、Rate Limit エラー（`toomanyrequests`）が発生することがある。これを防ぐため、S3 オブジェクトをセマフォとして使用し、同時に実行中の Docker ビルド数を上限未満に制御する。

---

## 設計方針

- **S3 セマフォ**: ロックの取得・解放を S3 オブジェクトの配置・削除で表現する
- **厳密でない排他制御**: 完全に同タイミングでのロック取得による一時的な上限超過は許容する
- **環境非依存**: ロック用バケットは dev/prod を問わず全ビルドで共有する単一バケット
- **ゾンビロック対策**: ビルド失敗などでロック解放されなかった場合も、S3 ライフサイクルルールによって自動回収する

---

## 仕組み

### ロックの取得

Docker ビルド開始前に S3 バケット内の既存ロックオブジェクト数を確認し、上限（3）未満であれば自身のロックオブジェクトを配置してビルドを開始する。上限以上であれば 30 秒ごとにポーリングし、空きが出るまで待機する。

### ロックの解放

Docker ビルドおよびイメージプッシュ完了後（成功・失敗問わず）に、自身のロックオブジェクトを削除する。ビルド失敗時も必ずロックが解放されるよう、解放処理は常に実行される条件で呼び出す。

### ロックオブジェクト

- **S3 バケット**: `nagiyu-docker-build-lock`（環境非依存の共有バケット）
- **オブジェクトキー**: `locks/{GITHUB_WORKFLOW}/{GITHUB_RUN_ID}/{GITHUB_JOB}`
- **オブジェクト内容**: ワークフロー名（`GITHUB_WORKFLOW`）、実行 ID（`GITHUB_RUN_ID`）、ジョブ名（`GITHUB_JOB`）、取得日時

---

## 制約と設定値

| 項目 | 値 | 備考 |
|-----|-----|-----|
| 同時実行上限 | 3 | 固定値 |
| ポーリング間隔 | 30 秒 | 固定値 |
| ゾンビロック自動削除 | 1 日経過後 | S3 ライフサイクルルール |

---

## IAM 権限

GitHub Actions 用ユーザーに付与済みの Application Policy に `s3:PutObject` / `s3:GetObject` / `s3:DeleteObject` / `s3:ListBucket` がリソース `*` で含まれているため、追加の IAM 変更は不要。

---

## 実装方式

ロック取得・解放のロジックはシェルスクリプトとして実装し、GitHub Actions の composite action（acquire/release）を通じて各ワークフローから呼び出す。

---

## インフラ管理

ロック用 S3 バケットは `infra/shared/` 配下の独立した CDK スタックで管理する。バケット名は固定値とし、各ワークフローから直接参照する。

### バケット設計

- 公開アクセスブロック: 有効
- バケットバージョニング: 不要（ロックオブジェクトは一時的）
- ライフサイクルルール: 配置から 1 日以上経過したオブジェクトを自動削除

---

## 運用

### ロックが残り続ける場合

S3 ライフサイクルルールにより、1 日後に自動削除される。緊急時はバケット内の `locks/` プレフィックス配下を手動で確認・削除する。

### Rate Limit エラーが依然として発生する場合

同時実行数をさらに絞るため、同時実行上限（3）の引き下げを検討する。上限を下げることで `public.ecr.aws` への同時アクセス数を減らし、Rate Limit の発生を抑制できる。上限値はロック取得・解放ロジックの設定値として管理されており、変更はその箇所のみで完結する。

---

## リトライ方針

### 背景・問題

S3 セマフォによる同時実行数制限（上限 3）を設けているが、一時的な集中アクセスにより `toomanyrequests` エラーが依然として発生することがある。エラーが出るたびに開発者が手動でリランする手間が生じ、CI の安定稼働を妨げていた。

### 決定

`toomanyrequests` エラーを検知した場合、自動で待機してリトライする仕組みを `build-docker-image` action に組み込む。

- 最大リトライ回数: 5 回
- 1 回あたりの待機時間: 60 秒
- リトライ対象: `toomanyrequests` のみ（それ以外のエラーは即座に失敗）

### 根拠・トレードオフ

- S3 ロックによる排他制御は競合状態でのスリップを完全には防げないため、補完的な対策として自動リトライが有効
- `toomanyrequests` は一時的なレート制限であり、短時間の待機で解消する性質を持つ
- リトライ対象を `toomanyrequests` のみに限定することで、他のエラー（設定ミスや権限エラーなど）は即座に通知され、誤魔化しにならない
- 最長 5 分（5 回 × 60 秒）の追加遅延を許容することで、手動リランの運用コストを排除する
