# dev-sync（prod → dev データ同期）

prod の DynamoDB データを dev 環境へコピーするプラットフォーム共通の同期機構について、背景と設計判断を残す。実装の詳細はコードを参照すること。本書はコードからは読み取れない「なぜ・どう判断したか」を対象とする。

---

## 背景・目的

運営者自身がプラットフォームの利用者でもあり、**dev 環境を日常利用のメイン（ドッグフーディングの場）にする**運用へ移行する。実利用による回帰検証が常時効き、「壊れたら自分が困る」当事者性が品質維持の動機になる。prod は「壊さない聖域」として温存する。

この移行の最大の課題は**データの扱い**である。サービスごとにデータの出自が異なり、一律には移せない。そこでデータを**出自で分類し、扱いを決める**。

---

## データの出自分類

| クラス | 正の所在 | dev での扱い |
|---|---|---|
| 外部 API 由来 | 外部サービス | dev が自前で再取得・再構築（同期不要） |
| コード / 設定由来 | git | デプロイで dev/prod が自動的に一致（同期不要） |
| 派生・エフェメラル（安価・再生成可） | なし | 使い捨て（同期不要） |
| 共有マスタ / 個人資産 / 高コスト派生 | prod | **prod 正・dev はコピー**（同期が要るのはここ） |

判断のポイント：

- 派生データであっても、**再生成が高コスト（外部 API の課金等）や過去分が非可逆**なものは、dev で作り直さず prod からコピーする。二重課金・二重負荷を避けるため。
- 個人データのうち dev で独立に運用したいもの（dev で動作確認のために自分で作るもの）は同期しない。

---

## 同期機構の概要

- **汎用コピー処理**：生の DynamoDB item を素通しで扱い、各サービスの内部表現（mapper / entity）に依存しない。これによりサービスと疎結合を保ち、1 つの機構で全サービスを賄う。
- **宣言的マニフェスト**：「どのテーブルから・どこへ・どの戦略で・いつ」同期するかを宣言として登録する。サービスを載せる作業はマニフェストへの追加で完結する。
- **冪等**：何回・いつ流しても同じ結果になる。スケジュール実行と手動実行の両方を想定する。

---

## 同期戦略：mirror と gsiWindow

更新の仕方は 2 つのツマミ（**スコープ**＝全件か直近ウィンドウか、**削除**＝prod に無い dev を消すか）で決まる。実用上は次の 2 戦略に集約する。

- **mirror**：全件を upsert し、prod に存在しない dev item を削除する（＝全件 + 削除あり）。
- **gsiWindow**：索引（GSI）で直近 N 日ぶんを取得して upsert する。削除しない（＝ウィンドウ + 削除なし）。

いずれも共通原則として **コピー先行 → 後から差分削除** の順で行う。全消し先行は dev が一瞬空になる窓を作るため避ける。

### 選定原理

起点は「**dev が prod の削除を反映する必要があるか**」。

- **必要**（prod に削除フローがある。廃止・削除を dev にも映したい）→ 削除あり。削除を安全に行うには対象範囲を全件走査する必要があるため、スコープは全件になる → **mirror**。
- **不要**（追記のみ・古いデータは不変・recency キーで範囲を有界化できる）→ 削除なしでよく、直近ウィンドウに縛れる → **gsiWindow**（安い）。

削除を有効にすると範囲は全件にせざるを得ない。ウィンドウだけ見て削除すると、未走査の prod 実在 item を誤って消すおそれがあるためである。この縛りにより「全件＋削除」と「ウィンドウ＋追記」が実質的なペアになる。

### gsiWindow の制約

- ウィンドウは GSI の**パーティション単位**でしか引けない。パーティションキーが取引所などで分かれている場合、**パーティションごとに 1 エントリ**を登録する。
- 未登録のパーティションは機構が見に行かないため、**取りこぼしても無音で欠落する**。対象（取引所等）が増えたら、必ずマニフェストにもエントリを追加する。
- GSI のソートキーが日付に接頭辞（例：`DATE#` で始まる）を持つ場合、ウィンドウ下限も**同じ接頭辞・同じ日付粒度**で組む必要がある。さもないと比較が常に成立して範囲が全件に広がり、ウィンドウが無効化される。

## 多層の安全設計

prod を壊さないことを最優先に、複数の層で担保する。

- **IAM 最小権限**：コピー元（prod）には読み取り専用、コピー先（dev）には書き込みのみを与える。prod への書き込み権限はそもそも発行しない。
- **コピー先の dev 限定**：コピー先テーブルは dev 系（命名が `-dev` で終わる）であることを必須とする。これを**実行時と合成（synth）時の両方で検査**し、条件を満たさない場合は中止する。prod への書き込み権限が合成時点で発行されること自体を防ぐ。
- **prod は常に read-only・冪等**：何度流しても安全。

## サービス別の同期方針

| サービス | 対象 | 戦略 | 備考 |
|---|---|---|---|
| Niconico | 楽曲（共有マスタ） | mirror | 削除フローあり。個人設定は対象外 |
| StockTracer | Exchange / Ticker（共有マスタ） | mirror | 削除（上場廃止等）あり |
| StockTracer | DailySummary（高コスト派生） | gsiWindow（取引所別・直近約 2 週間） | AI 解析・採点を内包。recency キーあり |

個人データ（保有・アラート・ユーザー設定等）はいずれも同期せず dev で自前運用する。

---

## dev バッチの扱い（高コスト派生の二重生成回避）

DailySummary のような高コスト派生は prod からコピーするため、**dev でも同じ生成バッチ（要約・採点）を自動実行すると外部 API を二重に叩く**ことになる。

- 生成バッチは「既に同等データがあればスキップ」する設計だが、**最新営業日ぶん**は「prod が生成済み・dev へコピー前」というレース窓で再生成され得る。これは毎日・全対象ぶんの二重課金になり無視できない。
- スキップ任せでは防ぎきれないため、**dev では当該バッチの自動スケジュールを停止**する。関数（デプロイ）は残すので、バッチを改修した際は**手動実行で dev 検証**できる。高コストな生成パスは「改修して検証したい時にだけ」手動で動かすのが、必要な検証を保ちつつ無駄な常時課金を断つ形になる。
- 一方、**個人データを生むバッチ（アラート評価等）は dev 稼働を継続**する。dev で当事者としてアラートを受けることがドッグフーディングの目的そのものであり、prod の焼き直しではないため。

判断軸は **「コピー対象データを再生産するバッチは止める／dev 固有データを生むバッチは残す」**。

---

## 運用

- **初回 backfill**：gsiWindow は初回のみウィンドウを大きく取って手動実行し、過去分を取り込む。以後は定常ウィンドウで日次同期する。mirror は初回も定常も同じ処理（初回は件数が多く時間がかかるだけ）。
- **対象の追加**：gsiWindow のパーティション（取引所等）が増えたら、マニフェストへのエントリ追加を忘れないこと（前述のとおり欠落が無音になる）。
- **後勝ち上書き**：dev 環境は同期による後勝ち上書きを許容する前提で運用する。
