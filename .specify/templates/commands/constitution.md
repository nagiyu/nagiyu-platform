# speckit.constitution コマンド

## 目的

`.specify/memory/constitution.md` のプロジェクト憲法を更新し、
依存するテンプレートファイルに変更を伝播する。

## 実行フロー

1. `.specify/memory/constitution.md` の現在のコンテンツを読み込む。
   - `[ALL_CAPS_IDENTIFIER]` 形式のすべてのプレースホルダトークンを特定する。
   - ユーザーが指定した原則数を尊重する（テンプレートより多くても少なくてもよい）。

2. プレースホルダの値を収集・導出する:
   - ユーザーの入力（会話）から値が提供されている場合はそれを使用する。
   - そうでない場合はリポジトリのコンテキスト（README、docs、以前の憲法バージョン）から推論する。
   - ガバナンス日付: `RATIFICATION_DATE` は元の採用日（不明な場合は TODO）、
     `LAST_AMENDED_DATE` は変更があった場合は本日の日付、なければ以前の値を維持する。
   - `CONSTITUTION_VERSION` はセマンティックバージョニングに従ってインクリメントする:
     - MAJOR: 後方互換性のないガバナンス/原則の削除または再定義
     - MINOR: 新原則/セクションの追加または実質的な拡充
     - PATCH: 表現の明確化、誤字修正、意味論的でない改善

3. 更新された憲法コンテンツを作成する:
   - すべてのプレースホルダを具体的なテキストに置換（未解決のブラケットトークンを残さない）。
   - 見出し階層を維持する。
   - 各原則セクション: 簡潔な名前行、非交渉のルールを記述した段落またはリスト、
     明確でない場合は明示的な根拠。
   - ガバナンスセクション: 改正手続き、バージョニングポリシー、コンプライアンスレビュー。
   - **すべての出力は日本語で記述すること（MUST）。**

4. 一貫性伝播チェックリスト:
   - `.specify/templates/plan-template.md` を読み込み、「憲法チェック」セクションが
     更新された原則と整合していることを確認する。
   - `.specify/templates/spec-template.md` を読み込み、スコープ/要件が整合しているか確認する。
   - `.specify/templates/tasks-template.md` を読み込み、タスク分類が
     新規/削除された原則駆動のタスク種別を反映しているか確認する。
   - `.specify/templates/commands/*.md` の各コマンドファイルを読み込み、
     エージェント固有の名前や古い参照が残っていないか確認する。
   - `README.md` などのランタイムガイダンスドキュメントを読み込み、
     変更された原則への参照を更新する。

5. 同期インパクトレポートを生成する（憲法ファイルの先頭に HTML コメントとして付加）:
   - バージョン変更: 旧 → 新
   - 変更された原則の一覧（名前が変更された場合は 旧名称 → 新名称）
   - 追加されたセクション
   - 削除されたセクション
   - 更新が必要なテンプレート（✅ 更新済み / ⚠️ 保留中）とファイルパス
   - 意図的に延期されたプレースホルダがある場合は追跡事項に記載する。

6. 最終出力前の検証:
   - 未説明のブラケットトークンが残っていないこと。
   - バージョン行がレポートと一致すること。
   - 日付が ISO 形式（YYYY-MM-DD）であること。
   - 原則が宣言的でテスト可能であり、曖昧な言語を含まないこと
     （「should」→「MUST」または「SHOULD」と根拠で置き換えること）。

7. 完成した憲法を `.specify/memory/constitution.md` に書き戻す（上書き）。

8. ユーザーへの最終サマリーを出力する:
   - 新バージョンとバージョンアップの根拠。
   - 手動フォローアップが必要なファイル。
   - 推奨コミットメッセージ（例: `docs: 憲法を vX.Y.Z に改正（原則追加 + ガバナンス更新）`）。

## フォーマット要件

- Markdown 見出しはテンプレートと同じ階層を使用する（昇格/降格しない）。
- 根拠の長い行は読みやすさのために折り返す（100文字目安）。
- セクション間に1行の空白行を保つ。
- 末尾の空白を避ける。
- **すべての出力（原則名、説明、根拠等）は日本語で記述すること（MUST）。**

## 注意事項

- 新しいテンプレートを作成しないこと。常に既存の `.specify/memory/constitution.md`
  ファイルを操作すること。
- 重要情報が欠如している場合（例: 批准日が本当に不明な場合）は、
  `TODO(<FIELD_NAME>): 説明` を挿入し、同期インパクトレポートの延期事項に含めること。
- ユーザーが部分的な更新（例: 1つの原則のみ改定）を提供した場合も、
  検証とバージョン決定のステップを実施すること。
