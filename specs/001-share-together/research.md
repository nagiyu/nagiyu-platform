# 調査レポート: みんなでシェアリスト (Share Together)

**ブランチ**: `copilot/define-requirements-again` | **日付**: 2025-06-12
**対応する計画**: [plan.md](./plan.md)

このドキュメントは実装計画のフェーズ0として、技術コンテキスト内の不明点を解消し、設計上の判断根拠を記録する。

---

## 1. DynamoDB シングルテーブル設計

### 決定事項
PK/SK ベースのシングルテーブルに `GSI1`（ユーザーのグループ逆引き）と `GSI2`（メールアドレスによるユーザー検索）の2つの GSI を設ける。

### 根拠
- 既存サービス（stock-tracker）が同パターン（PK/SK + 複数 GSI）を採用しており、プラットフォーム標準として確立されている
- DynamoDB のシングルテーブル設計は関連エンティティを同一テーブルで管理するため、コスト効率と一貫性が高い
- 本サービスで必要なアクセスパターン（詳細は `data-model.md` を参照）はすべて2つの GSI で網羅可能であると確認した

### 検討した代替案
- **マルチテーブル設計**: テーブルを User/List/Group などエンティティ別に分割する方式。シンプルで直感的だが、プラットフォーム標準からの逸脱となり、既存コードベースとの一貫性が損なわれるため却下。
- **GSI を3つ以上設ける**: グループリスト逆引き（GSI3）なども検討したが、PK/SK の工夫（`GROUP#{groupId}` の SK ベースクエリ）で代替可能なため不要と判断。

## 2. 認証・認可パターン

### 決定事項
既存サービス（niconico-mylist-assistant, stock-tracker）と同一の Auth Consumer パターンを採用する。
- `next-auth@^5.0.0-beta`（NextAuth v5）を使用
- Auth サービスが発行した JWT を `.nagiyu.com` ドメイン配下の Cookie 共有で受け取る
- `providers: []`（OAuth プロバイダーは Auth サービス側で管理）
- `session.strategy: 'jwt'`、maxAge 30 日
- Cookie 名は環境別サフィックスで分離（`dev`: `.dev` サフィックス、`prod`: なし）

### 根拠
- 既存サービス (`niconico-mylist-assistant/web/src/auth.ts`) で同パターンが実証済み
- Auth サービスとのインテグレーションは JWT 検証のみとすることが仕様制約に明記されている
- `.nagiyu.com` ドメインの Cookie 共有による SSO は既存インフラで確立されたアプローチ

### 検討した代替案
- **独自 OAuth フロー**: Share Together が直接 Google OAuth を実装する方式。Auth サービスとの二重管理になりプラットフォーム方針に反するため却下。
- **セッションベース認証**: DB 保存のセッション管理方式。JWT ステートレス認証に比べてインフラ複雑度が上がるため却下。

## 3. PWA 実装方針

### 決定事項
MVP v1 では以下のスコープで PWA を実装する:
- `public/manifest.json`: PWA マニフェスト（アプリ名、アイコン、display: standalone）
- `public/sw.js`: Service Worker シェル（キャッシュ戦略のみ、プッシュ通知購読コードはコメントアウト形式で記述）
- `ServiceWorkerRegistration` コンポーネント: Service Worker の登録処理

プッシュ通知（VAPID キー、SNS、EventBridge）は将来バージョンで対応。その際は `batch` パッケージを追加する。

### 根拠
- 仕様書の追加要件: 「PWA 必須（モバイルファースト）」「将来のプッシュ通知フレームワーク（シェルのみ、MVP v1 では未実装）」
- stock-tracker・niconico-mylist-assistant の両サービスで同パターン（manifest.json + sw.js + ServiceWorkerRegistration）が実装済み
- シェルを用意しておくことで将来のプッシュ通知実装時の差分が最小化される

### 検討した代替案
- **PWA なし**: MVP に絞ってマニフェストを省く方式。モバイルファーストという要件に反するため却下。
- **完全なプッシュ通知実装**: VAPID/SNS を含む完全実装。MVP スコープ外であり複雑度が高いため却下。

## 4. グループ招待通知の実装方式

### 決定事項
サービス内通知（In-app notification）方式を採用する。
- DynamoDB に招待レコード（`INVITE#` SK）を保存する
- ユーザーがログイン後または任意のページにアクセスした際、未処理の招待を API からポーリングして表示する
- ナビゲーションバーに「招待あり」のバッジ表示を行う

### 根拠
- 仕様書の制約: 「グループへの招待通知はサービス内通知（次回ログイン時またはページアクセス時に表示）で対応し、メール通知は対象外」
- シンプルな実装で済み、追加インフラ（SES、SNS 等）が不要

### 検討した代替案
- **メール通知**: SES 経由での招待メール送信。仕様のスコープ外として明示的に除外されているため却下。
- **WebSocket リアルタイム通知**: MVP スコープ外（リアルタイム同期なし）として明示されているため却下。

## 5. インフラ構成（CDK スタック）

### 決定事項
stock-tracker を参考にした以下のスタック構成を採用する:

| スタック | 役割 |
|---------|------|
| `DynamoDBStack` | シングルテーブル + 2 GSI |
| `EcrStack` | Web コンテナリポジトリ |
| `LambdaStack` | Lambda（コンテナ: web） |
| `IAMStack` | 開発用 IAM ユーザー（dev 環境のみ） |
| `CloudFrontStack` | CloudFront（`CloudFrontStackBase` 継承） |

**batch は MVP では含めない**（プッシュ通知実装時に追加）。
**SNS/EventBridge も MVP では含めない**（プッシュ通知実装時に追加）。

ドメイン設定:
- dev: `dev-share-together.nagiyu.com` → CNAME → CloudFront ディストリビューション
- prod: `share-together.nagiyu.com` → CNAME → CloudFront ディストリビューション
- CloudFront は `@nagiyu/infra-common` の `CloudFrontStackBase` を使用

### 根拠
- stock-tracker との一貫性維持（プラットフォーム標準パターン）
- バッチ処理が不要な MVP 段階ではシンプルな構成にとどめることが望ましい
- `CloudFrontStackBase` 使用により CDN 設定のメンテナンスコストを削減

### 検討した代替案
- **ECS/Fargate**: Lambda の代わりに常時起動コンテナを使用。コスト・管理コストともに Lambda の方が有利なため却下。
- **App Runner**: コンテナのフルマネージドサービス。Lambda との一貫性を優先して却下。

## 6. ユーザー検索方式（招待時）

### 決定事項
グループオーナーが招待時にユーザーを検索する際は **メールアドレスによる完全一致検索** を採用する。
- DynamoDB の GSI2（EmailIndex）: `GSI2PK = EMAIL#{email}` でユーザーを一意に特定
- ユーザー ID（`userId`）での検索は API レイヤーでの追加対応として保持するが、UI はメールアドレス入力を主とする

### 根拠
- 仕様書: 「ユーザー ID またはメールアドレスで検索してグループに招待できること（FR-011）」
- Auth サービスが Google OAuth のみをサポートし、メールアドレスで一意識別可能（仕様の仮定に明記）
- メールアドレス完全一致はプレフィックス検索と異なりインデックス設計がシンプル

### 検討した代替案
- **名前検索（部分一致）**: ユーザー名によるあいまい検索。DynamoDB では GSI でも部分一致が困難（scan が必要）なため MVP では除外。
- **ユーザー ID のみ**: メールアドレス検索を省略する方式。仕様（FR-011）で明示されているため却下。

## 7. グループ削除時のカスケード削除

### 決定事項
グループ削除時は以下を **アプリケーション層でトランザクション削除** する:
- グループメタデータ
- グループメンバーシップ（全メンバー）
- グループ共有リスト（全リスト）
- 各共有リスト内の ToDo アイテム（全アイテム）

DynamoDB の `TransactWriteItems` は 100 アイテム上限があるため、アイテム数が多い場合は複数バッチに分割する。

### 根拠
- 仕様書（FR-015）: 「グループを削除できること（削除時はグループ内の全共有リストとデータも削除される）」
- DynamoDB にはリレーショナル DB のような外部キー制約がないため、アプリケーション層での制御が必要

### 検討した代替案
- **DynamoDB Streams + Lambda**: 削除イベントをトリガーにカスケード削除するイベント駆動方式。MVP では過剰な複雑度のため却下。
- **TTL による自動削除**: TTL 属性を使った自動期限切れ削除。即時削除でないためデータ整合性リスクがあり却下。

## 8. CI/CD ワークフロー

### 決定事項
以下2つのワークフローを実装する（憲法 V 原則に準拠）:

| ワークフロー | トリガー | 内容 |
|-------------|---------|------|
| `share-together-web-verify-fast.yml` | `integration/**` への PR | ビルド・lint・format-check・ユニットテスト・E2E（chromium-mobile のみ） |
| `share-together-web-verify-full.yml` | `develop` への PR | ビルド・lint・format-check・ユニットテスト・カバレッジ（80%）・E2E（全3デバイス） |

ワークスペース指定はパッケージ名（`@nagiyu/share-together-web`）。

### 根拠
- 憲法 V の「2段階 PR 検証ワークフロー」に準拠
- 既存サービス（stock-tracker, niconico-mylist-assistant）と同一パターンを踏襲

## 9. ページリフレッシュによる最新化の設計方針

### 決定事項
リアルタイム同期は MVP 外のため、以下の UX 設計を採用する:
- 共有リストページには **「更新」ボタン** を設置し、手動でページを再フェッチできるようにする
- Next.js の `router.refresh()` または SWR/React Query の `mutate()` を活用
- 各ページは初回レンダリング時に必ずサーバーから最新データを取得する（キャッシュ無効化設定）

### 根拠
- 仕様書（SC-003）: 「他のメンバーがページを更新することで 5 秒以内に確認できる」
- リアルタイム同期（WebSocket 等）は仕様の制約として除外されている

### 検討した代替案
- **ポーリング**: 定期的に API を自動呼び出す方式。Lambda のコールドスタートコストと不要なリクエスト増加のため MVP では採用しない。
- **WebSocket / Server-Sent Events**: リアルタイム同期の手段。MVP スコープ外として明示されているため却下。
