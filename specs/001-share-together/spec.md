# 機能仕様書: みんなでシェアリスト (Share Together)

**機能ブランチ**: `001-share-together`
**作成日**: 2025-06-12
**ステータス**: ドラフト
**入力**: ユーザーの説明: "本リポジトリの共通プラットフォームのルールに則り、新サービス みんなでシェアリスト (Share Together) を作成する。これは、他者間で ToDo を共有できるツールである。本リポジトリの Auth サービスの認証機能によってユーザー認証を行い、そのユーザー間で ToDo を共有する。各ユーザーは、デフォルトで自分のみの ToDo リストに加え、追加で複数の自分のみの ToDo リストを作成でき、さらに複数ユーザー間でグループを作成することができ、そのグループのみが閲覧、共有できる複数の ToDo リストを作成することができる。ToDo の機能充実は将来的なバージョンアップで対応するものとし、本対応は初版として複数人で ToDo を共有できるところを MVP とする。"

## ユーザーシナリオとテスト *(必須)*

### ユーザーストーリー 1 - 個人 ToDo リストの管理 (優先度: P1)

ログイン済みのユーザーが、自分専用のデフォルト ToDo リストに ToDo を追加・編集・削除・完了にして、日常的なタスクを管理できる。

**この優先度の理由**: 個人 ToDo 管理はサービスの基盤となる機能であり、他のすべての機能（グループ共有等）の前提となる。これ単独でも最小限の価値を提供できる MVP コアである。

**独立したテスト**: 認証済みユーザー 1 名のみで、グループ機能を使わずに ToDo の作成・編集・削除・完了操作をすべて検証できる。

**受け入れシナリオ**:

1. **前提** ログイン済みユーザーがサービスにアクセスする、**操作** サービスのトップページを開く、**結果** デフォルトの個人 ToDo リストが表示され、既存の ToDo 一覧が確認できる
2. **前提** 個人 ToDo リストを表示している、**操作** 新しい ToDo のタイトルを入力して追加する、**結果** 入力した ToDo が一覧に追加され、未完了状態として表示される
3. **前提** ToDo が一覧に表示されている、**操作** ToDo を完了としてマークする、**結果** ToDo が完了状態になり、視覚的に区別して表示される
4. **前提** ToDo が一覧に表示されている、**操作** ToDo のタイトルを編集して保存する、**結果** 更新された内容が一覧に反映される
5. **前提** ToDo が一覧に表示されている、**操作** ToDo を削除する、**結果** 該当 ToDo が一覧から削除される

---

### ユーザーストーリー 2 - 複数の個人 ToDo リストの作成と切り替え (優先度: P2)

ログイン済みのユーザーが、用途ごとに複数の個人 ToDo リストを作成し、リストを切り替えながら各リストの ToDo を管理できる。

**この優先度の理由**: 「仕事用」「買い物リスト」などリスト単位で管理することでユーザーの利便性が大きく向上する。デフォルトリストとの親和性が高く、個人利用の完成度を高める。

**独立したテスト**: 認証済みユーザー 1 名のみで、複数のリスト作成・切り替え・削除をグループ機能を使わずに検証できる。

**受け入れシナリオ**:

1. **前提** ログイン済みユーザーがデフォルトの個人リストを表示している、**操作** 新しい個人 ToDo リスト名を入力して作成する、**結果** 新しいリストが作成され、リスト一覧に表示される
2. **前提** 複数の個人 ToDo リストが存在する、**操作** 別のリストを選択して切り替える、**結果** 選択したリストの ToDo 一覧が表示される
3. **前提** ユーザーが個人リストを表示している、**操作** リスト名を変更して保存する、**結果** 更新されたリスト名が一覧に反映される
4. **前提** ユーザーが個人リストを一覧表示している、**操作** デフォルト以外の個人リストを削除する、**結果** 該当リストとそのリスト内の ToDo がすべて削除される
5. **前提** ユーザーがデフォルトリストを表示している、**操作** デフォルトリストを削除しようとする、**結果** デフォルトリストは削除できないことがわかるメッセージが表示される

---

### ユーザーストーリー 3 - グループの作成とメンバー管理 (優先度: P2)

ログイン済みのユーザーが、他のユーザーを招待してグループを作成し、グループメンバーを管理できる。

**この優先度の理由**: グループ機能はサービスの「共有」という本質的価値を実現するための前提であり、グループ共有 ToDo リスト（P1相当）を使うには先にグループが必要となる。

**独立したテスト**: 認証済みユーザー 2 名以上で、グループ作成・招待・参加・脱退のフローを検証できる。

**受け入れシナリオ**:

1. **前提** ログイン済みユーザーがサービスを利用している、**操作** 新しいグループ名を入力してグループを作成する、**結果** グループが作成され、作成者がグループオーナーとして登録される
2. **前提** グループオーナーがグループを管理している、**操作** 他のユーザーをユーザー名または識別子で検索してグループに招待する、**結果** 招待が送信され、招待されたユーザーに通知される（サービス内通知）
3. **前提** ユーザーがグループへの招待を受け取っている、**操作** 招待を承認する、**結果** ユーザーがグループメンバーとして追加され、グループの ToDo リストにアクセスできるようになる
4. **前提** ユーザーがグループへの招待を受け取っている、**操作** 招待を拒否する、**結果** グループに追加されず、招待が破棄される
5. **前提** グループメンバーがグループに参加している、**操作** グループから脱退する、**結果** ユーザーはグループから除外され、グループの ToDo リストにアクセスできなくなる
6. **前提** グループオーナーがグループを管理している、**操作** 特定のメンバーをグループから除外する、**結果** 除外されたメンバーはグループの ToDo リストにアクセスできなくなる

---

### ユーザーストーリー 4 - グループ共有 ToDo リストの管理 (優先度: P1)

グループメンバー全員が、グループに紐づいた共有 ToDo リストを閲覧・追加・編集・削除でき、リアルタイムではなくとも最新状態を確認できる。

**この優先度の理由**: 複数人で ToDo を共有することが本サービスの MVP の中心価値であり、グループ内での ToDo 共同管理を実現する最重要機能である。

**独立したテスト**: 認証済みユーザー 2 名以上が同じグループに参加した状態で、ToDo の作成・確認・編集・削除を各メンバーが操作して検証できる。

**受け入れシナリオ**:

1. **前提** グループメンバーのユーザーがグループを表示している、**操作** グループに新しい共有 ToDo リストを作成する、**結果** 新しいリストが作成され、同グループの全メンバーがそのリストを閲覧できる
2. **前提** 共有 ToDo リストが存在するグループにアクセスする、**操作** 共有 ToDo リストを選択する、**結果** そのリストの ToDo 一覧が表示される
3. **前提** グループメンバーが共有 ToDo リストを表示している、**操作** 新しい ToDo を追加する、**結果** 追加した ToDo が共有リストに反映され、同グループの全メンバーがページを更新すると確認できる
4. **前提** グループメンバーが共有 ToDo リストを表示している、**操作** 他のメンバーが追加した ToDo を完了にマークする、**結果** 完了状態が共有リストに反映される
5. **前提** グループメンバーが共有 ToDo リストを表示している、**操作** ToDo を削除する、**結果** 削除後のリストが全メンバーに反映される
6. **前提** グループに参加していないユーザーがグループの ToDo リストの URL に直接アクセスしようとする、**操作** URL を入力してアクセスする、**結果** アクセスが拒否され、権限エラーまたはリダイレクトが発生する

---

### ユーザーストーリー 5 - 認証・ログイン (優先度: P1)

未ログインのユーザーがサービスにアクセスした際、Auth サービスを通じてシームレスにログインし、認証後にサービスを利用できる。

**この優先度の理由**: すべての機能はログイン状態を前提とするため、認証フローはサービス全体の前提条件である。

**独立したテスト**: 未認証状態でサービスにアクセスし、Auth サービスへのリダイレクト・認証後の戻り・ログアウト操作を検証できる。

**受け入れシナリオ**:

1. **前提** 未ログインのユーザーがサービスにアクセスする、**操作** サービスの URL を開く、**結果** Auth サービスのログインページにリダイレクトされる
2. **前提** Auth サービスでログインが完了する、**操作** Auth サービスの認証フローを完了する、**結果** Share Together サービスに戻り、ログイン状態でサービスを利用できる
3. **前提** ログイン済みユーザーがサービスを利用している、**操作** ログアウト操作を行う、**結果** ログアウトされ、以降サービスにアクセスすると再度ログインが要求される

---

### エッジケース

- グループから脱退したメンバーが以前アクセスしていた共有リストのURLを直接開いた場合は、アクセスが拒否される
- グループオーナーがグループを退出または削除した場合、グループに紐づく全共有 ToDo リストとそのデータが削除される
- 同じユーザーに対して重複してグループ招待が送信された場合、2件目以降の招待は受け付けない
- セッションが切れた状態で ToDo の追加・編集操作を行った場合、操作は失敗し、再ログインへ誘導される

## 要件 *(必須)*

### 機能要件

#### 認証

- **FR-001**: システムは、未認証ユーザーが Share Together サービスにアクセスした際に Auth サービスのログインページへリダイレクトすることを MUST とする
- **FR-002**: システムは、Auth サービスが発行した認証情報を検証してユーザーを識別することを MUST とする
- **FR-003**: システムは、ログアウト操作によってセッションを終了することを MUST とする

#### 個人 ToDo リスト

- **FR-004**: システムは、ユーザーアカウント作成（初回ログイン）時にデフォルトの個人 ToDo リストを自動生成することを MUST とする
- **FR-005**: ユーザーは、デフォルトリストに加えて追加の個人 ToDo リストを任意の数だけ作成できることを MUST とする
- **FR-006**: ユーザーは、デフォルト以外の個人 ToDo リストの名称を変更・削除できることを MUST とする
- **FR-007**: システムは、デフォルト個人 ToDo リストの削除を禁止することを MUST とする
- **FR-008**: ユーザーは、個人 ToDo リスト内で ToDo を作成・編集・削除・完了状態への変更ができることを MUST とする
- **FR-009**: システムは、個人 ToDo リストとその内容を本人以外には公開しないことを MUST とする

#### グループ管理

- **FR-010**: ユーザーは、グループを作成できることを MUST とする（作成者がオーナーとなる）
- **FR-011**: グループオーナーは、他のユーザーをグループに招待できることを MUST とする
- **FR-012**: 招待されたユーザーは、招待を承認または拒否できることを MUST とする
- **FR-013**: グループメンバーは、グループから自発的に脱退できることを MUST とする
- **FR-014**: グループオーナーは、任意のメンバーをグループから除外できることを MUST とする
- **FR-015**: グループオーナーは、グループを削除できることを MUST とする（削除時はグループ内の全共有リストとデータも削除される）
- **FR-016**: ユーザーは複数のグループに所属できることを MUST とする

#### グループ共有 ToDo リスト

- **FR-017**: グループメンバーは、グループに紐づく共有 ToDo リストを複数作成できることを MUST とする
- **FR-018**: グループメンバーは、グループの全共有 ToDo リストを閲覧できることを MUST とする
- **FR-019**: グループメンバーは、グループ共有 ToDo リスト内で ToDo を作成・編集・削除・完了状態への変更ができることを MUST とする
- **FR-020**: システムは、グループ共有 ToDo リストをそのグループのメンバー以外には公開しないことを MUST とする
- **FR-021**: グループメンバーは、グループ共有 ToDo リストの名称を変更・削除できることを MUST とする

### 対象外 (スコープ外)

以下の機能は将来バージョンでの対応とし、本 MVP では実装しない:

- ToDo への期限設定
- 期限に基づくプッシュ通知
- ToDo の種類・カテゴリ分類
- ToDo へのコメント・添付ファイル
- リアルタイム同期（他ユーザーの変更を即時反映）
- グループ内のロール・権限の細分化（オーナー以外への管理権限付与）

### 主要エンティティ

- **ユーザー (User)**: Auth サービスで識別されるアカウント。表示名と一意識別子を持つ
- **個人 ToDo リスト (Personal List)**: 特定ユーザーだけが所有・閲覧できる ToDo リスト。各ユーザーはデフォルトリスト 1 件を必ず持つ
- **グループ (Group)**: 複数ユーザーが属するチーム単位。オーナー 1 名と複数のメンバーで構成される
- **グループメンバーシップ (Membership)**: ユーザーとグループの紐付け。招待ステータス（招待中・承認済み・拒否済み）を持つ
- **グループ共有 ToDo リスト (Group List)**: 特定グループのメンバー全員が閲覧・操作できる ToDo リスト。グループに複数紐づく
- **ToDo アイテム (Todo Item)**: 個人リストまたはグループ共有リストに属するタスク単位。タイトル・完了状態・作成者・作成日時を持つ

## 成功基準 *(必須)*

### 測定可能な成果

- **SC-001**: ユーザーが初回ログイン後、30 秒以内に最初の ToDo を追加できる（操作の直感性）
- **SC-002**: グループ作成からメンバー招待・承認までのフローを 5 分以内に完了できる（グループ設定の容易さ）
- **SC-003**: グループメンバーが追加・更新した ToDo を、他のメンバーがページを更新することで 5 秒以内に確認できる（共有反映の速度）
- **SC-004**: 認証されていないユーザーや非メンバーが他者の個人リスト・グループリストにアクセスできない（アクセス制御の完全性: 0 件の不正アクセス許可）
- **SC-005**: 個人 ToDo リスト・グループ共有リスト双方でデータが失われることなく保存される（データ整合性: 操作後のデータ保持率 100%）
- **SC-006**: 既存の Auth サービスで認証済みのユーザーが、追加のログイン操作なしで Share Together を利用できる（SSO の実現）

## 前提と制約

### 前提条件

- 本サービスのユーザーは nagiyu プラットフォームの Auth サービスにアカウントを持っていること
- Auth サービスとの SSO 統合は `.nagiyu.com` ドメイン配下の Cookie 共有方式を前提とする
- 本サービスは nagiyu プラットフォームのモノレポ構成に従い、`services/share-together/` 配下に配置する

### 制約

- 認証・認可の実装は Auth サービスの Consumer パターン（JWT 検証のみ）に準拠し、Auth サービスが発行した JWT を検証する形とする
- 本 MVP ではリアルタイム同期（WebSocket 等）は対象外とし、ページ更新時に最新状態を表示する形で十分とする
- グループへの招待通知はサービス内通知（次回ログイン時またはページアクセス時に表示）で対応し、メール通知は対象外とする

### 仮定 (Assumptions)

- ユーザーの一意識別は Auth サービスが提供するユーザー ID で行う
- ToDo アイテムが持つ属性は MVP 段階ではタイトルと完了状態のみとする（期限・詳細説明は将来対応）
- グループメンバーはオーナーと一般メンバーの 2 種類のみとし、細かい権限管理は将来対応とする
- グループ共有 ToDo リストへの操作権限はすべてのメンバーに均等に付与する（作成・編集・削除）
