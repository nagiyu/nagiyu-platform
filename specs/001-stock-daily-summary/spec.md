# 機能仕様書: Stock Tracker 日次サマリー表示

**機能ブランチ**: `001-stock-daily-summary`
**作成日**: 2026-02-27
**ステータス**: ドラフト
**入力**: ユーザーの説明: "Stock Tracker にて日次サマリーを確認できるようにする。取引所ごとに、取引時間終了後にバッチを走らせ、その取引所に関連する各ティッカーについて、サマリーを表示する。初版としては、始値、終値、高値、低値が一覧で表示できればいい。バッチは取引所ごとに作成するのではなく、1時間に1回走るバッチを新設し、取引時間が終了した営業所があればサマライズをするイメージ。データは DynamoDB に保存し、ユーザーは直近のサマリーを画面から確認できるようにする。"

## ユーザーシナリオとテスト *(必須)*

### ユーザーストーリー 1 - 日次サマリーの閲覧 (優先度: P1)

ユーザーが Stock Tracker の画面から、各取引所に登録された各ティッカーの直近の日次サマリー（始値・高値・安値・終値）を一覧形式で確認できる。

**この優先度の理由**: サービスの中核となるユーザー向け価値提供。ユーザーが日々の株価動向を手軽に把握できることが本機能の主目的。

**独立したテスト**: 事前にサマリーデータを DynamoDB に登録した状態で、サマリー画面を表示し、ティッカーごとの始値・高値・安値・終値が正しく表示されることを確認できる。

**受け入れシナリオ**:

1. **前提** サマリーデータが DynamoDB に保存されている、**操作** ユーザーがサマリー画面にアクセスする、**結果** 取引所ごとのティッカー一覧と各ティッカーの始値・高値・安値・終値が表示される
2. **前提** サマリーデータが存在しない、**操作** ユーザーがサマリー画面にアクセスする、**結果** 「データがありません」等の空状態メッセージが表示される
3. **前提** 複数の取引所に複数のティッカーが登録されている、**操作** ユーザーがサマリー画面にアクセスする、**結果** 取引所ごとにグループ化されたティッカー一覧とサマリー情報が表示される

---

### ユーザーストーリー 2 - 日次サマリーの自動生成 (優先度: P1)

システムが1時間に1回バッチ処理を実行し、取引時間が終了した取引所に属する全ティッカーの日次サマリーを自動的に生成して DynamoDB に保存する。

**この優先度の理由**: サマリーデータがなければユーザーストーリー1は成立しない。自動生成が本機能のデータ供給源であるため、同等の優先度を持つ。

**独立したテスト**: バッチ処理を任意のタイミングで手動実行し、取引時間終了済みの取引所のティッカーについてサマリーデータが DynamoDB に保存されることを確認できる。

**受け入れシナリオ**:

1. **前提** 1時間バッチが実行される時点で、取引時間が終了した取引所が存在する、**操作** バッチが定期実行される、**結果** その取引所の全ティッカーのサマリー（始値・高値・安値・終値）が生成され DynamoDB に保存される
2. **前提** 1時間バッチが実行される時点で、全取引所が取引時間中である、**操作** バッチが定期実行される、**結果** サマリーの生成はスキップされ、何も保存されない
3. **前提** 取引時間終了済みの取引所でサマリーが既に当日分として保存済みである、**操作** バッチが再実行される、**結果** 既存のサマリーが上書き更新される（重複保存しない）
4. **前提** ティッカーが存在しない取引所がある、**操作** バッチが定期実行される、**結果** そのティッカー分の処理はスキップされ、他の取引所の処理には影響しない

---

### エッジケース

- 取引時間を定義していない取引所が存在する場合はどのように処理されるか？ → 取引時間定義がない取引所の処理はスキップし、ログに記録する
- TradingView API からデータが取得できない場合はどうなるか？ → エラーをログに記録し、該当ティッカーのサマリー保存はスキップして次のティッカーへ処理を継続する
- 当日のデータが非常に少量（例: 取引開始直後に取引所が閉鎖）の場合はどうなるか？ → 取得できたデータでサマリーを生成して保存する

## 要件 *(必須)*

### 機能要件

- **FR-001**: 既存の1時間間隔バッチに、取引時間終了済み取引所の検出ロジックを追加する MUST とする
- **FR-002**: バッチは取引時間が終了した取引所を特定し、その取引所に紐づく全ティッカーのサマリーを生成することを MUST とする
- **FR-003**: サマリーとして、当日の始値・高値・安値・終値を収集することを MUST とする
- **FR-004**: 生成されたサマリーデータを DynamoDB に保存することを MUST とする
- **FR-005**: 同一ティッカーの同一日付のサマリーが重複して保存されないこと（上書き更新）を MUST とする
- **FR-006**: ユーザーはログイン後、サマリー一覧画面で直近のサマリーを閲覧できることを MUST とする
- **FR-007**: サマリー一覧は取引所ごとのティッカー一覧形式で表示されることを MUST とする
- **FR-008**: サマリー画面では各ティッカーの銘柄名、シンボル、始値・高値・安値・終値を表示することを MUST とする
- **FR-009**: システムは1時間間隔バッチで取引所の取引時間終了を検出する際、取引所のタイムゾーンを考慮することを MUST とする
- **FR-010**: バッチ処理中にエラーが発生した場合、エラーをログに記録し、他のティッカーや取引所の処理を継続することを MUST とする

### 主要エンティティ *(データを扱う機能の場合)*

- **DailySummary**: 特定ティッカーの特定日の日次サマリー。属性として、ティッカーID・取引所ID・日付（YYYY-MM-DD形式）・始値・高値・安値・終値・作成日時・更新日時を持つ。Ticker および Exchange と関連を持つ。

## 成功基準 *(必須)*

### 測定可能な成果

- **SC-001**: 取引時間終了後、次回のバッチ実行（最大1時間後）までに全ティッカーのサマリーが自動生成される
- **SC-002**: ユーザーがサマリー画面にアクセスすると、直近の日次サマリー（始値・高値・安値・終値）が3秒以内に表示される
- **SC-003**: バッチ処理は一部のティッカーでエラーが発生しても、エラー率100%でない限り他のティッカーのサマリーを保存し続ける（部分的な成功を許容）
- **SC-004**: 同一ティッカーの同一日付のサマリーが重複して保存されない（冪等性を保証）

## 前提・依存関係

### 前提

- 既存の1時間間隔バッチ（hourly.ts）のインフラ（EventBridge Scheduler + Lambda）が利用可能である
- DynamoDB のシングルテーブル設計を継続して使用する
- サマリーデータ取得には既存の TradingView API クライアント（getChartData）を活用する
- 認証・認可の仕組みは既存のものを流用する

### 依存関係

- 取引所（Exchange）と ティッカー（Ticker）が事前に DynamoDB に登録されていること
- 既存の `isTradingHours` 関数を活用して取引時間終了の判定を行うこと
- TradingView API への接続が可能であること
